## 一、基本概念

### 1.1 传输层功能
- 为应用进程提供**端到端的逻辑通信**（网络层是主机到主机）
- 复用和分用
- 差错检测

### 1.2 端口号（16位，0~65535）

| 类别 | 范围 | 说明 | 举例 |
|------|------|------|------|
| **熟知端口** | 0~1023 | 分配给最常用的应用进程 | HTTP=80, FTP=21/20, DNS=53, SMTP=25, POP3=110 |
| **登记端口** | 1024~49151 | 需在IANA登记 | MySQL=3306 |
| **客户端端口** | 49152~65535 | 客户端临时使用 | — |

### 1.3 套接字Socket
- **Socket = (IP地址 : 端口号)**
- 唯一标识网络中的一条通信端点
- TCP连接 = (Socket1, Socket2) = (IP1:port1, IP2:port2)

---

## 二、UDP协议

### 2.1 特点
- **无连接**：发送前不需要建立连接
- **不可靠**：尽最大努力交付，不保证到达
- **面向报文**：应用层交多长的报文，UDP就加首部后发送（不拆分不合并）
- **无拥塞控制**：适合实时应用（视频、语音、DNS）
- **首部开销小**：8字节
- **支持一对一、一对多、多对一、多对多**

### 2.2 UDP报文格式（首部8字节）
```
| 源端口（2B） | 目的端口（2B） |
| 长度（2B）   | 校验和（2B）   |
|        数据部分               |
```

### 2.3 UDP校验和计算
1. 在UDP报文前加**12字节伪首部**（源IP、目的IP、全0、协议号17、UDP长度）
2. 校验和字段先置0
3. 将整个（伪首部+UDP首部+数据）按16位分组，求**二进制反码和**
4. 结果取反码填入校验和字段
5. 接收方同样加伪首部计算，结果应为全1

> ⚠️ 伪首部只用于校验和计算，不实际传输。

## 三、TCP协议

### 3.1 特点
- **面向连接**
- **可靠传输**
- **面向字节流**（把数据看作无结构的字节流）
- **全双工**
- **点对点**（一对一）
- 有**拥塞控制和流量控制**

### 3.2 TCP报文段首部格式（重要字段）

| 字段 | 大小 | 说明 |
|------|------|------|
| **源端口** | 2B | |
| **目的端口** | 2B | |
| **序号（seq）** | 4B | 本报文段数据第一个字节的编号 |
| **确认号（ack）** | 4B | 期望收到对方下一个报文段第一个字节的编号 |
| **数据偏移** | 4位 | 首部长度（单位4字节），最小5（20B） |
| 6个标志位 | 各1位 | URG ACK PSH RST SYN FIN |
| **窗口** | 2B | 接收窗口大小（rwnd） |
| 校验和 | 2B | |
| 紧急指针 | 2B | URG=1时有效 |

**6个标志位含义**：
- **URG**：紧急数据，配合紧急指针
- **ACK**：确认号有效，连接建立后所有报文ACK=1
- **PSH**：催促接收方尽快交付
- **RST**：重置连接（连接出现严重差错）
- **SYN**：同步，连接建立时置1
- **FIN**：释放连接

---

## 四、TCP三次握手

```
客户端                        服务器
  |                             |
  |--- SYN=1, seq=x --------->|  (①客户端发连接请求)
  |                             |  客户端: SYN_SENT
  |<-- SYN=1,ACK=1,seq=y,     |  (②服务器确认并发起)
  |    ack=x+1 ----------------|  服务器: SYN_RCVD
  |                             |
  |--- ACK=1, seq=x+1,        |  (③客户端确认)
  |    ack=y+1 --------------->|  
  |                             |  双方: ESTABLISHED
```

**状态转换**：
- 客户端：CLOSED → **SYN_SENT** → **ESTABLISHED**
- 服务器：CLOSED → **LISTEN** → **SYN_RCVD** → **ESTABLISHED**

**为什么三次？** 防止已失效的连接请求到达服务器造成资源浪费。两次握手无法让服务器确认客户端收到了SYN+ACK。

---

## 五、TCP四次挥手

```
客户端                        服务器
  |                             |
  |--- FIN=1, seq=u --------->|  (①客户端请求释放)
  |                             |  客户端: FIN_WAIT_1
  |<-- ACK=1, seq=v, ack=u+1 -|  (②服务器确认)
  |                             |  服务器: CLOSE_WAIT
  |                             |  客户端: FIN_WAIT_2
  |                             |
  |<-- FIN=1,ACK=1,seq=w,     |  (③服务器请求释放)
  |    ack=u+1 ----------------|  服务器: LAST_ACK
  |                             |
  |--- ACK=1, seq=u+1,        |  (④客户端确认)
  |    ack=w+1 --------------->|  
  |                             |  客户端: **TIME_WAIT**
  |         等待2MSL           |  服务器: CLOSED
  |                             |  客户端: CLOSED
```

**为什么四次？** TCP全双工，每个方向需要单独关闭。FIN表示"我不再发送数据"，但还能接收。

**TIME_WAIT（2MSL）**：
- MSL = 最长报文段寿命
- 等2MSL的原因：①确保最后ACK到达服务器 ②让本连接的所有报文在网络中消失

---

## 六、可靠传输机制

| 机制 | 说明 |
|------|------|
| **序号** | 给每个字节编号，保证有序 |
| **确认号（ACK）** | 累积确认：ack=N表示N之前所有字节已正确收到 |
| **超时重传** | 超时未收到ACK，重传该报文段 |
| **快速重传** | 收到**3个冗余ACK**（即连续4个相同ACK），立即重传（不等超时） |

---

## 七、流量控制

- 使用**滑动窗口**机制
- 接收方通过**rwnd（接收窗口）** 告知发送方自己还能接收多少数据
- 发送方的发送窗口 ≤ rwnd
- rwnd=0时，发送方暂停发送，但定期发送**零窗口探测报文**

---

## 八、拥塞控制

### 核心变量
- **cwnd（拥塞窗口）**：发送方维护
- **ssthresh（慢开始门限）**：初始值通常为16

### 四个算法

#### 慢开始（Slow Start）
- cwnd从1开始，**每经过一个RTT翻倍**（指数增长：1→2→4→8→16...）
- 当 cwnd ≥ ssthresh 时，转入**拥塞避免**

#### 拥塞避免（Congestion Avoidance）
- cwnd每经过一个RTT**加1**（线性增长）
- 出现超时：ssthresh = cwnd/2，cwnd=1，重新慢开始

#### 快重传（Fast Retransmit）
- 发送方连续收到**3个重复ACK**，立即重传丢失的报文段

#### 快恢复（Fast Recovery）
- 发生快重传时：ssthresh = cwnd/2，**cwnd = ssthresh**（不是1！）
- 直接进入**拥塞避免**阶段（线性增长）

### 窗口变化推导示例
> ssthresh=16，初始cwnd=1
> 慢开始：1→2→4→8→16
> 拥塞避免：17→18→19→...（加法增大）
> 假设在cwnd=24时超时：ssthresh=12, cwnd=1
> 慢开始：1→2→4→8→12
> 拥塞避免：13→14→...
> 假设在cwnd=16时收到3个重复ACK：ssthresh=8, cwnd=8
> 拥塞避免：9→10→...

## 十、典型例题

### 例题1：三次握手
> TCP建立连接时，若服务器收到SYN报文后，发送SYN+ACK但该报文丢失，此时双方状态？

**解**：客户端重传SYN（超时），服务器仍在SYN_RCVD。若多次超时，服务器关闭连接。

### 例题2：拥塞控制
> 初始ssthresh=32，cwnd从1开始慢开始，若在cwnd=40时超时，求超时后的ssthresh和cwnd变化序列。

**解**：ssthresh=40/2=20，cwnd=1→2→4→8→16→20（慢开始到20）→21→22→...（拥塞避免）

### 例题3：快重传快恢复
> ssthresh=16，cwnd=20时收到3个重复ACK。

**解**：ssthresh=20/2=10，cwnd=10（不是1），直接进入拥塞避免：10→11→12→...

### 例题4：UDP校验和
> UDP数据报中伪首部的作用是什么？

**解**：伪首部包含源IP、目的IP等信息，仅用于校验和计算（不传输），确保UDP数据报到达正确主机和端口。

## 十一、易错点
1. 三次握手中**第三次不需要SYN=1**，只需ACK=1
2. TCP的确认是**累积确认**，ack=N表示N之前全部收到
3. 超时重传后cwnd=1重新慢开始；快重传后cwnd=ssthresh进入拥塞避免
4. 流量控制是**端到端**（接收方控制），拥塞控制是**全局**（网络状态）
5. UDP是面向**报文**，TCP是面向**字节流**

### 传输层功能的深入理解

**端口的作用**：
1. 通过"端口号"标识本主机的一个特定进程
   - **每台主机的端口号是相互独立的**（端口号为主机内部的资源，只要一台主机内部不重复就可）
   - **TCP、UDP两种协议的端口号是相互独立的**（一台主机的TCP和UDP可能会使用同一个端口号）
2. TCP或UDP协议，通过**Socket套接字 = {IP地址 : 端口号}**，唯一地标识网络中的一台主机上的一个应用进程
3. 端口号的分类（只是一种建议标准，并非强制执行）
   - 服务器使用的端口号：熟知端口号0~1023（通常用于被熟知的重要应用程序）、登记端口号1024~49151
   - 客户端使用的端口号：短暂端口号49152~65535

**传输层的四大功能**：
1. **实现端到端的通信**
2. **复用和分用**：
   - 复用（从上到下）：在发送数据的时候，同一台主机上的多个进程可以使用同一个传输层协议
   - 分用（从下到上）：在接收数据的时候，传输层可以把数据正确交付到目的进程
3. **差错检测**：
   - **TCP**检查出错后就丢弃数据，并**通知发送方重传**
   - **UDP**检查出错后直接丢弃数据，**不通知发送方重传**
4. **向应用层提供两种服务**：
   - **面向连接的、可靠的**端到端传输服务（**TCP**）：确保数据正确/完整，但开销大、实时性较差
   - **无连接的、不可靠的**端到端的服务（**UDP**）：数据可能出错/丢失，但是速度快、开销小

**面向连接与无连接的区别**：
- **面向连接**：传输前先打招呼，先**确认对方已经准备好接收数据**，传输结束也要**通知对方已结束**
- **无连接**：不打招呼，直接把数据传给对方
- **可靠传输**：接收方使用"确认机制"
- **不可靠传输**：接收方无论何种情况都不会给发送方反馈

### TCP与UDP区别的补充

| 比较项 | UDP | TCP |
|--------|-----|-----|
| **报文拆分** | 不支持报文拆分、重装，应用层的报文长度不能超过UDP协议上限 | 支持报文拆分、重装，应用层的报文较长也可以 |
| **首部大小** | 很小，只占8B | 更大，20~60B |
| **连接与可靠性** | 无连接的、不可靠的、不支持流量控制和拥塞控制（**可靠性可以交给应用层处理**） | 有连接的、可靠的、支持流量控制和拥塞控制 |
| **传输模式** | 支持一对一（封装成单播IP数据报）、一对多传输（封装成广播/多播IP数据报） | 仅支持一对一传输（因为通信双方的传输层必须先建立连接） |
| **数据形式** | 面向报文 | 面向字节流 |

### UDP校验的补充

**UDP校验——发送方**：
1. 传输层的UDP协议在计算检验和之前，先添加**伪首部**
2. 把伪首部、首部、数据部分以16bit为一组，每组进行二进制加法（最高位产生的进位需要回卷）
3. 将最终的加法结果逐位取反，就得到16bit检验和，将其填入UDP首部
4. 去掉伪首部，并将UDP数据报交给网络层，封装成IP数据报

**UDP校验——接收方**：
1. 网络层向传输层递交UDP数据报
2. 传输层在UDP数据报之前，添加伪首部
3. 把伪首部、UDP首部、数据部分以16bit为一组，进行二进制加法（最高位产生的进位需要回卷）
4. 如果加法结果为**全1**，说明没有比特错误，于是接收该UDP数据报，并根据目的端口号，向应用层递交报文
5. 如果加法结果不是全1，说明有差错，于是丢弃该UDP数据报

**重要结论**：
- **UDP校验和校验内容为：伪首部、UDP首部、用户数据**
- **伪首部包括IP分组报头的一部分**

### TCP报文段的补充

**TCP报文段重要字段详解**：

| 字段 | 说明 |
|------|------|
| **序号seq** | 用于标记**TCP数据部分**第一个字节在原始字节流的位置，每个字节对应一个序号。**起始序号**是发送方自己设置的，**不一定从0开始** |
| **确认号ack_seq** | 用于反馈，表示序号**在该确认号之前的所有字节都已正确收到**，希望收到下一个的seq |
| **数据偏移** | 标识TCP报文首部长度，以×4为单位 |
| **URG** | URG=1时，紧急指针有效，表示这是紧急数据，应尽快插队发送 |
| **ACK** | 当ACK=0时，ack_seq无效；当ACK=1时，ack_seq有效 |
| **PSH** | PSH=1时，表示接收方尽快回复（用于**交互式**通信） |
| **RST** | RST=1时，表示出现严重差错（如主机崩溃），必须释放连接，也可用于拒绝一个非法报文段 |
| **SYN** | SYN=1时，表示这是一个**连接请求或连接接收**报文 |
| **FIN** | FIN=1时，此报文段的发送方的数据已发送完毕，要求释放传输连接 |
| **窗口rwnd** | 表示接收窗口的大小，即从报文段首部中的ack_seq算起，接收方还能接受多少数据（以**字节为单位**） |

**标志位总结**：
- 若SYN=1，可称此报文段为SYN段（**只有握手1、握手2的SYN=1，其余都为0**）
- 若FIN=1，可称此报文段为FIN段（**只有挥手1、挥手3的FIN=1，其余都为0**）
- 若ACK=1，可称此报文段为ACK段（**只有握手1的ACK=0，其余都为1**）

### TCP连接管理的补充

**重要规则**：
- 建立连接，必须是**客户端**先进行握手
- 释放连接，可以是客户端也可以是服务器先进行挥手

**最大段长MSS**：TCP可传输的最大有效数据长度（通常MSS不会设置太大，以免在IP层被分片）

**每次建立TCP连接，可以传输多个报文（双向）**。TCP是面向字节流的，无论传输多少报文，在TCP看来都是一连串字节流。

**建立连接耗时分析**：
- 从发出握手1，到**客户端**进程可以发送数据，至少需要多久？**1RTT**
- 从发出握手1，到**服务器**进程可以发送数据，至少需要多久？**1.5RTT**

**释放连接耗时**：
- 从客户发出挥手①，到**客户端**进程进入CLOSE状态，至少需要多久？**1RTT + 2MSL**
- 从客户发出握手①，到**服务器**进程进入CLOSE状态，至少需要多久？**1.5RTT**
- 如果服务器进程收到挥手①时，已经没有待传送数据，那么可以连续发出挥手②、挥手③

**TIME-WAIT状态**：
- 最长报文段寿命MSL：是由TCP协议规定的一个固定时间长度
- 客户进程收到挥手③后，立即进入TIME-WAIT状态，并启动"TIME-WAIT计时器"，倒计时2MSL后才能进入CLOSE状态
- 如果等待期间重复收到挥手③就重置计时器（防止挥手4没有发到服务器，引起了挥手③的重发）

### TCP可靠传输的补充

**发送缓冲区机制**：
- 当发送方收到ACK段后，将发送缓冲区中的数据左移（溢出即删除），然后把剩余位置放入待发送的数据
- 一个端口可以建立多个TCP连接
- 发送端会根据服务器返回的rwnd，实时调整发送窗口

**超时重传机制**：每发出一个报文段，就设置超时重传计时器

### 拥塞控制的补充

**流量控制 vs 拥塞控制**：
- **流量控制**：控制**端到端**的数据发送量，是"**局部的**"
- **拥塞控制**：控制**整个网络**中每台主机的数据发送量，降低路由器负载，是"**全局的**"

**如何判断网络拥塞**：
- 发出的每个报文段，都能顺利地收到ACK确认——**不拥塞**
- 发出的报文段未能按时收到ACK，引发**超时重传**——**严重拥塞**
- 收到冗余ACK，引发**快重传**——**有点拥塞**

**如果检测到网络拥塞怎么办**：
- **迅速减少发送的数据量**
- **严重拥塞**就**迅速缩小**拥塞窗口（cwnd）
- **有点拥塞**就**适当缩小**拥塞窗口

**发送窗口的上限值 = min[rwnd, cwnd]**

**拥塞控制考题特点**：
1. 常与"流量控制"综合考察，此时需要考虑接收窗口大小
2. 通常只涉及单向传输（TCP连接双方只有一方发送数据）
3. **通常默认每个TCP报文段都以最大段长MSS满载数据**
4. **拥塞窗口的大小常以"MSS的倍数"作为单位**
5. 接收方收到一个报文段，会"立即确认"，而不是"推迟确认"（会导致冗余ACK，快重传）

**慢开始算法**：cwnd值从1开始，每收到一个ACK，就让cwnd+1（翻倍也不会超过ssthresh）
- 当cwnd < ssthresh时适用
- 当严重拥塞的时候，迅速降低cwnd

**拥塞避免算法**：在一个RTT内，即使收到多个ACK，也只能让cwnd+1
- 当cwnd ≥ ssthresh时适用
- 在不拥塞时，调大cwnd

**超时重传时的处理**：
- ssthresh = 超时时的cwnd / 2（ssthresh不能小于2）
- cwnd = 1
- **超时重传时，默认是第0个RTT，此时cwnd=1**

**快恢复算法**（快重传机制）：
- **快重传**：当发送方收到三个确认号相同的冗余ACK时，立即重传对应报文段
- **快恢复算法**：一旦发生快重传，就将阈值、cwnd都设为当前cwnd的一半，然后切换为"拥塞避免算法"

**重要结论**：
- **为保证数据传输的可靠性，TCP采用了对报文段确认的机制**
- **TCP中滑动窗口的值设置过大，对主机的影响：由于传送的数据过多而使路由器变得拥挤、主机可能丢失分组**
- **主机1的x端口与主机2的y端口已经建立了一个TCP连接，若想在这两个端口之间建立一个TCP连接，则会建立失败，不影响先建立连接的传输**
