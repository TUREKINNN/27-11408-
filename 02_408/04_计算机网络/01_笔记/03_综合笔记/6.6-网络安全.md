## 一、网络安全基本概念

### 1.1 安全威胁分类

| 威胁类型 | 含义 | 举例 |
|----------|------|------|
| **被动攻击** | 窃听/监视，不修改数据 | 流量分析、窃听 |
| **主动攻击** | 篡改、伪造、中断 | 中间人攻击、DoS、重放攻击 |

### 1.2 安全服务三要素（CIA三角）

```
        机密性 (Confidentiality)
           /\
          /  \
         /    \
        /  安全  \
       /   系统   \
      /____________\
完整性              可用性
(Integrity)    (Availability)
```

| 属性 | 含义 | 实现手段 |
|------|------|----------|
| **机密性** | 信息不被未授权者获取 | 加密技术 |
| **完整性** | 信息不被未授权篡改 | 数字签名、MAC、哈希 |
| **可用性** | 授权用户能正常访问资源 | 冗余、负载均衡、防DDoS |

> 408还会考查**认证性**（确认通信方身份）和**不可否认性**（发送方不能否认已发送的消息），这两个通过**数字签名**实现。

## 二、对称加密 vs 非对称加密

### 2.1 对称加密（Symmetric Encryption）

> **定义**：加密和解密使用**同一把密钥**。

$$C = E_K(P), \quad P = D_K(C)$$

其中 $P$ 为明文，$C$ 为密文，$K$ 为密钥。

**常见算法**：

| 算法 | 密钥长度 | 特点 |
|------|----------|------|
| DES | 56位 | 已不安全，已被AES取代 |
| 3DES | 112/168位 | 三次DES加密 |
| AES | 128/192/256位 | 当前主流标准 |
| RC4 | 可变 | 流密码，曾用于SSL |
| IDEA | 128位 | 用于PGP |

**优点**：加密速度快，适合大量数据加密
**缺点**：密钥分发困难（$n$ 个用户需要 $\binom{n}{2} = \frac{n(n-1)}{2}$ 个密钥）

### 2.2 非对称加密（Asymmetric Encryption）

> **定义**：加密和解密使用**不同的密钥**（公钥和私钥）。

$$C = E_{K_{pub}}(P), \quad P = D_{K_{priv}}(C)$$

**常见算法**：

| 算法 | 基于的数学问题 | 典型应用 |
|------|---------------|----------|
| RSA | 大整数分解 | 数字签名、密钥交换 |
| ECC | 椭圆曲线离散对数 | 移动端加密、数字签名 |
| Diffie-Hellman | 离散对数问题 | 密钥交换 |
| DSA | 离散对数问题 | 数字签名（不能加密） |

**优点**：解决了密钥分发问题（公钥可公开）
**缺点**：加密速度慢（比对称加密慢100-1000倍）

### 2.3 混合加密体制

实际应用中通常采用**混合加密**：

```
发送方：
1. 随机生成会话密钥 K（对称密钥）
2. 用对称密钥 K 加密消息 M → C = E_K(M)     （快）
3. 用接收方公钥加密会话密钥 → C_K = E_{K_pub}(K) （保护密钥）
4. 发送 C 和 C_K

接收方：
1. 用私钥解密得到会话密钥 → K = D_{K_priv}(C_K)
2. 用会话密钥解密消息 → M = D_K(C)           （快）
```

> 💡 **408高频考点**：混合加密结合了对称加密的**速度**和非对称加密的**密钥分发安全性**。

### 2.4 对比总结表

| 对比维度 | 对称加密 | 非对称加密 |
|----------|---------|-----------|
| 密钥数量 | 1个共享密钥 | 2个（公钥+私钥） |
| 加密速度 | 快 | 慢 |
| 密钥分发 | 困难 | 公钥可公开 |
| 密钥数量（n用户） | $n(n-1)/2$ | $2n$ |
| 典型应用 | 数据加密 | 密钥交换、数字签名 |
| 代表算法 | AES、DES | RSA、ECC |

## 三、数字签名原理

### 3.1 数字签名的作用

数字签名提供以下安全服务：

| 服务 | 说明 |
|------|------|
| **认证性** | 验证消息发送者的身份 |
| **完整性** | 验证消息未被篡改 |
| **不可否认性** | 发送者不能否认已发送的消息 |

### 3.2 数字签名过程

> **核心思想**：用**发送方的私钥**加密消息摘要，接收方用**发送方的公钥**验证。

```
=== 签名过程（发送方A）===

消息 M
  │
  ├──→ Hash函数 ──→ 摘要 H = Hash(M)
  │                      │
  │                      ▼
  │              用A的私钥加密摘要
  │              S = E_{A_priv}(H)   ← 这就是数字签名
  │                      │
  └──────────────────────┘
                    │
              发送 (M, S)

=== 验证过程（接收方B）===

接收到 (M, S)
  │
  ├──→ 对M计算摘要 H' = Hash(M)
  │
  ├──→ 用A的公钥解密签名 H = D_{A_pub}(S)
  │
  └──→ 比较 H' == H ?
         │
    ┌────┴────┐
   YES        NO
  验证通过    验证失败
```

### 3.3 常用Hash函数

| 算法 | 输出长度 | 特点 |
|------|----------|------|
| MD5 | 128位 | 已发现碰撞，不建议使用 |
| SHA-1 | 160位 | 已被攻破，逐步淘汰 |
| SHA-256 | 256位 | 当前主流 |

**Hash函数性质**：
- **单向性**：从摘要无法反推原文
- **抗碰撞性**：难以找到两个不同消息有相同摘要
- **雪崩效应**：原文微小变化导致摘要巨大变化

## 四、SSL/TLS协议

### 4.1 协议层次

SSL/TLS位于**传输层之上、应用层之下**，为应用层提供安全服务。

```
┌──────────────┐
│   应用层      │  HTTP, SMTP, FTP ...
├──────────────┤
│  SSL/TLS     │  ← 安全层（位于TCP之上）
├──────────────┤
│   传输层      │  TCP
├──────────────┤
│   网络层      │  IP
├──────────────┤
│  数据链路层   │
├──────────────┤
│  物理层       │
└──────────────┘
```

**典型应用**：HTTPS = HTTP + SSL/TLS（默认端口443）

### 4.2 SSL/TLS协议组成

| 子协议 | 功能 |
|--------|------|
| **握手协议（Handshake）** | 协商加密算法、交换密钥、身份认证 |
| **记录协议（Record）** | 对应用数据进行分片、压缩、加密、MAC |
| **告警协议（Alert）** | 报告错误和警告 |
| **更改密码规范协议** | 通知对方切换到新协商的加密参数 |

### 4.3 SSL握手过程（重点）

```
    客户端 (Client)                         服务器 (Server)
        │                                       │
        │──── ClientHello ─────────────────────→│
        │     (支持的加密套件列表、随机数R1)        │
        │                                       │
        │←─── ServerHello ─────────────────────│
        │     (选定的加密套件、随机数R2)            │
        │                                       │
        │←─── Certificate ─────────────────────│
        │     (服务器证书/公钥)                    │
        │                                       │
        │←─── ServerHelloDone ─────────────────│
        │     (服务器问候结束)                     │
        │                                       │
        │──── ClientKeyExchange ───────────────→│
        │     (预主密钥，用服务器公钥加密)           │
        │                                       │
        │──── [ChangeCipherSpec] ──────────────→│
        │──── Finished ────────────────────────→│
        │                                       │
        │←─── [ChangeCipherSpec] ──────────────│
        │←─── Finished ───────────────────────│
        │                                       │
        │◄══════ 加密的应用数据 ═══════════════►│
```

**握手过程详解**：

| 步骤 | 消息 | 内容 | 作用 |
|------|------|------|------|
| 1 | ClientHello | 支持的密码套件、客户端随机数R1 | 发起连接 |
| 2 | ServerHello | 选定的密码套件、服务端随机数R2 | 确认参数 |
| 3 | Certificate | 服务器X.509证书 | 身份认证 |
| 4 | ServerHelloDone | 空消息 | 表示握手消息发送完毕 |
| 5 | ClientKeyExchange | 预主密钥（用服务器公钥加密） | 密钥协商 |
| 6 | ChangeCipherSpec | 通知切换加密参数 | 协商完成 |
| 7 | Finished | 对握手消息的验证 | 验证完整性 |

**主密钥生成**：
$$MasterSecret = PRF(PreMasterSecret, \text{"master secret"}, R1 \parallel R2)$$

会话密钥从主密钥派生，用于后续数据加密。

### 4.4 SSL记录协议

```
应用数据
    │
    ▼
分片（最大 2^14 字节）
    │
    ▼
压缩（可选）
    │
    ▼
计算 MAC（消息认证码）
    │
    ▼
加密（对称加密）
    │
    ▼
添加 SSL 记录头
    │
    ▼
交给 TCP 传输
```

## 五、防火墙类型

### 5.1 包过滤防火墙（Packet Filtering）

> **工作层次**：网络层（第3层）

| 特性 | 说明 |
|------|------|
| **过滤依据** | 源IP、目的IP、协议类型、源端口、目的端口 |
| **工作方式** | 逐包检查，根据规则表决定允许/丢弃 |
| **优点** | 速度快、对用户透明 |
| **缺点** | 不能检查数据内容、不能跟踪连接状态、易受IP欺骗 |

### 5.2 状态检测防火墙（Stateful Inspection）

> **工作层次**：传输层（第4层）

| 特性 | 说明 |
|------|------|
| **过滤依据** | 包信息 + **连接状态表** |
| **工作方式** | 跟踪TCP连接状态（SYN、ESTABLISHED、FIN等） |
| **优点** | 安全性高于包过滤，能识别非法连接 |
| **缺点** | 性能开销较大 |

**示例**：内部主机向外部发送SYN请求，状态表记录该连接。只有对应的SYN-ACK回复才被允许通过，未匹配的SYN-ACK被拒绝。

### 5.3 应用网关/代理防火墙（Application Gateway）

> **工作层次**：应用层（第7层）

| 特性 | 说明 |
|------|------|
| **过滤依据** | 应用层数据内容 |
| **工作方式** | 完全解析应用协议，充当代理转发 |
| **优点** | 安全性最高，能过滤应用层内容 |
| **缺点** | 速度最慢，每种应用需要专门的代理程序 |

### 5.4 三种防火墙对比

| 对比维度 | 包过滤 | 状态检测 | 应用网关 |
|----------|--------|---------|---------|
| **工作层次** | 网络层 | 传输层 | 应用层 |
| **检查内容** | IP头部 | IP+TCP头部+状态 | 完整应用数据 |
| **安全性** | 低 | 中 | 高 |
| **速度** | 快 | 中 | 慢 |
| **透明性** | 透明 | 透明 | 不透明（需代理） |
| **连接跟踪** | 无 | 有 | 有 |

## 六、IPSec协议

### 6.1 概述

> **IPSec（IP Security）**：在网络层为IP数据报提供安全服务的协议族。

**提供的安全服务**：
- 数据机密性（加密）
- 数据完整性（认证）
- 数据源认证（身份验证）
- 防重放攻击（序列号）

### 6.2 IPSec的两种协议

#### AH（Authentication Header，认证头）

| 特性 | 说明 |
|------|------|
| **协议号** | 51 |
| **提供的服务** | 数据完整性 + 数据源认证 + 防重放 |
| **不提供** | ❌ 机密性（不加密） |
| **保护范围** | 包括IP头部（除可变字段外） |

#### ESP（Encapsulating Security Payload，封装安全载荷）

| 特性 | 说明 |
|------|------|
| **协议号** | 50 |
| **提供的服务** | 数据完整性 + 数据源认证 + 防重放 + **机密性** |
| **保护范围** | 仅保护ESP载荷部分（不保护外部IP头） |

### 6.3 两种协议对比

| 对比维度 | AH | ESP |
|----------|-----|-----|
| 机密性 | ❌ | ✅ |
| 完整性 | ✅ | ✅ |
| 数据源认证 | ✅ | ✅ |
| 防重放 | ✅ | ✅ |
| 保护IP头 | ✅（除可变字段） | ❌ |

> ⚠️ **408易混淆点**：AH保护IP头部但不加密；ESP加密但不保护外部IP头部。

### 6.4 两种工作模式

#### 传输模式（Transport Mode）

```
┌──────────┬──────────┬──────────┐
│ IP头部   │ IPSec头  │  原始数据  │  ← 不改变IP头
│（原始）   │(AH/ESP)  │   载荷    │
└──────────┴──────────┴──────────┘
```

| 特性 | 说明 |
|------|------|
| **应用场景** | 主机到主机（端到端）通信 |
| **IP头部** | 保持不变 |
| **保护对象** | 仅保护上层数据 |
| **典型用途** | 局域网内主机间安全通信 |

#### 隧道模式（Tunnel Mode）

```
┌──────────┬──────────┬──────────┬──────────┐
│ 新IP头部  │ IPSec头  │ 原IP头部  │  原始数据  │
│（外部）   │(AH/ESP)  │（内部）   │   载荷    │
└──────────┴──────────┴──────────┴──────────┘
```

| 特性 | 说明 |
|------|------|
| **应用场景** | 网关到网关（VPN）通信 |
| **IP头部** | 添加新的外部IP头，原始IP头被封装 |
| **保护对象** | 保护整个原始IP数据报 |
| **典型用途** | 站点间VPN |

### 6.5 两种模式对比

| 对比维度 | 传输模式 | 隧道模式 |
|----------|---------|---------|
| **IP头部** | 不变 | 新增外部IP头 |
| **应用场景** | 端到端 | 网关到网关（VPN） |
| **保护范围** | 仅上层数据 | 整个原始IP包 |
| **性能开销** | 小 | 较大（额外IP头） |
| **隐蔽性** | 源目IP可见 | 内部IP被隐藏 |

### 6.6 安全关联（SA）

> **安全关联（Security Association）**：IPSec的核心概念，是两个通信实体之间的一个单向"安全通道"。

SA由三元组唯一标识：
$$SA = (SPI, \text{目的IP地址}, \text{安全协议标识})$$

| 参数 | 含义 |
|------|------|
| **SPI** | 安全参数索引，32位标识符 |
| **目的IP** | SA终点的IP地址 |
| **协议标识** | AH或ESP |

> **注意**：SA是**单向**的，双向通信需要两个SA。

## 七、典型例题

### 例题1：加密算法分类

**题目**：以下加密算法中，属于非对称加密算法的是（ ）。

A. AES    B. DES    C. RSA    D. IDEA

**答案**：C

**解析**：AES、DES、IDEA均为对称加密算法。RSA基于大整数分解问题，是典型的非对称加密算法。

### 例题2：数字签名

**题目**：在数字签名过程中，发送方使用（ ）对消息摘要进行加密。

A. 发送方公钥    B. 发送方私钥    C. 接收方公钥    D. 接收方私钥

**答案**：B

**解析**：数字签名使用**发送方的私钥**对摘要加密，接收方使用发送方的公钥验证。这与数据加密相反（数据加密用接收方公钥加密）。

> **记忆口诀**："签名私钥加密，验签公钥解密；数据公钥加密，私钥解密还原"

### 例题3：SSL/TLS

**题目**：在SSL/TLS握手过程中，用于生成主密钥的预主密钥是由（ ）加密传输的。

A. 客户端使用服务器的公钥
B. 客户端使用服务器的私钥
C. 服务器使用客户端的公钥
D. 服务器使用客户端的私钥

**答案**：A

**解析**：客户端生成预主密钥，使用**服务器的公钥**（来自服务器证书）加密后发送给服务器。服务器用私钥解密得到预主密钥，双方据此生成相同的主密钥和会话密钥。

### 例题4：防火墙类型

**题目**：能够检查应用层数据内容的防火墙是（ ）。

A. 包过滤防火墙    B. 状态检测防火墙
C. 应用网关防火墙    D. 以上都不能

**答案**：C

**解析**：包过滤防火墙工作在网络层，只检查IP头部；状态检测防火墙工作在传输层，跟踪连接状态；只有应用网关防火墙工作在应用层，能检查完整的应用层数据内容。

### 例题5：IPSec

**题目**：在IPSec协议中，既能提供数据完整性又能提供机密性服务的是（ ）；工作在隧道模式下适合构建VPN的是（ ）。

A. AH协议    B. ESP协议    C. 两者都可以    D. 两者都不可以

**答案**：B；B

**解析**：
- AH只提供完整性认证，不提供机密性（不加密）
- ESP既提供完整性又提供机密性
- 隧道模式是IPSec VPN的常用工作模式，ESP在隧道模式下既加密又认证

### 例题6：IPSec模式（2024年改编）

**题目**：说明IPSec传输模式和隧道模式的区别，并指出哪种模式适合构建站点间VPN。

**解答**：

| 对比项 | 传输模式 | 隧道模式 |
|--------|---------|---------|
| IP头部处理 | 不修改原始IP头 | 添加新的外部IP头，封装原始IP包 |
| 保护范围 | 仅保护上层协议数据 | 保护整个原始IP数据报 |
| 应用场景 | 主机到主机的端到端通信 | 网关到网关的VPN通信 |

**隧道模式**适合构建站点间VPN，原因：
1. 隧道模式隐藏了内部网络的IP地址结构
2. 可以封装任意IP数据报，支持整个子网间的通信
3. 外部IP头仅需指定两端VPN网关地址

## 八、易错点

> - **数字签名**：发送方**私钥**加密摘要 → 提供认证性、不可否认性
> - **数据加密**：接收方**公钥**加密数据 → 提供机密性
> - 两者密钥使用方向相反，切勿混淆！

> - AH能保护IP头但**不加密**数据
> - ESP**加密**数据但不保护外部IP头
> - 题目问"提供机密性"→ 必选ESP

> - 传输模式：不改IP头，主机到主机
> - 隧道模式：加新IP头，网关到网关（VPN）
> - 记忆："隧道=加壳=VPN"

> - SSL/TLS在**传输层之上、应用层之下**
> - 不是网络层协议，也不是应用层协议
> - HTTPS = HTTP + SSL/TLS，不是HTTP + IPSec

> - 一个SA只提供一个方向的安全服务
> - 双向通信需要建立**两个**SA
> - 题目中经常忽略这一点
