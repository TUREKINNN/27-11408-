## 1️⃣ 生产者-消费者问题

### 📌 问题描述

> 多个生产者往**容量 N 的缓冲区**放产品，多个消费者从缓冲区取产品。  
> 缓冲区满时生产者等待，缓冲区空时消费者等待。

### 📊 信号量定义

```c
semaphore mutex = 1;    // 互斥访问缓冲区
semaphore empty = N;    // 缓冲区空位数
semaphore full = 0;     // 缓冲区已占用位
```

### 🔁 生产者进程

```
┌──────────────┐
│   Producer   │
├──────────────┤
│ P(empty)     │ ← 申请空位
│ P(mutex)     │ ← 互斥锁
│ 放入产品     │ ← 临界区
│ V(mutex)     │ ← 释放锁
│ V(full)      │ ← 增加产品数
└──────────────┘
```

### 🔁 消费者进程

```
┌──────────────┐
│   Consumer   │
├──────────────┤
│ P(full)      │ ← 申请产品
│ P(mutex)     │ ← 互斥锁
│ 取出产品     │ ← 临界区
│ V(mutex)     │ ← 释放锁
│ V(empty)     │ ← 增加空位数
└──────────────┘
```

### ⚠️ 易错点

> 原因：若先 P(mutex) 后 P(empty)，缓冲区满时 P(empty) 阻塞，但 mutex 仍被占用，导致**死锁**。

## 2️⃣ 读者-写者问题

### 📌 问题描述

> 多个读者可**同时读**共享数据，但写者必须**独占**访问。  
> 读者优先级 ≥ 写者（写者可能饥饿）。

### 📊 信号量定义

```c
semaphore rmutex = 1;   // 保护 rcount 的互斥锁
semaphore wmutex = 1;   // 写者互斥锁
int rcount = 0;         // 读者计数
```

### 🔁 读者进程

```
┌────────────────┐
│     Reader     │
├────────────────┤
│ P(rmutex)      │ ← 锁住 rcount
│ rcount++       │ ← 增加读者数
│ if (rcount==1) │
│   P(wmutex)    │ ← 第一个读者锁写者
│ V(rmutex)      │ ← 释放 rcount 锁
│                │
│ 【读操作】     │ ← 可多个读者同时读
│                │
│ P(rmutex)      │
│ rcount--       │
│ if (rcount==0) │
│   V(wmutex)    │ ← 最后一个读者释放写者
│ V(rmutex)      │
└────────────────┘
```

### 🔁 写者进程

```
┌────────────────┐
│     Writer     │
├────────────────┤
│ P(wmutex)      │ ← 锁住写者
│ 【写操作】     │ ← 独占
│ V(wmutex)      │
└────────────────┘
```

### ⚠️ 易错点

> 解决方案：加一个 `wcount` 计数（记录等待的写者），或使用**读写公平锁**。

## 3️⃣ 哲学家进餐问题

### 📌 问题描述

> 5 个哲学家围坐，桌上有 5 根筷子，哲学家**思考/进餐**循环。  
> 进餐需要**左右两边的筷子**（每个哲学家独占一根筷子）。

### 📊 信号量定义

```c
semaphore chopstick[5] = {1,1,1,1,1};  // 5 根筷子
```

### 🔁 哲学家 i 进程（基础版 → 死锁！）

```
┌──────────────────┐
│ Philosopher i    │
├──────────────────┤
│ P(chopstick[i])  │ ← 拿左边筷子
│ P(chopstick[(i+1)%5]) │ ← 拿右边筷子
│ 【进餐】         │
│ V(chopstick[i])  │ ← 放左边筷子
│ V(chopstick[(i+1)%5]) │ ← 放右边筷子
│ 【思考】         │
└──────────────────┘
```

### 🚨 死锁原因


### ✅ 解决方案（3 种）

**方案 1：限制同时进餐人数 ≤ 4**

```c
semaphore count = 4;  // 最多 4 人进餐

Philosopher i:
  P(count);
  P(chopstick[i]);
  P(chopstick[(i+1)%5]);
  【进餐】
  V(chopstick[i]);
  V(chopstick[(i+1)%5]);
  V(count);
  【思考】
```

**方案 2：奇偶哲学家拿筷子顺序不同**

```
奇数号哲学家：先左后右
偶数号哲学家：先右后左
```

**方案 3：AND 型信号量（一次性锁多个）**

```c
Swait(chopstick[i], chopstick[(i+1)%5]);  // 原子操作
【进餐】
Ssignal(chopstick[i], chopstick[(i+1)%5]);
```

## 4️⃣ 吸烟者问题

### 📌 问题描述

> 3 个吸烟者 + 1 个供应者。  
> 每个吸烟者需要**两种原料**才能卷烟（自己有第三种）。  
> 供应者每次放出**两种原料**，持有对应原料的吸烟者卷烟抽掉。

### 📊 信号量定义

```c
semaphore offer1 = 0;  // 原料 1+2 组合
semaphore offer2 = 0;  // 原料 2+3 组合
semaphore offer3 = 0;  // 原料 3+1 组合
semaphore finish = 0;  // 完成卷烟
```

### 🔁 供应者进程

```
┌──────────────────┐
│    Provider      │
├──────────────────┤
│ while(1) {       │
│   随机提供两种   │
│   V(offerX);     │ ← 提供对应组合
│   P(finish);     │ ← 等待吸烟者完成
│ }                │
└──────────────────┘
```

### 🔁 吸烟者 i 进程

```
┌──────────────────┐
│   Smoker i       │
├──────────────────┤
│ P(offerX);       │ ← 等待对应原料组合
│ 【卷烟 + 抽】    │
│ V(finish);       │ ← 通知供应者完成
└──────────────────┘
```

## 🎯 PV 操作·答题模板

```c
semaphore S1 = 初值1;   // 互斥锁
semaphore S2 = 初值2;   // 同步信号

P(S);   // P操作 = wait(S) = S--
V(S);   // V操作 = signal(S) = S++

// 必须成对出现：
// 1. 互斥 P/V 在同一进程
// 2. 同步 P/V 在不同进程（先 P 后 V）
// 3. P 操作顺序：先同步后互斥
```

---

## ⚠️ 8 大易错点

> 1. ❌ P 顺序反了（同步 P 应在互斥 P 之前）
> 2. ❌ V 顺序无所谓（释放顺序不影响正确性）
> 3. ❌ 漏掉 V 操作（导致信号量永远 ≤ 0）
> 4. ❌ 多余的 P 操作（导致不必要的阻塞）
> 5. ❌ 互斥 P 嵌套死锁（多个 mutex 顺序不一致）
> 6. ❌ 计数信号量忘加 1
> 7. ❌ AND 型信号量忘了"全部满足才放行"
> 8. ❌ 哲学家问题的"每人拿左"陷阱
