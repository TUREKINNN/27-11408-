### 1.1 操作系统的基本概念

#### 1.1.1 操作系统的定义与功能

> **操作系统（Operating System, OS）**：控制和管理整个计算机系统的**硬件与软件资源**，合理地组织计算机工作流程，为用户和应用程序提供服务的**程序集合**。它是计算机系统中最基本的**系统软件**，是用户与计算机硬件之间的**接口**，也是计算机系统资源的管理者。

**从三个维度理解操作系统**：

| 维度 | 描述 | 通俗理解 |
|:------|:------|:----------|
| **系统资源的管理者** | 控制和管理整个计算机系统的硬件和软件资源 | "**管家**"——管理 CPU、内存、设备等 |
| **向上层提供方便易用的服务** | 把复杂的硬件功能封装成简单易用的服务 | "**翻译官**"——把硬件指令翻译成用户能理解的操作 |
| **最接近硬件的层次** | 对硬件机器的拓展，将裸机改造成功能更强的机器 | "**改装师**"——让硬件能协调工作 |

> 一台电脑的诞生流程：**裸机（纯硬件）→ 安装操作系统 → 安装应用程序 → 用户使用**。没有任何软件支持的计算机称为**裸机**，覆盖了软件的机器称为**扩充机器**或**虚拟机**。

**操作系统的五大功能**：

| 功能 | 详细说明 |
|:-----|:---------|
| **处理机管理（进程管理）** | 进程控制、进程同步、进程通信、进程调度——核心是 CPU 资源的分配与回收 |
| **存储器管理（内存管理）** | 内存分配与回收、地址映射、内存保护与共享、内存扩充（虚拟内存） |
| **文件管理** | 文件存储空间管理、目录管理、文件读写管理与保护 |
| **设备管理** | 缓冲管理、设备分配、设备处理（驱动）、I/O 调度 |
| **用户接口** | 向用户提供命令接口、程序接口（系统调用）、图形用户接口（GUI） |

> **"进存文设用"** —— 处理器（进）、存储器（存）、文件（文）、设备（设）、用户接口（用）

> - **Step 1**：在文件夹中找到 QQ 安装位置 → **文件管理**
> - **Step 2**：双击打开 QQ.exe，将相关数据放入内存 → **存储器管理**（内存分配）
> - **Step 3**：QQ 进程被 CPU 处理 → **处理机管理**（进程调度）
> - **Step 4**：使用摄像头进行视频通话 → **设备管理**（设备分配）

#### 1.1.2 操作系统的特征

操作系统有四大基本特征：**并发**、**共享**、**虚拟**、**异步**。其中**并发和共享是最基本的两个特征**，二者互为存在条件。

**（1）并发（Concurrence）**

| 概念 | 定义 | 要求 |
|:------|:------|:------|
| **并发** | 两个或多个事件在**同一时间间隔**内发生 | 单处理器即可实现（宏观并行，微观串行） |
| **并行** | 两个或多个事件在**同一时刻**同时发生 | 需要多处理器/多核 |

> **并发 ≠ 并行**。并发是交替执行（逻辑上的同时），并行是物理上的同时。
>
> - **单核 CPU**：同一时刻只能执行一个程序，各程序只能**并发**执行
> - **多核 CPU**（如 4 核）：同一时刻最多可同时执行 4 个程序，可实现**并行**执行
> - 即使是 4 核 CPU，只要有超过 4 个程序需要"同时"运行，**并发性**依然是必不可少的
> - **操作系统伴随着"多道程序技术"出现，操作系统和程序并发一起诞生**

**（2）共享（Sharing）**

系统资源可供多个并发执行的进程共同使用。

| 共享方式 | 描述 | 生活类比 |
|:----------|:------|:----------|
| **互斥共享** | 一个时间段内**只允许一个**进程访问 | QQ 和微信同时视频——摄像头同一时刻只能分配给一个程序 |
| **同时共享** | 宏观上多进程"同时"访问（微观可能交替） | QQ 发文件 A + 微信发文件 B——都在访问硬盘（分时共享） |

**（3）虚拟（Virtual）**

通过某种技术将一个**物理上的实体**变为若干个**逻辑上的对应物**。

| 虚拟技术 | 物理实体 | 逻辑对应物 | 实现方式 |
|:----------|:----------|:-----------|:----------|
| **虚拟存储器** | 实际 4 GB 内存 | 用户看来远大于 4 GB | **空分复用技术** |
| **虚拟处理器** | 1 个单核 CPU | 用户看来有多个 CPU | **时分复用技术** |

> 没有并发性，就谈不上虚拟性。如果只有一个程序运行，虚拟技术就失去了意义。

**（4）异步（Asynchronism）**

在多道程序环境下，由于**资源有限**，进程的执行不是一贯到底的，而是**走走停停**，以**不可预知的速度**向前推进。

> 只有系统拥有并发性，才有可能导致异步性。如果系统只能串行运行各程序，每个程序的执行会一贯到底。

**四大特征之间的内在关系**：

```
并发性 ──互为存在条件──→  共享性
   │                       │
   │  失去并发性 → 共享性无意义     │
   │  失去共享性 → 无法并发       │
   │                       │
   └──┬──┘
      ↓
   虚拟性 ← 没有并发就谈不上虚拟
      ↓
   异步性 ← 只有并发才导致走走停停
```

> 1. ❌ "并发就是并行" → ✅ 并发是交替执行（同一时间间隔），并行是同时执行（同一时刻）
> 2. ❌ "OS 的最基本特征是共享" → ✅ **并发**和**共享**互为存在条件，缺一不可
> 3. ❌ "分时系统就是实时系统" → ✅ 分时强调交互性，实时强调及时性和可靠性

#### 1.1.3 操作系统提供的接口

操作系统向上层（用户/应用程序）提供两类接口：

```
操作系统提供的接口
├── 命令接口（供用户直接使用）
│   ├── 联机命令接口（交互式）
│   │    用户说一句，系统做一句
│   │    例：cmd 中的 time 命令、shell 命令行
│   └── 脱机命令接口（批处理式）
│        用户说一堆，系统做一堆
│        例：.bat 批处理文件、shell 脚本
├── 程序接口（系统调用，供应用程序使用）
│    由一系列系统调用组成
│    是应用程序请求操作系统服务的**唯一方式**
│    例：printf() 底层调用 write() 系统调用
└── 图形用户接口（GUI）
     用户通过形象的图形界面操作，无需记忆复杂命令
     例：Windows 拖拽文件到回收站（底层仍调用系统调用）
```

> **命令接口**：用户通过输入命令直接控制计算机行为的接口。
> **程序接口（系统调用）**：应用程序通过系统调用间接请求 OS 服务的接口。
> **GUI**：通过窗口、图标、菜单等图形元素操作计算机的用户接口。
>
> 联机命令接口、脱机命令接口、GUI 统称为**用户接口**。

### 1.2 操作系统的发展与分类

#### 1.2.1 操作系统的发展历程

| 阶段 | 年代 | 特点 | 关键技术 | 优缺点 |
|:------|:------|:------|:----------|:------|
| **手工操作** | 1940s | 无 OS，用户独占全机，人工插拔线路 | — | 效率极低，CPU 严重等待人工操作 |
| **单道批处理** | 1950s | 自动性、顺序性、单道性；一批作业自动切换 | 脱机 I/O 技术（卫星机） | 缓解人机速度矛盾，但 CPU 仍大量空闲 |
| **多道批处理** | 1960s | 多道程序并发执行，宏观并行、微观串行 | 中断技术、通道技术 | CPU 利用率大幅提高，但无交互能力 |
| **分时系统** | 1970s | 多路性、独立性、及时性、交互性 | 时间片轮转 | 用户可实时交互，Unix 是典型代表 |
| **实时系统** | 1970s~ | 及时性、高可靠性、确定性 | 硬实时/软实时调度 | 满足严格时间约束，用于工业控制、武器系统 |

> - **脱机 I/O 技术**：用一台外围计算机（卫星机）专门处理 I/O 操作，将慢速设备与主机分离，主机只处理计算任务。
> - **多道程序技术**：内存中同时存放多道程序，当一道程序因 I/O 阻塞时，CPU 立即切换到另一道程序执行。
> - **时间片轮转**：CPU 时间被分割为若干个时间片，每个进程轮流获得一个时间片执行。

#### 1.2.2 操作系统的分类与对比

**按使用环境分类**：

| 类型 | 核心目标 | 交互性 | 及时性 | 可靠性 | 代表系统 |
|:------|:----------|:------|:------|:------|:----------|
| **批处理系统** | 吞吐量最大化 | 无 | 不要求 | 一般 | IBM OS/360 |
| **分时系统** | 公平响应多用户 | 强 | 较短响应 | 一般 | Unix、Linux 服务器 |
| **实时系统** | 满足时间约束 | 有限 | 极强（硬实时）| 极高 | VxWorks、RT-Linux |
| **网络操作系统** | 网络资源共享 | 有 | 不要求 | 较高 | Windows Server |
| **分布式操作系统** | 多机协同，透明性 | 有 | 不要求 | 高 | Amoeba、Plan 9 |
| **个人计算机 OS** | 用户友好 | 强 | 不要求 | 一般 | Windows、macOS |

**实时系统的进一步分类**：

| 类型 | 含义 | 例子 |
|:------|:------|:------|
| **硬实时系统** | 必须在**绝对严格**的时间期限内完成任务，否则造成严重后果 | 导弹控制系统、自动驾驶 |
| **软实时系统** | 偶尔违反时间期限可以接受，不会造成致命后果 | 视频播放、在线会议 |

**按用户数量分类**：

| 类型 | 描述 | 例子 |
|:------|:------|:------|
| **单用户单任务** | 一个用户，一次只能运行一个任务 | MS-DOS |
| **单用户多任务** | 一个用户，同时可运行多个任务 | Windows、macOS |
| **多用户多任务** | 多个用户同时使用，各自可运行多个任务 | Linux、Unix |

> 1. **分时系统** ≠ **实时系统**。分时系统的核心目标是**公平交互**，实时系统的核心目标是**时间约束**。
> 2. **网络 OS** ≠ **分布式 OS**。网络 OS 中用户能感知到网络的存在（需显式指定远程资源）；分布式 OS 中用户感觉不到多台机器的存在（透明性）。

### 1.3 操作系统的运行环境

#### 1.3.1 处理器的运行模式

> 现代 CPU 通常支持至少两种运行模式，以保护操作系统不被用户程序破坏：
> - **内核态（核心态 / 管态 / 内核模式 / Kernel Mode）**：CPU 运行在最高特权级，可执行**所有指令**（包括特权指令），可访问所有内存和硬件资源。
> - **用户态（目态 / 用户模式 / User Mode）**：CPU 运行在受限模式，只能执行**非特权指令**，只能访问受限地址空间。

**x86 架构特权级（Ring 0 ~ Ring 3）**：

| 级别 | 运行内容 | 说明 |
|:------|:----------|:------|
| **Ring 0** | 操作系统内核 | 最高特权级，内核态 |
| **Ring 1** | 设备驱动程序（部分 OS） | 未在主流 OS 中使用 |
| **Ring 2** | 设备驱动程序（部分 OS） | 未在主流 OS 中使用 |
| **Ring 3** | 用户应用程序 | 最低特权级，用户态 |

> 现代操作系统（Linux、Windows）实际上只使用 **Ring 0（内核态）** 和 **Ring 3（用户态）**。

**特权指令与非特权指令**：

| 类型 | 说明 | 例子 |
|:------|:------|:------|
| **特权指令** | 只能在内核态执行的指令，涉及系统关键资源 | I/O 指令、置中断屏蔽指令、存取保护寄存器、清内存指令、修改页表、停机指令 |
| **非特权指令** | 用户态可执行的普通指令 | 算术运算指令、逻辑运算指令、数据传输指令、访管指令（陷入指令） |

> 访管指令（陷入指令 / trap / INT / syscall）是一条**非特权指令**，可在用户态执行，执行后 CPU 进入内核态。它是用户程序**进入内核态的唯一途径**。

**内核态与用户态的切换机制**：

```
用户态 → 内核态（三种触发方式）：
  ① 系统调用（用户程序主动请求 OS 服务）
  ② 中断（外部设备发出的中断信号）
  ③ 异常（CPU 执行指令时产生的错误/陷阱）
     ——以上统称为"中断/异常机制"，是进入内核态的唯一途径

内核态 → 用户态：
  通过设置程序状态字寄存器（PSW），将 CPU 模式位改为用户态
```

> - 用户态 → 内核态：只能通过**中断/异常**进入
> - 内核态 → 用户态：通过**修改 PSW** 返回
> - **进程切换**必须在**内核态**完成（需要修改 PCB、保存/恢复上下文等特权操作）

#### 1.3.2 中断与异常

> **中断（广义）**：CPU 对系统发生的某个事件做出的一种反应，暂停当前程序，转而处理该事件，处理完毕后返回断点继续执行。
>
> 408 考纲采用**狭义分类**：中断 = 外中断；异常 = 内中断。

**中断的完整分类**：

```
中断（广义）
├── 中断（外中断 / 硬中断 / Interrupt）
│    来源：CPU 外部设备
│    与当前指令无关，异步发生
│   ├── 可屏蔽中断（INTR）
│   │    可通过中断屏蔽寄存器屏蔽
│   │    例：I/O 设备中断（键盘、鼠标、磁盘）
│   └── 不可屏蔽中断（NMI）
│        不可被屏蔽，必须立即响应
│        例：电源掉电、内存奇偶校验错
│
└── 异常（内中断 / 陷入 / Exception）
     来源：CPU 内部
     与当前指令有关，同步发生
    ├── 故障（Fault）
    │    可恢复的异常，处理后重新执行该指令
    │    例：缺页异常、除零错误、缺段异常
    ├── 陷阱（Trap）
    │    有意安排的"异常"，用于系统调用
    │    例：int 0x80（Linux 系统调用）、syscall 指令
    └── 终止（Abort）
         不可恢复的致命错误
         例：硬件错误、双故障（Double Fault）
```

**内中断 vs 外中断 核心对比**：

| 比较项 | 中断（外中断） | 异常（内中断） |
|:--------|:---------------|:---------------|
| **来源** | CPU **外部**（I/O 设备、时钟） | CPU **内部**（指令执行） |
| **触发时机** | 与**当前指令无关**，随时可能发生 | 与**当前指令有关**，同步发生 |
| **可否屏蔽** | 多数可屏蔽（除 NMI） | **不可屏蔽**（由当前指令引起，必须处理） |
| **处理方式** | 内核态响应，处理后返回下一条指令 | 根据类型：故障重新执行当前指令，陷阱执行下一条指令 |
| **典型例子** | I/O 完成中断、时钟中断、DMA 中断 | 除零错误、缺页异常、非法指令、断点、系统调用 |

> 1. ❌ "异常可以屏蔽" → ✅ 异常不可屏蔽（由当前指令引起，必须处理）
> 2. ❌ "系统调用是外中断" → ✅ 系统调用通过**陷入指令**触发，属于**内中断（陷阱）**
> 3. ❌ "缺页异常是外中断" → ✅ 缺页异常是 CPU 执行访存指令时产生的**内中断（故障）**

**中断处理过程（完整十步）**：

```
① 关中断（CPU 设置中断屏蔽位，防止新中断打断当前处理）
② 保存断点（硬件自动将 PC 和 PSW 压入内核栈）
③ 识别中断源（硬件查询中断控制器，获取中断向量号）
④ 查中断向量表，获取中断服务程序入口地址
⑤ 保存现场（将通用寄存器等压入内核栈，软件完成）
⑥ 保存屏蔽字（如果需要中断嵌套）
⑦ 开中断（允许更高优先级的中断嵌套进入）
⑧ 执行中断服务程序（核心处理逻辑）
⑨ 关中断（为恢复现场做准备）
⑩ 恢复现场和屏蔽字 → 开中断 → 执行 IRET 指令返回断点
```

> **中断向量表**：一张存储在内存固定位置的表，以**中断向量号**为索引，每个表项存储对应**中断服务程序的入口地址**。中断向量号由中断控制器（如 8259A PIC / APIC）提供。

**中断隐指令**（硬件自动完成的操作，不需要编程）：
- 关中断
- 保存断点（PC + PSW）
- 识别中断源，获取中断向量号

#### 1.3.3 系统调用

> **系统调用（System Call）**：操作系统提供给应用程序的**程序接口**，是用户程序取得操作系统服务的**唯一途径**。应用程序通过系统调用来请求操作系统内核代为完成某些关键操作。

**为什么需要系统调用？**
- 用户程序不能直接访问硬件资源（磁盘、网络等）
- 用户程序不能执行特权指令
- 系统调用提供受控的、安全的内核服务访问通道

**系统调用的完整执行过程**：

```
① 用户程序调用库函数（如 printf("hello")）
② 库函数内部将系统调用号（如 write=4）和参数放入约定寄存器
   Linux x86: eax=系统调用号, ebx/ecx/edx=参数
③ 执行陷入指令（int 0x80 / sysenter / syscall），触发陷阱异常
④ CPU 硬件自动切换到内核态，跳到中断向量表中对应的入口
⑤ 内核的 system_call 入口代码保存用户态上下文
⑥ 通过系统调用号查系统调用表（sys_call_table[]），获得处理函数地址
⑦ 执行对应的系统调用服务程序（如 sys_write）
⑧ 将返回值放入约定寄存器（如 eax）
⑨ 恢复用户态上下文
⑩ 执行 IRET/sysret 指令，CPU 切换回用户态，返回用户程序继续执行
```

**系统调用与普通库函数的本质区别**：

| 比较项 | 系统调用 | 普通库函数 |
|:--------|:----------|:-------------|
| **运行状态** | **内核态**执行 | **用户态**执行 |
| **入口机制** | 陷入指令（trap/INT），触发中断 | 普通 call 指令，函数跳转 |
| **特权级别** | 最高特权（Ring 0） | 最低特权（Ring 3） |
| **可访问资源** | 全部系统资源 | 仅自身地址空间 |
| **开销** | 大（用户态 ↔ 内核态切换） | 小（进程内跳转） |
| **返回值** | 可能引起进程切换 | 必定返回调用者 |
| **层次关系** | 库函数可能内部调用系统调用 | 纯粹的库函数不涉及系统调用 |

**库函数与系统调用的层次关系**：

```c
// 用户程序 → 库函数 → 系统调用
#include <stdio.h>
int main() {
    printf("hello");        // 用户程序，用户态
    // ↓
    // printf() 库函数内部调用 write() 系统调用
    // ↓
    // write() 系统调用：int 0x80 → 内核态执行 → 操作显存
    return 0;
}
```

**系统调用与普通函数调用的流程对比**：

| 步骤 | 系统调用（如 write） | 普通函数（如 strlen） |
|:------|:---------------------|:----------------------|
| ① 调用方式 | 陷入指令（int 0x80 / syscall） | call 指令 |
| ② 特权级切换 | **是**（Ring 3 → Ring 0） | **否** |
| ③ 栈切换 | **是**（用户栈 → 内核栈） | **否** |
| ④ 进程上下文 | 可能切换 | 不切换 |
| ⑤ 返回方式 | IRET / sysret 指令 | ret 指令 |

> 1. 系统调用在用户态**发起**（执行 trap 指令），在**内核态**执行
> 2. 区分"发生"和"执行"——"系统调用发生"指在用户态执行陷入指令
> 3. **进程切换**只能在**内核态**完成（需要修改 PCB 等特权操作）
> 4. 用户程序想获得 OS 服务，**唯一途径**是系统调用

### 1.4 操作系统的体系结构

#### 1.4.1 大内核与微内核

> **操作系统内核**：OS 中运行在**内核态**、常驻内存的那部分核心代码，负责管理进程、内存、设备、文件等底层功能。内核与用户程序运行在不同的特权级上，以保障系统安全。

**大内核（宏内核 / 单内核 / Monolithic Kernel）**：

将操作系统的主要功能模块（进程管理、内存管理、文件系统、设备驱动、网络协议栈等）全部集成在内核中，运行在内核态。

```
┌───────────────────────────────────────────────┐
│              用户应用程序（用户态）               │
├───────────────────────────────────────────────┤
│  ┌─────────────────────────────────────────┐  │
│  │           大内核（内核态）                   │  │
│  │  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐   │  │
│  │  │进程管理│ │内存管理│ │文件系统│ │设备驱动│   │  │
│  │  └──────┘ └──────┘ └──────┘ └──────┘   │  │
│  │  ┌──────┐ ┌──────┐                       │  │
│  │  │网络协议│ │系统调用│                       │  │
│  │  └──────┘ └──────┘                       │  │
│  └─────────────────────────────────────────┘  │
├───────────────────────────────────────────────┤
│                 硬件                            │
└───────────────────────────────────────────────┘
```

**优点**：
- **性能高**：各模块在同一地址空间，函数调用即可通信，无需进程间切换
- **代码成熟**：Linux、Unix 经过数十年验证

**缺点**：
- **可靠性差**：任一内核模块的崩溃（如驱动程序 bug）会导致整个系统崩溃（内核恐慌 / Blue Screen）
- **可维护性差**：代码高度耦合，修改一个模块可能影响其他模块
- **体积庞大**：内核映像巨大，资源占用高

**代表系统**：**Linux**、**Unix**（BSD、Solaris）、**Windows**（Windows NT 本质上是混合内核，但偏向宏内核）

**微内核（Microkernel）**：

将内核功能减到**最少**——仅保留最核心的功能（进程间通信 IPC、基本调度、内存管理、中断处理），其余功能（文件系统、设备驱动、网络协议栈等）都移出内核态，作为**用户态的独立服务进程**运行。

```
┌───────────────────────────────────────────────────────┐
│                 用户态                                  │
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐          │
│  │文件服务│ │设备驱动│ │网络服务│ │用户程序│  ...       │
│  └────────┘ └────────┘ └────────┘ └────────┘          │
│         ↕ IPC      ↕ IPC      ↕ IPC                    │
├───────────────────────────────────────────────────────┤
│  ┌───────────────────────────────────────────────┐    │
│  │          微内核（内核态）                         │    │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐       │    │
│  │  │ IPC 通信  │ │基本调度   │ │内存管理   │       │    │
│  │  └──────────┘ └──────────┘ └──────────┘       │    │
│  │  ┌──────────┐                                  │    │
│  │  │中断处理   │                                  │    │
│  │  └──────────┘                                  │    │
│  └───────────────────────────────────────────────┘    │
├───────────────────────────────────────────────────────┤
│                     硬件                                │
└───────────────────────────────────────────────────────┘
```

**优点**：
- **可靠性高**：某服务进程崩溃不影响内核和其他服务
- **可扩展性好**：添加新服务只需增加新的用户态进程
- **易于移植**：内核代码量小，移植到新平台时只需修改少量代码
- **易于维护和调试**：各服务进程独立，便于测试和修改

**缺点**：
- **性能较低**：服务进程间频繁的 IPC 通信开销大，多次模式切换（用户 ↔ 内核）
- **实现复杂**：IPC 机制的设计和优化难度大

**代表系统**：**Minix**（教学用）、**QNX**（嵌入式实时系统）、**Mach**（macOS 底层）、**L4**（微内核家族）

**大内核 vs 微内核 综合对比**：

| 对比维度 | 大内核（宏内核） | 微内核 |
|:----------|:------------------|:--------|
| **内核态功能** | 多（文件系统、网络、驱动全在内核） | 少（仅 IPC、基本调度、内存管理） |
| **用户态功能** | 少 | 多（文件系统、网络、驱动作为服务进程） |
| **通信方式** | 函数调用（快速） | 进程间通信 IPC（较慢） |
| **性能** | **高**（切换开销小） | **低**（频繁 IPC 和模式切换） |
| **可靠性** | 差（一模块崩溃全系统崩溃） | **好**（服务独立，互不影响） |
| **可扩展性** | 差 | **好** |
| **可移植性** | 差 | **好** |
| **代码规模** | 内核代码量巨大 | 内核代码精简 |
| **代表系统** | Linux、Unix、Windows | Minix、QNX、Mach、L4 |

> - Linux 是**典型的大内核（宏内核）**
> - Minix、QNX 是**典型的微内核**
> - 大内核**性能高但可靠性低**，微内核**可靠性高但性能低**
> - Windows NT 属于**混合内核**（偏向宏内核，吸收了微内核的部分设计思想）

#### 1.4.2 分层结构与模块化结构

**分层结构（Layered Structure）**：

> 将操作系统划分为若干层次，每层只能使用**直接下层**提供的功能和服务，不能跨层调用。

```
┌─────────────────────┐
│   用户接口层（第 N 层）   │  ← 最上层，面向用户
├─────────────────────┤
│    ...              │
├─────────────────────┤
│   文件系统层          │
├─────────────────────┤
│   内存管理层          │
├─────────────────────┤
│   进程调度层          │
├─────────────────────┤
│   硬件驱动层（第 0 层）  │  ← 最下层，面向硬件
└─────────────────────┘
```

**优点**：结构清晰、易于调试和验证（可逐层测试）、每层功能单向依赖

**缺点**：层次划分困难、层层调用导致额外开销、修改某层可能引发连锁反应

**代表系统**：**THE 操作系统**（Dijkstra 设计，首个采用分层结构的 OS）

**模块化结构（Modular Structure）**：

> 将操作系统内核划分为一系列可**独立编译、动态加载/卸载**的功能模块。各模块通过规定的接口相互调用，内核核心保持最小。

**Linux 内核的模块化管理**：

```bash
# 列出已加载的内核模块
lsmod

# 动态加载模块
insmod mydriver.ko    # 加载模块
modprobe mydriver     # 自动解决依赖并加载

# 卸载模块
rmmod mydriver        # 卸载模块
```

**Linux 内核模块类型**：

| 模块类型 | 功能 | 例子 |
|:----------|:------|:------|
| **设备驱动** | 驱动硬件设备 | `nvidia.ko`、`e1000.ko` |
| **文件系统** | 支持特定文件系统 | `ext4.ko`、`ntfs.ko` |
| **网络协议** | 实现网络协议功能 | `tcp_bbr.ko`、`nf_conntrack.ko` |

**优点**：
- 内核体积可控（按需加载）
- 易于扩展（第三方可开发内核模块）
- 支持热插拔（设备驱动模块随设备插拔加载/卸载）
- 维护方便（模块间接口清晰）

**缺点**：
- 模块间的接口设计需要精心规划
- 模块 crash 仍可能导致内核崩溃（与宏内核共性问题）

> Linux 整体属于**大内核体系**，但内部采用**模块化**的组织方式，兼具两种架构的优势。

#### 1.4.3 各种体系结构总结对比

| 体系结构 | 核心思想 | 优点 | 缺点 | 代表 |
|:----------|:----------|:------|:------|:------|
| **整体式（大内核）** | 所有功能集成在内核，直接函数调用 | 性能最高 | 可靠性差、可维护性差 | Linux、Unix |
| **层次式** | 分层设计，仅能调用下层服务 | 结构清晰，易于验证 | 效率低，层次划分困难 | THE |
| **微内核** | 内核最小化，其他功能外移用户态 | 可靠性高，可扩展 | IPC 开销大，性能较低 | Minix、QNX、Mach |
| **模块化** | 内核核心 + 可动态加载的模块 | 灵活性高，按需加载 | 模块 crash 影响整体 | Linux、Solaris |
| **虚拟机结构** | 在物理机上模拟多个逻辑计算机 | 隔离性强 | 资源开销大 | VMware、KVM |

> 1. Linux 同时属于**大内核**（整体架构）+ **模块化**（组织方式）
> 2. Windows NT 是**混合内核**，在宏内核基础上吸收了微内核的部分思想
> 3. 微内核的**性能瓶颈**在于 IPC 通信开销

#### 2.1.1 进程的概念与特征

**一、进程的定义**

> **进程（Process）** 是进程实体的运行过程，是系统进行**资源分配**和**调度**的一个独立单位。

要理解进程，首先需要区分**程序**与**进程**这两个核心概念：

| 对比项 | 程序 | 进程 |
|--------|------|------|
| 本质 | **静态**的，存放在磁盘里的可执行文件 | **动态**的，程序的一次执行过程 |
| 生命周期 | 永久存在（只要文件不删除） | 有创建、运行、消亡的生命周期 |
| 组成 | 代码 + 数据 | **PCB + 程序段 + 数据段** |
| 并发性 | 不可并发 | 可并发执行 |
| 对应关系 | 一个程序可对应多个进程 | 一个进程可执行多个程序 |
| 示例 | `QQ.exe` 文件 | 可同时启动多次 QQ，每次对应一个进程 |

- **进程实体（进程映像）**：由 PCB、程序段、数据段三部分组成，是**静态**的快照
- **进程**：是进程实体的**运行过程**，是**动态**的
- 进程实体反映了进程在某一时刻的状态（如 `x++` 执行后 `x=2`）

> **"进程实体"是静态的，"进程"是动态的。** 考题中常将二者混用，但需要准确理解"进程映像"指的是内存中的静态布局，而"进程"指的是运行过程本身。

**二、进程的组成**

进程实体（进程映像）由三部分组成：

```
进程实体（进程映像）
├── PCB ─────── 给操作系统用的（管理者所需的信息）
├── 程序段 ──── 给进程自己用的（包含指令）
└── 数据段 ──── 给进程自己用的（包含数据）
```

#### 1. PCB（Process Control Block，进程控制块）

> **PCB 是进程存在的唯一标志！** 当进程被创建时，操作系统为其创建 PCB；当进程结束时，回收其 PCB。

PCB 中包含的信息：

| 信息类别 | 具体内容 | 作用 |
|----------|----------|------|
| 进程描述信息 | PID（进程ID）、UID（用户ID） | 让操作系统**区分各个进程** |
| 进程控制和管理信息 | 进程当前状态、优先级 | 实现操作系统对进程的**管理** |
| 资源分配清单 | 内存使用情况、I/O 设备、打开的文件 | 实现操作系统对**资源的管理** |
| 处理机相关信息 | 程序计数器(PC)、通用寄存器、PSW 等 | 实现进程**上下文切换** |

#### 2. 程序段

- 包含程序的**指令**（机器指令序列）
- 是给**进程自己**用的
- 同一个程序的多个进程可共享相同的程序段内容

#### 3. 数据段

- 包含运行过程中产生的各种**数据**（全局变量、局部变量、常量等）
- 是给**进程自己**用的
- 不同进程的数据段各不相同

#### 4. 运行时的内存布局

```
┌─────────────────────────────────────┐
│              内存                    │
│  ┌───────────────────────────────┐  │
│  │             PCB               │  │
│  │  PC:   下一条指令地址          │  │
│  │  IR:   当前正在执行的指令       │  │
│  │  PSW:  程序状态字寄存器         │  │
│  │  通用寄存器: 运行中间结果       │  │
│  ├───────────────────────────────┤  │
│  │          程序段                │  │
│  │  指令1, 指令2, 指令3, ...      │  │
│  ├───────────────────────────────┤  │
│  │          数据段                │  │
│  │  x=1, ...                    │  │
│  └───────────────────────────────┘  │
└─────────────────────────────────────┘
```

**关键寄存器：**

| 寄存器 | 全称 | 作用 |
|--------|------|------|
| **PC** | 程序计数器 | 存放**下一条**指令的地址 |
| **IR** | 指令寄存器 | 存放当前**正在执行**的指令 |
| **PSW** | 程序状态字寄存器 | 存放程序状态信息（如条件码、中断屏蔽等） |
| 通用寄存器 | — | 存放运行过程中的中间结果 |

> 一条高级语言代码翻译过来可能对应**多条机器指令**。例如 `x++` 可能是：`LOAD R1, [x]` → `ADD R1, 1` → `STORE [x], R1`。

**三、进程的五大特征**

相比于程序（静态），进程拥有以下**五大特征**：

| 特征 | 说明 | 重要程度 |
|------|------|----------|
| **动态性** | 进程最基本的特征。进程是程序的一次执行过程，动态地产生、变化和消亡 | ⭐⭐⭐ **最基本** |
| **并发性** | 多个进程实体同存于内存中，能在一段时间内同时运行 | ⭐⭐ |
| **独立性** | 进程是系统进行资源分配和调度的**独立单位** | ⭐⭐ |
| **异步性** | 各进程以各自独立的、不可预知的速度向前推进，导致运行结果**不确定性** | ⭐⭐⭐ |
| **结构性** | 每个进程配置一个 PCB，从结构上看，进程实体由程序段、数据段和 PCB 组成 | ⭐ |

> **异步性**是导致并发程序执行结果不确定性的根本原因。这是 PV 操作、管程等同步机制存在的必要性基础。

---

#### 2.1.2 进程的状态与转换

**一、进程的五种基本状态**

| 状态 | 英文 | 说明 |
|------|------|------|
| **创建态** | New | 进程正在被创建，OS 为其分配资源、初始化 PCB |
| **就绪态** | Ready | 已具备运行条件（获得了除 CPU 之外的所有资源），等待 CPU |
| **运行态** | Running | 正在 CPU 上执行。单 CPU 下同时最多一个；多核 CPU 下可多个 |
| **阻塞态** | Blocked/Waiting | 等待某个事件发生（如等待 I/O、信号量），事件发生前无法继续 |
| **终止态** | Terminated | 进程执行完毕被撤销，OS 回收资源和 PCB |

> - **就绪态**：等**CPU**（万事俱备，只欠 CPU）
> - **阻塞态**：等**事件**（如 I/O 完成、资源可用）
> - CPU 空闲时，阻塞进程即使被唤醒，也先进入**就绪态**再被调度运行

**二、状态转换（五状态模型）**

```
            创建态
              │
              ▼
    ┌─────────────────┐     被调度选中       ┌──────────────┐
    │     就绪态        │ ──────────────────► │    运行态      │
    │   (Ready)        │                     │  (Running)    │
    └─────────────────┘ ◄─────────────────── └──────────────┘
              ▲         时间片到 / 主动让出CPU       │
              │                                     │
      等待的事件发生                      等待事件/申请资源
              │                            (主动行为)
              │                                     │
    ┌─────────────────┐                     ▼
    │     阻塞态        │               终止态(Terminated)
    │   (Blocked)      │
    └─────────────────┘
```

**关键转换条件：**

| 转换 | 触发条件 | 行为性质 |
|------|----------|----------|
| 创建态 → 就绪态 | 系统完成创建进程的一系列工作（分配资源、初始化 PCB） | 系统行为 |
| 就绪态 → 运行态 | 进程被调度程序选中，分配 CPU | 系统行为 |
| 运行态 → 就绪态 | 时间片到，或被更高优先级进程抢占 | 被动 |
| 运行态 → 阻塞态 | 进程主动请求等待某事件（I/O、信号量等） | **主动行为** |
| 阻塞态 → 就绪态 | 等待的事件发生（由其他进程或 OS 唤醒） | **被动行为** |
| 运行态 → 终止态 | 进程执行完毕或异常终止（`exit` 系统调用） | 主动行为 |

> **以下两种转换是不存在的：**
> - ❌ **阻塞态 → 运行态**：必须先回到就绪态，再由调度程序选中上 CPU
> - ❌ **就绪态 → 阻塞态**：就绪态进程没有运行，不可能主动等待事件
>
> 口诀：**就绪和阻塞不直接到运行，阻塞必须先经过就绪。**

> **题目**：下列关于进程的叙述中，正确的是（ ）
> A. 进程获得 CPU 运行是通过调度得到的
> B. 优先级是进程调度的重要依据，一旦确定不能改变
> C. 在单 CPU 系统中，任何时刻最多有一个进程处于运行态
> D. 进程申请 CPU 得不到满足时，其状态变为阻塞态
>
> **答案**：A、C
> - A ✓：进程从就绪 → 运行通过调度
> - B ✗：优先级可以动态改变（如动态优先级调度）
> - C ✓：单 CPU 同一时刻只有一个进程运行
> - D ✗：申请 CPU 得不到满足 → 处于**就绪态**（不是阻塞态，阻塞态是等待 I/O 等事件）

**三、进程的组织方式**

#### 1. 链接方式

按照进程状态将 PCB 组成多个队列（链表）：

```
执行指针 ──────► PCB2（当前运行态的进程，单CPU只有一个）

就绪队列指针 ──► PCB5 → PCB1 → PCB4（优先级高的放队头）

阻塞队列指针 ──► PCB3 → PCB7 → PCB6（可按阻塞原因细分多个阻塞队列）
```

- **执行指针**：指向当前处于运行态的进程
- **就绪队列指针**：指向就绪态 PCB 链表
- **阻塞队列指针**：指向阻塞态 PCB 链表

#### 2. 索引方式

根据进程状态建立多张索引表：

```
执行指针 ──────► PCB2

就绪表指针 ────► 就绪索引表 → PCB5, PCB1, PCB4
阻塞表指针 ────► 阻塞索引表 → PCB6, PCB7, PCB3
```

**四、七状态模型（含挂起态）**

在五状态模型的基础上，通过**中级调度**引入挂起操作，形成七状态模型：

| 状态 | 说明 |
|------|------|
| 创建态 | 正在创建，分配资源、初始化 PCB |
| **就绪态** | 在**内存**中，已经具备运行条件，等待 CPU |
| **就绪挂起** | 被调到**外存**，但已具备运行条件，只需重新调入内存 |
| 运行态 | 正在 CPU 上执行 |
| **阻塞态** | 在**内存**中，等待某个事件发生 |
| **阻塞挂起** | 被调到**外存**，同时等待某个事件 |
| 终止态 | 进程执行完毕，回收资源 |

```
        创建态
           │
           ▼
    ┌──→ 就绪态 ◄──────┐
    │      │            │
    │      ▼            │ 激活
    │    运行态          │
    │      │            │
    │      ▼            │
    │    阻塞态 ────────┘
    │      │
    │      ▼ 挂起
    │    阻塞挂起
    │      │
    │      ▼ 事件出现
    │    就绪挂起
    │      │
    │      ▼ 激活
    └──── 就绪态
           │
           ▼
        终止态
```

> | 对比 | 阻塞态 | 挂起态 |
> |------|--------|--------|
> | 进程映像位置 | 在**内存**中 | 被调到**外存**（磁盘） |
> | 激活条件 | 等待的事件发生 | 重新调入内存 |
> | 引发原因 | 进程主动等待资源/事件 | 内存不足，被中级调度换出 |

#### 2.1.3 进程控制

**一、什么是进程控制？**

进程控制的主要功能是对系统中的所有进程实施有效的管理——**创建**新进程、**撤销**已有进程、实现进程**状态转换**。

> 简化理解：进程控制的本质就是实现**进程状态转换**。

**二、用"原语"实现进程控制**

#### 1. 原语的概念

> **原语（Primitive）**是一种特殊的程序，它的执行具有**原子性**。也就是说，这段程序的运行必须**一气呵成，不可中断**。

原语位于操作系统的内核层：

```
用户 ─────── 应用程序（软件）
              │
操作系统 ──── 进程管理、存储器管理、设备管理等功能
              │
内核 ──────── 原语（设备驱动、CPU 切换等）
              │
裸机 ──────── 纯硬件
```

#### 2. 为什么需要原子性？

如果不能"一气呵成"，就可能导致关键数据结构信息不一致：

> 例：假设进程 2 等待的事件发生，OS 至少需要做两件事：
> ① 将 PCB2 的 `state` 设为"就绪态"
> ② 将 PCB2 从阻塞队列移到就绪队列
>
> 如果在完成 ① 后收到中断信号，导致 ② 未执行，则 PCB2 的 `state=就绪态`，但它却仍在阻塞队列里 → **数据不一致**。

#### 3. 如何实现原子性？

使用**关中断指令**和**开中断指令**这两个**特权指令**实现：

```
……指令1 指令2 │ 关中断指令 │ 指令a 指令b │ 开中断指令 │ 指令3 ……
               │←── 原语执行期间，不检查中断信号 ──→│
```

- CPU 执行关中断指令后，**不再例行检查中断信号**
- 直到执行开中断指令后才恢复检查
- 关中断与开中断之间的指令序列**不可被中断** → 实现"原子性"

> 这两个特权指令**不允许用户程序使用**，只能在**内核态**执行。否则用户可以一直关中断，系统将无法正常工作。

**三、进程控制相关的原语**

#### 1. 创建原语（创建态 → 就绪态）

```
创建原语的操作：
1. 申请空白 PCB
2. 为新进程分配所需资源
3. 初始化 PCB（填写 PID、状态等）
4. 将 PCB 插入就绪队列
```

引起进程创建的事件：用户登录、作业调度、提供服务、应用请求。

#### 2. 撤销原语（就绪态/阻塞态/运行态 → 终止态 → 无）

```
撤销原语的操作：
1. 从 PCB 集合中找到要终止的进程的 PCB
2. 若进程正在运行，立即剥夺 CPU，将 CPU 分配给其他进程
3. 终止其所有子进程（进程间的关系是树形结构）
4. 将该进程拥有的所有资源归还给父进程或操作系统
5. 删除 PCB
```

引起进程终止的事件：
- 进程自己请求终止（`exit` 系统调用）
- 异常结束（整数除以 0、非法使用特权指令，被 OS 强行杀掉）
- 外界干预（Ctrl+Alt+Delete，用户选择杀掉进程）

#### 3. 阻塞原语（运行态 → 阻塞态）

```
阻塞原语的操作：
1. 找到要阻塞的进程的 PCB
2. 保护进程运行现场（保存寄存器等上下文信息），将 PCB state 设为"阻塞态"
3. 将 PCB 插入相应事件的阻塞队列
```

#### 4. 唤醒原语（阻塞态 → 就绪态）

```
唤醒原语的操作：
1. 在事件的阻塞队列中找到该进程的 PCB
2. 将 PCB 从阻塞队列移出，state 设为"就绪态"
3. 将 PCB 插入就绪队列
```

> **阻塞原语和唤醒原语必须成对使用！** 因何事阻塞，就应由何事唤醒。一个进程不能无缘无故被唤醒，也不能由不相关的事件唤醒。

#### 5. 切换原语（运行态 ↔ 就绪态）

```
切换原语的操作：
1. 将运行进程的运行环境（寄存器数据等）存入其 PCB
2. 将该进程的 PCB 移入相应队列（就绪队列或阻塞队列）
3. 选择另一个进程执行，并更新其 PCB
4. 根据新进程的 PCB 恢复其运行环境（寄存器、PC 等）
```

**四、进程上下文（Context）**

进程切换时需要保存和恢复的信息称为**进程上下文**：

| 信息 | 说明 |
|------|------|
| PC（程序计数器） | 下一条指令的地址 |
| PSW（程序状态字） | 程序状态信息 |
| 通用寄存器 | 运行过程中的中间结果 |
| 其他必要信息 | 如栈指针、内存管理信息等 |

**上下文切换过程：**

```
进程A运行中
    │
    ▼
保存A的上下文到A的PCB
    │
    ▼
从B的PCB恢复B的上下文
    │
    ▼
进程B开始运行
```

> 上下文切换是一个**纯开销**操作——切换期间 CPU 不执行任何用户程序。因此，频繁的进程切换会降低系统效率。这也是线程引入的动机之一——同一进程内的线程切换开销远小于进程切换。

---

#### 2.1.4 进程通信

**一、什么是进程间通信？**

> **进程间通信（IPC, Inter-Process Communication）**是指两个进程之间进行**数据交换**的过程。

**为什么需要 OS 支持？** 进程是分配系统资源的单位（包括内存地址空间），各进程拥有的内存地址空间**相互独立**。为保证安全，一个进程**不能直接访问**另一个进程的地址空间。因此，进程通信需要操作系统提供支持。

---

**二、共享存储**

操作系统在内存中划出一块**共享存储区**，两个进程通过这片共享区域交换数据。

```
进程P地址空间     共享存储区      进程Q地址空间
    ┌─────┐    ┌─────────┐    ┌─────┐
    │     │    │         │    │     │
    │  P  │◄──►│  共享区  │◄──►│  Q  │
    │     │    │         │    │     │
    └─────┘    └─────────┘    └─────┘
```

**两种共享方式：**

| 方式 | 说明 | 特点 |
|------|------|------|
| **基于数据结构的共享** | 共享空间里只能放特定数据结构（如长度为 10 的数组） | 速度慢、限制多，**低级通信** |
| **基于存储区的共享** | 数据的形式、存放位置由通信进程自己控制 | 速度快，**高级通信** |

**注意事项：**
- 各进程对共享空间的访问必须是**互斥**的
- 可使用 **P/V 操作**实现同步互斥
- 通过"增加页表项/段表项"可将同一片共享内存映射到各进程的地址空间

---

**三、消息传递**

进程间的数据交换以**格式化的消息（Message）**为单位。通过 OS 提供的**发送消息/接收消息**原语进行数据交换。

**消息格式：**

```
┌────────────────┬──────────────┐
│    消息头       │    消息体     │
│  - 发送进程ID   │              │
│  - 接收进程ID   │   实际数据    │
│  - 消息长度     │              │
└────────────────┴──────────────┘
```

#### 1. 直接通信方式

发送方**指名道姓**地发送消息给接收方：

```c
send(Q, msg);      // 进程P发送消息给进程Q
receive(P, &msg);  // 进程Q从进程P接收消息
```

特点：发送和接收双方必须显式指定对方进程名。

#### 2. 间接通信方式（信箱通信）

以**"信箱"**作为中间实体进行消息传递：

```c
send(mailbox_A, msg);        // 发送到信箱A
receive(mailbox_A, &msg);    // 从信箱A接收
```

- 可以多个进程往同一个信箱 `send` 消息
- 也可以多个进程从同一个信箱中 `receive` 消息

---

**四、管道通信**

"管道"是一个特殊的**共享文件**（又名 `pipe` 文件），本质是在内存中开辟的一个大小固定的**内存缓冲区**。

```
进程P ──写数据──► [管道（内存缓冲区）] ──读数据──► 进程Q
```

**管道的 5 条关键规则：**

| 编号 | 规则 | 考点提示 |
|------|------|----------|
| ① | 管道只能采用**半双工**通信，某一时间段内只能**单向**传输。双向通信需设置**两个管道** | 基础特性 |
| ② | 各进程要**互斥**地访问管道（由操作系统实现） | 互斥 |
| ③ | 管道**写满**时，写进程**阻塞**，直到读进程取走数据后唤醒写进程 | 同步 |
| ④ | 管道**读空**时，读进程**阻塞**，直到写进程写入数据后唤醒读进程 | 同步 |
| ⑤ | 数据一旦被读出就**彻底消失**。一个管道允许多个写进程，但只允许**一个读进程** | ⭐ 408 真题 |

**补充说明：**
- 写进程往管道写数据，即使管道没被写满，只要管道**没空**，读进程就可以从管道读数据
- 读进程从管道读数据，即使管道没被读空，只要管道**没满**，写进程就可以往管道写数据

> 一个管道允许**多个写进程**同时写入，但**一个读进程**读取。这是官方真题答案，与某些教材描述不同，务必以此为准。

---

**五、三种通信方式对比**

| 对比项 | 共享存储 | 消息传递 | 管道通信 |
|--------|----------|----------|----------|
| 通信速度 | **最快** | 较慢 | 中等 |
| 同步方式 | 需自己实现同步互斥 | 自带同步 | 自带同步（写满/读空阻塞） |
| 实现机制 | 内存映射 | send/receive 原语 | 共享文件（内存缓冲区） |
| 适用场景 | 大量数据、高速通信 | 分布式、跨网络 | 父子进程、简单流式通信 |
| 方向性 | 双向 | 双向/多向 | **半双工** |

#### 2.1.5 线程的概念与特点

**一、为什么要引入线程？**

**引入前的问题：**
- 没有进程之前，系统中各程序只能**串行执行**
- 引入进程后，不同程序可以**并发**执行
- 但一个进程内部仍然是**串行**的

**实际需求举例：**
一个 QQ 进程需要同时处理视频聊天、文字聊天、传送文件——这些功能不可能由一个程序顺序处理实现。

**引入线程后的效果：**

```
引入前：                      引入后：
进程1                        进程
├── 代码1                    ├── 线程1（视频聊天）
├── 代码2                    ├── 线程2（文字聊天）
├── 代码3                    └── 线程3（传文件）
└── ……
                             进程
进程2                        ├── 线程4
├── 代码a                    └── 线程5
├── 代码b
├── 代码c
└── ……
```

> 引入**线程**来增加并发度，让一个进程内也可以并发处理各种任务。

**二、什么是线程？**

> **线程（Thread）**是一个基本的**CPU 执行单元**，也是**程序执行流的最小单位**。

- 可以把线程理解为**"轻量级进程"**
- 引入线程后，不仅是**进程之间**可以并发，进程内的**各线程之间**也可以并发

**引入线程后的核心变化：**

| 对比项 | 引入线程前 | 引入线程后 |
|--------|-----------|-----------|
| 资源分配 | 进程是资源分配和调度的基本单位 | **进程**只作为除 CPU 之外的**系统资源的分配单元** |
| 调度单位 | 进程是程序执行流的最小单位 | **线程**成为**处理机的调度单位**（程序执行流的最小单位） |
| 并发性 | 只有进程间可以并发 | 进程间、线程间都可以并发 |
| 系统开销 | 进程切换开销大 | **同一进程内**的线程切换开销小 |

> **去图书馆看书：**
> - 桌子 = 处理机
> - 人 = 进程
> - 看不同的书 = 线程
>
> **切换进程运行环境**：有一个不认识的人要用桌子 → 你需要把你的书收走，他把自己的书放到桌上（**开销大**）
>
> **同一进程内的线程切换**：你的舍友要用这张书桌 → 可以不把桌子上的书收走（**开销小**）

**三、线程的属性**

| 属性 | 说明 |
|------|------|
| 独立调度 | 线程是处理机调度的基本单位 |
| 共享资源 | 同一进程内的各线程共享进程的**大部分**资源（地址空间、打开的文件、信号处理等） |
| 独立上下文 | 每个线程有自己的线程 ID、程序计数器(PC)、寄存器组、栈 |
| 并发性 | 同一进程内的线程也可以并发执行（在多核 CPU 上可真正并行） |
| 状态 | 线程也有就绪、运行、阻塞等基本状态 |

**四、线程的实现方式**

#### 1. 用户级线程（User-Level Thread, ULT）

- 由**应用程序**通过线程库（如 POSIX Pthreads）实现
- 线程管理的**所有工作**都由应用程序在**用户态**完成
- **内核意识不到**线程的存在（内核只看到一个进程）
- 线程切换**不需要**切换到内核态，速度快

#### 2. 内核级线程（Kernel-Level Thread, KLT）

- 线程管理的**所有工作**由**内核**完成
- 内核中为每个内核级线程保留**线程控制块（TCB）**
- 线程切换需要**内核态**支持
- 操作系统调度以**内核级线程**为单位

#### 3. ULT vs KLT 对比

| 比较项 | 用户级线程（ULT） | 内核级线程（KLT） |
|--------|------------------|-------------------|
| 实现者 | 应用程序（线程库） | 操作系统内核 |
| 调度位置 | 用户态，无需系统调用 | 内核态 |
| 切换速度 | **快**（不需要模式切换） | **慢**（需要模式切换） |
| 阻塞影响 | 一个线程阻塞 → **整个进程阻塞** | 仅阻塞该线程 |
| 多处理器 | **不能**真正并行（内核只调度进程） | **可以**真正并行（多核同时运行） |
| 内核感知 | 内核意识不到线程存在 | 内核为每个线程维护 TCB |

**五、多线程模型**

| 模型 | 映射关系 | 说明 | 优缺点 |
|------|----------|------|--------|
| **多对一** | 多个 ULT ↦ 一个 KLT | 多个用户级线程映射到一个内核级线程 | 开销小，但**一个线程阻塞整个进程阻塞**，不能多核并行 |
| **一对一** | 一个 ULT ↦ 一个 KLT | 每个用户级线程对应一个内核级线程 | **并发性好**，可多核并行，但开销大（每创建一个用户线程就要创建一个内核线程） |
| **多对多** | $n$ 个 ULT ↦ $m$ 个 KLT（$n \ge m$） | $n$ 个用户级线程映射到 $m$ 个内核级线程 | **折中方案**，兼具并发性和效率 |

> 现代操作系统（Linux、Windows、macOS）普遍采用**一对一**线程模型。

---

#### 2.1.6 进程与线程的比较

从多个维度对比进程与线程：

| 比较维度 | 进程 | 线程 |
|----------|------|------|
| **基本单位** | 资源分配的基本单位 | CPU 调度的基本单位（程序执行流的最小单位） |
| **调度** | 进程切换开销大，需切换地址空间 | 同一进程内线程切换开销小，不切换地址空间 |
| **并发性** | 进程间可以并发 | 进程间、线程间都可以并发 |
| **拥有资源** | 拥有独立的地址空间、文件、内存等资源 | 共享所属进程的资源，仅拥有少量私有资源（栈、寄存器、PC） |
| **系统开销** | 创建/撤销/切换开销**大** | 创建/撤销/切换开销**小** |
| **地址空间** | 各进程地址空间相互独立 | 同一进程的各线程共享地址空间 |
| **通信方式** | 需通过 IPC（共享存储、消息传递、管道） | 可直接读写进程数据段来通信（比 IPC 简单） |
| **独立性** | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |
| **多核支持** | 不同进程可在不同核心上并行 | 内核级线程可在不同核心上并行；用户级线程不行 |

> **"进程管资源，线程管执行"**
> - 进程 = 资源分配单位（有独立的地址空间）
> - 线程 = CPU 调度单位（共享同一进程的资源）
> - 同一进程内线程切换：快（地址空间不变）
> - 不同进程间切换：慢（地址空间切换）

#### 2.2.1 调度的概念

**一、调度的三个层次**

> 调度按照发生的层次分为三级：**高级调度（作业调度）**、**中级调度（内存调度）**、**低级调度（进程调度）**。

**1. 高级调度（作业调度 / 长程调度）**

- **方向**：外存 → 内存（面向**作业**）
- 按一定原则从外存上**作业后备队列**中挑选一个作业调入内存，并**创建进程**（分配 PCB、资源等）
- 每个作业只调入一次、调出一次；作业调入时建立 PCB，调出时才撤销
- 发生频率**最低**（分钟级）

**2. 低级调度（进程调度 / 短程调度）**

- **方向**：内存 → CPU（面向**进程 / 内核级线程**）
- 按照某种策略从**就绪队列**中选取一个进程，将处理机分配给它
- 是操作系统**最基本**的调度，频率**最高**（一般几十毫秒一次）

**3. 中级调度（内存调度 / 中程调度）**

- **方向**：外存 ↔ 内存（面向**进程**）
- 内存资源紧张时，将暂时不能运行的进程数据**挂起到外存**（挂起态）；内存空闲或进程需要运行时再**激活调回内存**
- 被挂起进程的 PCB 组织为**挂起队列**
- 同一进程可被多次调出、调入，发生频率**中等**

**三层调度对比**：

| 对比项 | 高级调度（作业调度） | 中级调度（内存调度） | 低级调度（进程调度） |
|:--:|:--|:--|:--|
| **做什么** | 从后备队列选作业调入内存 | 从挂起队列选进程调回内存 | 从就绪队列选进程分配 CPU |
| **调度发生在** | 外存 → 内存 | 外存 → 内存 | 内存 → CPU |
| **发生频率** | 最低 | 中等 | 最高 |
| **对进程状态的影响** | 创建态 → 就绪态 | 挂起态 → 就绪态（激活） | 就绪态 → 运行态 |

**二、七状态模型**

在五状态模型基础上引入**挂起**操作，形成七种状态：

```
         创建态
            │  (作业调度) 创建进程
            ▼
    ┌──→  就绪态  ◄────────┐
    │       │               │
    │       ▼ 调度          │ 激活
    │     运行态             │
    │       │               │
    │       ▼ I/O请求       │
    │     阻塞态 ──────────┘
    │       │
    │       ▼ 挂起到外存
    │    阻塞挂起
    │       │
    │       ▼ 等待的事件出现
    │    就绪挂起
    │       │
    │       ▼ 激活
    └────── 就绪态
            │
            ▼ 任务完成
          终止态
```

**两种挂起细分类别：**

| 状态 | 说明 |
|:--|:--|
| **就绪挂起** | 进程在外存，已具备运行条件，只需调入内存即可转为就绪态 |
| **阻塞挂起** | 进程在外存，仍在等待某事件发生 |

> **挂起 ≠ 阻塞**：挂起态进程映像在**外存**，阻塞态进程映像在**内存**。挂起是内存调度（中级调度）行为，阻塞是进程自身主动等待事件。

#### 2.2.2 调度的时机、切换与过程

**一、需要调度的情况**

当发生以下事件时，触发调度程序（scheduler）进行新一轮调度决策：

1. **进程主动放弃 CPU**（非抢占触发）：
   - 进程正常终止
   - 进程因 I/O 请求、等待信号量等事件主动阻塞
   - 进程调用 `wait()`、`sleep()` 等系统调用

2. **进程被动放弃 CPU**（抢占触发）：
   - 当前进程时间片用完
   - 更高优先级的进程到达就绪队列
   - 中断返回时发现有更高优先级进程就绪

**二、不能进行调度的情况**

以下情形即使发生了上述事件，也**不能**立即调度：

1. **处理中断的过程中**：中断处理程序通常屏蔽了同级或更低级中断，调度时机未到
2. **执行原子操作 / 原语的过程中**：如信号量 P/V 原语等，需一气呵成，不能被调度打断
3. **在内核临界区中**：访问内核共享数据结构时，需要保持互斥，不能切换进程（否则可能破坏内核数据一致性）

> 内核临界区**不等于**一般临界区。进程在内核临界区期间不能被调度，但退出临界区后可以；而用户态临界区通常不限制调度。

**三、进程切换的步骤**

```
进程A → 进程B 的切换过程：

Step 1: 保存进程A的上下文（PC、PSW、通用寄存器等 → PCB_A）
Step 2: 将进程A的PCB移入相应队列（就绪/阻塞/挂起）
Step 3: 选择下一个要运行的进程B（调度算法决策）
Step 4: 从进程B的PCB中恢复其上下文（PCB_B → 寄存器）
Step 5: 将CPU控制权交给进程B，开始执行
```

> **上下文切换开销**：保存和恢复上下文的纯 overhead，不包含进程实际执行的时间。切换过于频繁会降低系统效率——这是 RR 算法时间片不能设得太小的根本原因。

---

#### 2.2.3 调度方式

| 方式 | 定义 | 典型算法 |
|:--|:--|:--|
| **非抢占式（Nonpreemptive）** | 进程获取 CPU 后一直运行直到**主动放弃**（终止或阻塞），期间不能被强制剥夺 | FCFS、SJF、HRRN |
| **抢占式（Preemptive）** | 操作系统可根据某种规则（时间片到、更高优先级到达等）**强制暂停**当前运行进程，将 CPU 分配给其他进程 | SRTN、RR、优先级抢占式、MLFQ |

> 现代通用操作系统（Linux、Windows、macOS 等）通常使用**抢占式**调度，以保证交互响应性和多任务公平性。

---

#### 2.2.4 调度性能评价

**核心指标**

| 指标 | 定义 | 公式 |
|:--|:--|:--|
| **CPU 利用率** | CPU 处于"忙"状态的时间比例 | $\text{CPU利用率} = \frac{\text{CPU工作时间}}{\text{总时间}}$ |
| **系统吞吐量** | 单位时间内完成的作业/进程数量 | — |
| **周转时间** | 进程从**提交到完成**的全部时间 | $T_{\text{周转}} = \text{完成时间} - \text{到达时间}$ |
| **带权周转时间** | 周转时间与运行时间的比值（≥1，越接近 1 越好） | $W = \frac{T_{\text{周转}}}{T_{\text{运行}}} = \frac{\text{完成时间} - \text{到达时间}}{\text{运行时间}}$ |
| **等待时间** | 进程在就绪队列中等待 CPU 处理的时间总和 | $T_{\text{等待}} = T_{\text{周转}} - T_{\text{运行}} - T_{\text{I/O}}$（本题仅 CPU，简化为周转 − 运行） |
| **响应时间** | 从用户提交请求到**首次产生响应**的时间 | 适用于交互式系统，区别于周转时间（周转时间是到完成） |

> - **周转时间 = 等待时间 + 运行时间 + I/O 时间**（通用公式）
> - 对于纯 CPU 型进程（无 I/O 操作）场景，**等待时间 = 周转时间 − 运行时间**

#### 2.2.5 经典调度算法

**一、先来先服务（FCFS）**

> **First Come First Serve**，按进程到达就绪队列的先后顺序分配 CPU，公平如排队。

- **调度方式**：非抢占式
- **适用场景**：作业调度、进程调度均可

**例题**：

| 进程 | 到达时间 | 运行时间 |
|:--|:--|:--|
| P1 | 0 | 7 |
| P2 | 2 | 4 |
| P3 | 4 | 1 |
| P4 | 5 | 4 |

- 调度顺序：P1 → P2 → P3 → P4

```
|   P1   |   P2   |P3|   P4   |
0       7       11 12       16
```

| 进程 | 完成 | 周转 = 完成−到达 | 带权 = 周转÷运行 | 等待 = 周转−运行 |
|:--|:--|:--|:--|:--|
| P1 | 7 | 7−0 = 7 | 7÷7 = 1.00 | 7−7 = 0 |
| P2 | 11 | 11−2 = 9 | 9÷4 = 2.25 | 9−4 = 5 |
| P3 | 12 | 12−4 = 8 | 8÷1 = 8.00 | 8−1 = 7 |
| P4 | 16 | 16−5 = 11 | 11÷4 = 2.75 | 11−4 = 7 |

- 平均周转 = $(7+9+8+11)/4 = \mathbf{8.75}$
- 平均带权周转 = $(1+2.25+8+2.75)/4 = \mathbf{3.50}$
- 平均等待 = $(0+5+7+7)/4 = \mathbf{4.75}$

#### 优缺点

- ✅ 公平、实现简单，不会饥饿
- ❌ 对长作业有利，短作业可能排在长作业后面等待很久——**护航效应（Convoy Effect）**

> **Convoy 效应**：一个运行时间很长的进程先获得 CPU，后续所有短进程都只能排在它后面等待，如同车队中一辆慢车挡在车队最前面拖慢所有人。

---

**二、短作业优先（SJF / SPF）**

> **Shortest Job First**，优先调度运行时间最短的进程，可获得**最优的平均周转时间和平均等待时间**（在所有进程同时可达时）。

- **非抢占式 SJF（SPF）**：每次调度时从**已到达**进程中选运行时间最短的
- **抢占式 SJF（SRTN）**：每当新进程到达，比较所有未完成进程的**剩余时间**，选最短者运行

**例题**：

- 调度顺序：P1 → P3 → P2 → P4

```
|   P1   |P3|   P2   |   P4   |
0       7  8       12       16
```

| 进程 | 完成 | 周转 | 带权 | 等待 |
|:--|:--|:--|:--|:--|
| P1 | 7 | 7 | 1.00 | 0 |
| P3 | 8 | 4 | 4.00 | 3 |
| P2 | 12 | 10 | 2.50 | 6 |
| P4 | 16 | 11 | 2.75 | 7 |

- 平均周转 = $\mathbf{8.00}$（< FCFS 的 8.75）
- 平均带权周转 = $\mathbf{2.56}$（< FCFS 的 3.50）

> 非抢占 SJF 虽然整体优于 FCFS，但 P3 的带权周转达 4.00（仅执行 1 单位却等待了 3 单位），说明短作业也可能因到达晚而等待。

**例题**：

```
|P1|P2|P3|P2|   P4   |      P1      |
0  2  4  5  7       11              16
```

各时刻决策：
- 0：P1(剩余 7) 到达，运行 P1
- 2：P2(4) 到达，P1(5)、P2(4) → P2 剩余更短，抢占
- 4：P3(1) 到达，P2(2)、P1(5)、P3(1) → P3 最短，抢占
- 5：P3 完成，P4(6) 到达 → P2(2)、P1(5)、P4(6) → 运行 P2
- 7：P2 完成 → P1(5)、P4(4) → 运行 P4
- 11：P4 完成 → P1(5) → 运行 P1
- 16：全部完成

| 进程 | 完成 | 周转 | 带权 | 等待 |
|:--|:--|:--|:--|:--|
| P1 | 16 | 16 | 2.28 | 9 |
| P2 | 7 | 5 | 1.25 | 1 |
| P3 | 5 | 1 | 1.00 | 0 |
| P4 | 11 | 6 | 1.50 | 2 |

- 平均周转 = $\mathbf{7.00}$（< 非抢占 SJF 的 8）
- 平均带权周转 = $\mathbf{1.50}$
- 平均等待 = $\mathbf{3.00}$

#### 关键结论与易错点

> 1. 默认"短作业优先"指**非抢占式 SJF**，题目未说明时按非抢占处理
> 2. SJF 最优性条件：**在所有进程同时可达时**，SJF 平均等待/周转时间最少
> 3. **考虑不同到达时间**时，SRTN 才是理论上平均等待/周转时间最优的算法
> 4. SJF/SPF **会导致饥饿**：长作业在源源不断的短作业中永远得不到服务

---

**三、高响应比优先（HRRN）**

> **Highest Response Ratio Next**，综合考虑等待时间和服务时间，每次调度选择**响应比最高**的进程。

$$
\text{响应比 } R = \frac{\text{等待时间} + \text{要求服务时间}}{\text{要求服务时间}} = 1 + \frac{\text{等待时间}}{\text{要求服务时间}}
$$

- **调度方式**：非抢占式
- **是否饥饿**：不会

#### 响应比的三种情形：

| 条件 | 响应比表现 | 偏好的进程 |
|:--|:--|:--|
| 等待时间相同时 | $R \propto 1/T_{\text{运行}}$，运行时间越短 $R$ 越大 | 短进程（类似 SJF） |
| 运行时间相同时 | $R \propto T_{\text{等待}}$，等待越久 $R$ 越大 | 先到的进程（类似 FCFS） |
| 长作业等待够久 | $R$ 随等待增长，最终被选中 | **不会饥饿** |

> HRRN 是 FCFS 和 SJF 的折中方案：兼顾短作业优先和长作业公平，是早期批处理系统中较为理想的调度算法。

---

**批处理三算法对比**

| 算法 | 抢占？ | 优点 | 缺点 | 饥饿？ |
|:--|:--|:--|:--|:--|
| **FCFS** | 非抢占 | 公平，实现简单 | 对短作业不利（Convoy 效应） | 不会 |
| **SJF / SRTN** | SJF 非抢占 / SRTN 抢占 | 平均等待/周转时间最优 | 对长作业不利，需预估运行时间 | **会** |
| **HRRN** | 非抢占 | 兼顾等待与运行时间，不饥饿 | 需计算响应比，有额外开销 | 不会 |

> FCFS、SJF、HRRN 主要适用于**早期批处理系统**，注重**公平性**和**吞吐效率**，不强调交互响应时间。

---

**四、时间片轮转（RR）**

> **Round-Robin**，就绪队列上的进程轮流使用 CPU，每个进程每次最多运行一个**时间片**（quantum），时间片用完则排到队尾重新等待。

- **调度方式**：抢占式（时钟中断触发）
- **就绪队列**：按到达顺序排队（FCFS 队列）

**例题**：

| 进程 | 到达时间 | 运行时间 |
|:--|:--|:--|
| P1 | 0 | 5 |
| P2 | 2 | 4 |
| P3 | 4 | 1 |
| P4 | 5 | 6 |

```
|P1|P2|P1|P3|P2|P4|P1|P4| P4|
0  2  4  6  7  9  11 12 14  16
```

详细轮转过程（括号内为剩余执行时间）：

| 时刻 | 事件 | 就绪队列（头→尾） |
|:--|:--|:--|
| 0 | P1(5) 到达，运行 | P1(5) |
| 2 | P2(4) 到达，P1 时间片用尽 | P2(4)→P1(3) |
| 4 | P3(1) 到达，P2 时间片用尽 | P1(3)→P3(1)→P2(2) |
| 5 | P4(6) 到达（P1 运行中） | P3(1)→P2(2)→P4(6) |
| 6 | P1 时间片用尽 | P3(1)→P2(2)→P4(6)→P1(1) |
| 7 | P3 运行完成（1<2，提前释放 CPU） | P2(2)→P4(6)→P1(1) |
| 9 | P2 时间片用尽且完成 | P4(6)→P1(1) |
| 11 | P4 时间片用尽 | P1(1)→P4(4) |
| 12 | P1 完成 | P4(4) |
| 14 | P4 时间片用尽，队列空，P4 继续运行 | P4(2) |
| 16 | 全部完成 | — |

> 时间片用完时进程排到队尾，但如果就绪队列为**空**（其他进程都在等待），当前进程**不会被剥夺**，而是继续分配下一时间片——这避免了不必要的上下文切换。

#### 时间片大小的影响

| 时间片大小 | 影响 |
|:--|:--|
| **太大** | 退化为 FCFS；响应时间变差 |
| **太小** | 进程切换频繁，系统开销大；实际执行时间占比下降 |
| **合理**（略大于典型交互所需时间） | 兼顾响应时间和 CPU 利用率 |

---

**五、优先级调度**

> 每个进程绑定一个优先级，调度时选择**优先级最高**的进程运行。

- **调度方式**：既有抢占式也有非抢占式

#### 优先级分类

| 类型 | 说明 |
|:--|:--|
| **静态优先级** | 创建时确定，整个生命周期不变 |
| **动态优先级** | 运行时可根据情况动态调整（如 HRRN 的响应比可视为动态优先级） |

#### 优先级设置一般原则

| 准则 | 示例 |
|:--|:--|
| 系统进程 > 用户进程 | — |
| 交互型进程 > 非交互型（计算型）| 保证交互响应 |
| I/O 型进程 > 计算型进程 | I/O 设备慢，尽早启动 I/O 可提高设备利用率 |

#### 饥饿问题

> 静态优先级调度**会导致饥饿**（starvation）：低优先级进程可能长期得不到 CPU。
> **解决方案**——**老化（Aging）**：随时间推移逐步提高等待进程的优先级，确保最终会运行。

---

**六、多级反馈队列（MLFQ）**

> **Multilevel Feedback Queue**，是对 FCFS、RR、SJF 的综合折中，也是最复杂实用的调度算法（UNIX 系统采用）。

#### 算法机制

```
优先级 高                            优先级 低
  Q1 (RR, 时间片=1)  ──时间片用尽──→  Q2 (RR, 时间片=2)  ──时间片用尽──→  Q3 (RR, 时间片=4)
       ↑ 新进程进入                          ↑                                   ↑
       │                                     │                                   │
       └───── 更高优先级队列非空时抢占 ────────┘───────────────────────────────────┘
```

**规则（以 3 级队列为例）：**

1. 设 $n$ 级就绪队列，优先级 $Q_1 > Q_2 > \dots > Q_n$，时间片 $T_1 < T_2 < \dots < T_n$
2. 新进程到达 → 进入最优先的 **$Q_1$ 队尾**（FCFS）
3. 每次调度**优先从 $Q_1$** 开始寻找可运行进程
4. $Q_k$ 进程中用完时间片仍未完成 → **降级**到 $Q_{k+1}$ 队尾
5. 最下级 $Q_n$ 上用完时间片 → 回到 $Q_n$ 队尾（不再降级，Round-Robin）
6. **仅当 $Q_1 \sim Q_{k-1}$ 全部为空**时，才调度 $Q_k$ 中的进程
7. **抢占规则**：$Q_k$ 进程运行期间，若 $Q_1 \sim Q_{k-1}$ 中有新进程进入，新进程**抢占** CPU，原进程放回 $Q_k$ 队尾

**例题**：

| 进程 | 到达 | 运行 |
|:--|:--|:--|
| P1 | 0 | 8 |
| P2 | 1 | 4 |
| P3 | 5 | 1 |

设置 Q1($T=1$)、Q2($T=2$)、Q3($T=4$)，执行序列：

```
P1(Q1:1)→P2(Q1:1)→P1(Q2:2)→P2(Q2:1)→P3(Q1:1)→P2(Q3:2)→P1(Q3:4)→P1(Q3:1)
```

#### 优缺点分析

| 特性 | 体现 |
|:--|:--|
| **公平** | 多级队列各自 Round-Robin，不会让任一进程长期饥饿（FCFS 优点） |
| **响应快** | 新进程一律进入 Q1，首次响应时间短（RR 优点） |
| **短进程优先** | 短进程在 Q1/Q2 即可完成，无需降级（SJF 优点） |
| **无需预估** | 不必事先估计运行时间，通过"用不完降级"自动区分长短进程 |
| **灵活可调** | 队列数、各段时间片大小、降级策略均可配置 |
| **缺点** | 实现相对复杂；最下级队列上的长进程可能饥饿（因会被更高优先级队列**持续抢占**） |

> MLFQ 并非完全不饥饿。如果上级队列持续有新进程进入，最下级的长进程可能久久得不到 CPU。实际系统通过设置时间片上限、老化等机制缓解。

---

**全六种算法对比表**

| 算法 | 调度方式 | 适用场景 | 优点 | 缺点 | 饥饿 |
|:--|:--|:--|:--|:--|:--|
| **FCFS** | 非抢占 | 批处理 | 公平，简单 | 短作业等待长（Convoy 效应） | 不会 |
| **SJF** | 非抢占 | 批处理 | 平均等待/周转最优 | 对长作业不利，需预估运行时间 | **会** |
| **SRTN** | 抢占 | 批处理 | 考虑到达时间时平均等待/周转最优 | 需预估剩余时间，抢占开销 | **会** |
| **HRRN** | 非抢占 | 批处理 | 兼顾等待与运行时间，不饥饿 | 需每次计算响应比 | 不会 |
| **RR** | 抢占 | 分时/交互式 | 公平，每个进程都有响应 | 频繁切换有开销；无优先级区分 | 不会 |
| **优先级** | 抢占/非抢占 | 实时/交互式 | 区分紧急程度，灵活 | 低优先级可能饥饿 | **会** |
| **MLFQ** | 抢占 | 通用（UNIX） | 综合 FCFS+RR+SJF 优点，无需预估 | 实现复杂，下级可能饥饿 | 可能 |

### 2.3 进程同步与互斥

> 并发进程间存在两种制约关系：**直接制约（同步）** 与 **间接制约（互斥）**。同步互斥是进程管理核心考点，几乎每套 408 真题均有 PV 操作大题。

---

#### 2.3.1 同步与互斥的基本概念

**一、进程同步**

进程具有 **异步性**：各并发执行的进程以各自独立的、不可预知的速度向前推进。

> **同步**（直接制约关系）：多个进程在某些操作上需要按照一定的 **先后顺序** 执行。

**示例：**

```
进程A：① 从缓冲区读数据 → ② 数据预处理 → ③ 写回缓冲区 → ④ 其他任务
进程B：① 准备工作 → ② 从缓冲区读数据 → ③ 后续处理
```

进程 B 的 ② 必须在进程 A 的 ③ **之后** 执行——这就是 **同步关系**（前驱后继）。

**二、进程互斥**

> **临界资源**：一个时间段内只允许 **一个进程** 访问的资源（如打印机、摄像头、共享变量、内存缓冲区等）。
>
> **互斥**（间接制约关系）：当一个进程访问某临界资源时，另一个想要访问该资源的进程 **必须等待**。

#### 临界区的逻辑结构

对临界资源的互斥访问在逻辑上分为四个部分：

```c
do {
    /* 进入区 */     // 检查是否可进入，设置"上锁"标志
    /* 临界区 */     // 访问临界资源的代码段
    /* 退出区 */     // 解除"锁"标志
    /* 剩余区 */     // 做其他处理
} while (true);
```

| 部分 | 说明 |
|------|------|
| **进入区（entry section）** | 检查可否进入临界区，设置访问标志（"上锁"） |
| **临界区（critical section）** | 访问临界资源的代码段（也称"临界段"） |
| **退出区（exit section）** | 解除访问标志（"解锁"） |
| **剩余区（remainder section）** | 做其他处理 |

#### 互斥访问的四大准则

> | 准则 | 含义 |
> |------|------|
> | **空闲让进** | 临界区空闲时，允许一个请求进入的进程 **立即进入** |
> | **忙则等待** | 已有进程在临界区时，其他进程 **必须等待** |
> | **有限等待** | 请求访问的进程能在 **有限时间内** 进入临界区（不饥饿） |
> | **让权等待** | 不能进入临界区时，应立即 **释放处理机**，防止忙等 |

> **让权等待** 不是所有算法都能满足。Peterson 算法、整型信号量均不满足让权等待。

---

**三、实现互斥的软件方法**

#### 3.1 单标志法

每个进程进入临界区的权限只能被另一个进程赋予。用一个公共整型变量 `turn` 指示允许进入临界区的进程编号。

```c
int turn = 0;  // 初始允许 P0 进入

// 进程 P0
while (turn != 0);   // 进入区：忙等
critical_section();
turn = 1;            // 退出区：权限交给 P1

// 进程 P1
while (turn != 1);   // 进入区：忙等
critical_section();
turn = 0;            // 退出区：权限交给 P0
```

> 单标志法 **违反"空闲让进"**：若 P0 一直不进入临界区，turn 永远为 0，P1 无法进入——即使临界区空闲。

#### 3.2 双标志先检查法

每个进程设置自己的标志 `flag[i]`，先检查对方标志再设置自己的标志。

```c
bool flag[2] = {false, false};

// 进程 Pi
while (flag[j]);      // 进入区：先检查对方是否想进入
flag[i] = true;       // 进入区：再设置自己的标志
critical_section();
flag[i] = false;      // 退出区

// 进程 Pj 同理
```

> "检查"和"设置"两个操作 **不是原子操作**。若 Pi 检查 flag[j]==false 后、设置 flag[i]=true 前发生切换，Pj 也检查到 flag[i]==false，两者将 **同时进入临界区**，违反"忙则等待"。

#### 3.3 双标志后检查法

先设置自己的标志，再检查对方标志，解决先检查后设置的竞态。

```c
bool flag[2] = {false, false};

// 进程 Pi
flag[i] = true;       // 进入区：先设置自己的标志
while (flag[j]);      // 进入区：再检查对方
critical_section();
flag[i] = false;      // 退出区
```

> 若 Pi 和 Pj 近乎同时设置 flag[i]=true 和 flag[j]=true，接着各自检查对方——发现对方 flag 也为 true，双双进入 while 忙等，导致 **死锁**（违反"有限等待"和"让权等待"）。

#### 3.4 Peterson 算法

结合单标志和双标志思想："flag 表示意愿 + turn 表示谦让"。

```c
bool flag[2] = {false, false};
int turn = 0;

// 进程 Pi（P0: i=0,j=1; P1: i=1,j=0）
flag[i] = true;                  // ① 表明自己想进入
turn = j;                        // ② 谦让对方先进入
while (flag[j] && turn == j);    // ③ 对方想进 且 谦让对方 → 等待
critical_section();
flag[i] = false;                 // ④ 退出区
```

> Peterson 算法利用 `flag[2]` 表示进程进入意愿，`turn` 解决冲突——当两个进程同时想进入时，`turn` 最终记录的值对应的进程将"谦让"。该算法满足"空闲让进"、"忙则等待"、"有限等待"，但 **不满足"让权等待"**（`while` 循环会忙等）。

---

**四、实现互斥的硬件方法**

#### 4.1 中断屏蔽法（关中断）

```c
关中断;              // 进入区
critical_section();
开中断;              // 退出区
```

> - 只适用于单处理机系统
> - 滥用关中断会严重影响系统效率
> - 不适用于多 CPU（关中断只关当前 CPU，其他 CPU 仍可访问临界区）

#### 4.2 Test-and-Set 指令（TS / TSL）

```c
// 硬件原子指令：读出标志并设为 true
bool TestAndSet(bool *lock) {
    bool old = *lock;
    *lock = true;
    return old;
}

// 使用
while (TestAndSet(&lock));   // 进入区
critical_section();
lock = false;                 // 退出区
```

> 优点：由硬件直接支持，实现简单；适用于多处理机。缺点：**不满足"让权等待"**（while 循环忙等）。

#### 4.3 Swap / XCHG 指令

```c
// 硬件原子指令：交换两个变量的值
void Swap(bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}

// 使用
bool key = true;
do {
    Swap(&lock, &key);        // 若 lock 原为 true，key 被置 true，循环继续
} while (key == true);        // key 为 false 时成功进入
critical_section();
lock = false;
```

> 与 TS 指令类似，不满足"让权等待"。

---

#### 2.3.2 信号量机制

> 1965 年，荷兰学者 Dijkstra 提出 **信号量机制**，解决了软件/硬件互斥方案的两个问题：
> 1. 进入区的"检查"与"上锁"操作无法一气呵成
> 2. 无法实现"让权等待"

**一、整型信号量**

```c
int S = 1;               // 初始化整型信号量

void wait(int S) {        // P 操作（申请资源）
    while (S <= 0);       // 资源不够，忙等
    S = S - 1;
}

void signal(int S) {      // V 操作（释放资源）
    S = S + 1;
}
```

| 项目 | 说明 |
|------|------|
| 优点 | "检查"和"上锁"一气呵成（原语实现），避免并发问题 |
| 缺点 | **不满足"让权等待"**（while 循环忙等） |

> 所谓"一气呵成"是指 P/V 原语内部由关中断实现，不会被中断。但整型信号量的 P 操作在资源不足时会让进程忙等，CPU 空转。

**二、记录型信号量 ⭐⭐⭐**

解决整型信号量的"忙等"问题，引入阻塞队列。

```c
typedef struct {
    int value;             // 剩余资源数
    struct process *L;     // 等待队列（阻塞队列）
} semaphore;

void wait(semaphore *S) {  // P 操作
    S->value--;
    if (S->value < 0) {
        block(S->L);       // 将当前进程加入等待队列，运行态→阻塞态
    }
}

void signal(semaphore *S) { // V 操作
    S->value++;
    if (S->value <= 0) {
        wakeup(S->L);      // 唤醒等待队列第一个进程，阻塞态→就绪态
    }
}
```

#### S->value 的含义 ⭐⭐⭐

| S->value 的值 | 含义 |
|:---:|---|
| $>0$ | 还有空闲资源，`value` 即为可用资源数 |
| $=0$ | 资源恰好分配完，无等待进程 |
| $<0$ | 资源已分配完，有 $\lvert \text{value}\rvert$ 个进程在等待 |

#### 举例：系统中有 2 台打印机

```
初始：S->value = 2, S->L = null

P1 执行 P(S)：S->value = 1   （剩 1 台打印机）
P2 执行 P(S)：S->value = 0   （打印机恰好分配完）
P3 执行 P(S)：S->value = -1  （1 个进程在等待，P3 被阻塞）
P4 执行 P(S)：S->value = -2  （2 个进程在等待，P4 被阻塞）

P1 释放打印机 V(S)：S->value = -1+1 = 0
  → S->value + 1 <= 0，说明有进程在等待 → 唤醒 P3

P2 释放打印机 V(S)：S->value = 0+1 = 1
  → S->value + 1 > 0，已无等待进程
```

#### 用记录型信号量实现互斥

```c
semaphore mutex = 1;

// 进程 Pi
P(mutex);       // 加锁
/* 临界区 */
V(mutex);       // 解锁
```

#### 用记录型信号量实现同步

```c
semaphore S = 0;          // 同步信号量初值 0

// 进程 P1（先执行的操作）
/* 操作代码1... */
V(S);                     // "前 V"：通知 P2

// 进程 P2（后执行的操作）
P(S);                     // "后 P"：等待 P1 完成
/* 操作代码2... */
```

> 同步口诀：**前 V 后 P**——先执行的操作后做 V，后执行的操作先做 P。

> - P 操作可能阻塞进程，V 操作不会阻塞
> - 实现互斥的 P 操作必须在实现同步的 P 操作 **之后**，否则可能死锁
> - 408 考试中 P(S)/V(S) 默认 S 为记录型信号量

---

**三、AND 型信号量**

> 某些场合进程需要同时获得两个或以上共享资源才能执行。若分别申请，可能出现 **死锁**。AND 同步机制的基本思想：将进程在整个运行过程中所需的 **所有临界资源一次性全部分配**，使用完后一起释放。

```c
// AND 型信号量——Swait 和 Ssignal
void Swait(semaphore S1, semaphore S2, ..., semaphore Sn) {
    if (S1->value >= 1 && S2->value >= 1 && ... && Sn->value >= 1) {
        for (i = 1; i <= n; i++)
            Si->value--;
    } else {
        // 将进程放入第一个不满足条件的资源的等待队列
        // 同时将进程的程序计数器设置为 Swait 开头（重新判断）
        block();
    }
}

void Ssignal(semaphore S1, semaphore S2, ..., semaphore Sn) {
    for (i = 1; i <= n; i++) {
        Si->value++;
        // 唤醒所有因该资源而阻塞的进程
        wakeup(Si->L);
    }
}
```

> AND 型信号量确保"要么全部分配，要么一个都不分配"，从根本上消除了因同时持有部分资源等待另一部分资源而导致的死锁。

---

**四、信号量集**

> 信号量集是对 AND 型信号量的推广。对每种资源的需求不限于 1 个，可以一次性申请多个，且可设置一个 **下限值**（低于该值不分配）。

```c
// 信号量集 Swait
// S：信号量；t：下限值（低于 t 不分配）；d：需求量
void Swait(semaphore S1, int t1, int d1,
           semaphore S2, int t2, int d2,
           ..., semaphore Sn, int tn, int dn) {
    if (S1->value >= t1 && S2->value >= t2 && ... && Sn->value >= tn) {
        for (i = 1; i <= n; i++)
            Si->value -= di;
    } else {
        block();
    }
}

void Ssignal(semaphore S1, int d1, ..., semaphore Sn, int dn) {
    for (i = 1; i <= n; i++) {
        Si->value += di;
        wakeup(Si->L);
    }
}
```

**特殊情况：**

| 参数取值 | 等价于 |
|:---:|------|
| $t=1,\ d=1$ | 普通记录型信号量 `wait(S)` |
| $t=1,\ d=1,\ n>1$ | AND 型信号量 |
| $t=0,\ d=1,\ n=1$ | 可控开关（$S\geq 0$ 时允许进入） |

#### 2.3.3 管程

> **管程（Monitor）** 是由一组局部于管程的共享数据结构和对该数据结构进行操作的一组过程（函数）所组成的软件模块。管程将分散在各个进程中的临界区集中起来统一管理，进程只能通过调用管程中的过程来间接访问共享数据。

**一、管程的组成**

1. **局部于管程的共享变量**：只能被管程内的过程访问
2. **对共享数据结构进行操作的一组过程**（函数）
3. **初始化代码**：对管程内共享变量初始化

> 管程的基本特性：**每次仅允许一个进程进入管程**，从而自动实现互斥——进入管程时的互斥由编译器或运行时自动保证，程序员无需显式编写 PV 操作。

**二、条件变量**

> 当进程进入管程却因某条件不满足需要等待时，管程提供 **条件变量（condition variable）** 来阻塞并释放管程使用权。条件变量上的两种操作：
> - `x.wait`：调用进程自己阻塞，释放管程，让其他进程进入
> - `x.signal`：唤醒一个因 `x.wait` 被阻塞的进程

> `x.signal` 与信号量的 `signal` 不同：若没有进程因 `x` 阻塞，`x.signal` 不做任何事（不保存信号）。这就是二者的本质区别。

**三、管程解决生产者-消费者**

```c
monitor ProducerConsumer {
    condition full, empty;
    int count = 0;  // 缓冲区中产品个数

    void insert(Item item) {
        if (count == N)  empty.wait();  // 缓冲区满，等待
        /* 放入产品 */
        count++;
        if (count == 1)  full.signal(); // 有了第一个产品，唤醒消费者
    }

    Item remove() {
        if (count == 0)  full.wait();   // 缓冲区空，等待
        /* 取出产品 */
        count--;
        if (count == N-1)  empty.signal(); // 有了空位，唤醒生产者
        return item;
    }

    // 初始化
    { count = 0; }
}

// 生产者进程
producer() {
    while (1) {
        item = 生产一个产品;
        ProducerConsumer.insert(item);
    }
}

// 消费者进程
consumer() {
    while (1) {
        item = ProducerConsumer.remove();
        消费产品 item;
    }
}
```

> 与信号量方案对比：管程将同步互斥逻辑封装在模块内部，进程调用时无需关心 PV 操作细节，降低了编程复杂度。

#### 2.3.4 经典同步问题

**一、生产者-消费者问题**

#### 1.1 多缓冲（n 个缓冲区）

> 一组生产者进程和一组消费者进程共享一个初始为空、大小为 $n$ 的缓冲区。生产者每次放入一个产品，消费者每次取出一个产品。

**关系分析：**

- 同步关系（两对）：① 生产者 → 消费者（有产品才能消费）；② 消费者 → 生产者（有空位才能生产）
- 互斥关系：所有进程对缓冲区的访问互斥

```c
semaphore mutex = 1;  // 互斥信号量
semaphore empty = n;  // 同步信号量，空闲缓冲区数
semaphore full = 0;   // 同步信号量，产品数量

producer() {
    while (1) {
        生产一个产品;
        P(empty);                // 申请一个空闲缓冲区
        P(mutex);                // 加锁
        把产品放入缓冲区;
        V(mutex);                // 解锁
        V(full);                 // 产品数+1，通知消费者
    }
}

consumer() {
    while (1) {
        P(full);                 // 申请一个产品
        P(mutex);                // 加锁
        从缓冲区取出一个产品;
        V(mutex);                // 解锁
        V(empty);                // 空闲缓冲区+1，通知生产者
        使用产品;
    }
}
```

> **P 操作顺序至关重要！** 必须 `P(同步信号量) → P(互斥信号量)`。若交换为 `P(mutex)` 在前——当缓冲区满时，生产者先拿到 mutex 再在 `P(empty)` 上阻塞，消费者因无法获得 mutex 而无法 `V(empty)` → **死锁**。V 操作的顺序则可交换，因为 V 不会阻塞进程。

#### 1.2 单缓冲（缓冲大小为 1）

```c
semaphore mutex = 1;
semaphore empty = 1;  // 缓冲区大小为 1
semaphore full = 0;

// 生产者/消费者进程代码同上，仅 empty 初值由 n 变为 1
```

#### 1.3 PV 题目解题三步法

> **第一步：关系分析** → 找出各进程间的同步、互斥关系
> **第二步：整理思路** → 根据操作流程确定 P/V 操作的大致顺序
> **第三步：设置信号量** → 互斥信号量初值一般为 1，同步信号量初值为对应资源的初始数量

> **核心口诀：实现"一前一后"需要"前 V 后 P"。**

**二、读者-写者问题**

#### 2.1 问题描述

> 读者和写者共享文件。多个读者可同时读（不改变数据），但写者与任何进程（读或写）不能同时访问。

**约束条件：**
1. 允许多个读者同时读
2. 只允许一个写者写
3. 写者写时，不允许其他读者/写者访问
4. 写者写前，应让已有读者和写者全部退出

#### 2.2 方案一：读者优先

```c
semaphore rw = 1;        // 文件互斥访问
int count = 0;           // 当前读进程数
semaphore mutex = 1;     // 保护 count

reader() {
    while (1) {
        P(mutex);
        if (count == 0)  P(rw);   // 第一个读者负责"上锁"
        count++;
        V(mutex);

        读文件...;

        P(mutex);
        count--;
        if (count == 0)  V(rw);   // 最后一个读者负责"解锁"
        V(mutex);
    }
}

writer() {
    while (1) {
        P(rw);                    // 写前上锁
        写文件...;
        V(rw);                    // 写后解锁
    }
}
```

> 只要有读者在读，新读者就能继续进入——写者可能 **永远阻塞（饥饿）**。该方案称"读者优先"。

#### 2.3 方案二：读写公平法

引入排队信号量 `w` 实现先来先服务，阻止写者饥饿：

```c
semaphore rw = 1;
int count = 0;
semaphore mutex = 1;
semaphore w = 1;          // 排队信号量

reader() {
    while (1) {
        P(w);                       // 申请排队
        P(mutex);
        if (count == 0)  P(rw);
        count++;
        V(mutex);
        V(w);                       // 释放排队（让后续进程竞争）

        读文件...;

        P(mutex);
        count--;
        if (count == 0)  V(rw);
        V(mutex);
    }
}

writer() {
    while (1) {
        P(w);                       // 申请排队
        P(rw);
        写文件...;
        V(rw);
        V(w);                       // 释放排队
    }
}
```

> 并发情况分析：只要读/写进程先抢到 `w`，后续同类进程若也抵达 `P(w)`，则看谁先获得—实现了 **先来先服务**，写者不会饥饿。不是真正的"写优先"，而是读写公平法。

> - `count` 是普通整型变量，不是信号量，必须用 mutex 保护
> - 第一个读者 `P(rw)`，最后一个读者 `V(rw)`——这个"首尾判断"是核心设计
> - 读者不改变数据是允许同时读的前提

**三、哲学家进餐问题**

#### 3.1 问题描述

> 5 名哲学家围坐圆桌，每两人之间放一根筷子（共 5 根）。哲学家只有同时拿到左右两根筷子才能进餐。每个哲学家进程需要 **同时持有两个临界资源**，核心问题是 **避免死锁**。

#### 3.2 朴素实现（存在死锁！）

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};

Pi() {
    while (1) {
        P(chopstick[i]);           // 拿左筷子
        P(chopstick[(i+1)%5]);     // 拿右筷子
        吃饭...;
        V(chopstick[i]);           // 放左
        V(chopstick[(i+1)%5]);     // 放右
        思考...;
    }
}
```

**死锁：** 若 5 位哲学家 **同时拿起左手边的筷子**，每人都在等右侧的筷子 → 死锁！

#### 3.3 三种解决方案

**方案一：限制同时进餐人数**（最多 4 人）

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};
semaphore limit = 4;                // 最多 4 人同时进餐

Pi() {
    while (1) {
        P(limit);                   // 限制人数
        P(chopstick[i]);
        P(chopstick[(i+1)%5]);
        V(limit);                   // 释放限制
        吃饭...;
        V(chopstick[i]);
        V(chopstick[(i+1)%5]);
        思考...;
    }
}
```

**方案二：奇偶号不同拿筷子顺序**（破坏循环等待）

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};

Pi() {
    while (1) {
        if (i % 2 == 1) {           // 奇数号：先左后右
            P(chopstick[i]);
            P(chopstick[(i+1)%5]);
        } else {                    // 偶数号：先右后左
            P(chopstick[(i+1)%5]);
            P(chopstick[i]);
        }
        吃饭...;
        V(chopstick[i]);
        V(chopstick[(i+1)%5]);
        思考...;
    }
}
```

**方案三：互斥取筷子**（保证一次性获取两根）

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};
semaphore mutex = 1;                // 互斥地取筷子

Pi() {
    while (1) {
        P(mutex);                   // 原子取筷
        P(chopstick[i]);
        P(chopstick[(i+1)%5]);
        V(mutex);
        吃饭...;
        V(chopstick[i]);
        V(chopstick[(i+1)%5]);
        思考...;
    }
}
```

#### 3.4 方案对比

| 方案 | 思想 | 优点 | 缺点 |
|------|------|------|------|
| 限制进餐人数 | 最多 4 人同时竞争 | 实现简单 | 并发度降低 |
| 奇偶号不同顺序 | 破坏循环等待条件 | 并发度较高 | 代码稍复杂 |
| 互斥取筷子 | 保证原子性一次性获取 | 实现最简单 | 并发度最低（每次只一人取筷） |

> - 朴素实现直接导致死锁，考试中 **绝对不能** 使用
> - 死锁是哲学家进餐问题的 **核心考点**
> - 方案二（奇偶号）是 **最常考的解法**

**四、吸烟者问题**

> 一个供应者和三个吸烟者。供应者每次提供两种材料（桌上随机放两种），缺少第三种材料的吸烟者取走并制作香烟，吸完后通知供应者继续提供下一轮。

**问题描述：**
- 供应者：无限供应三种材料（烟草、纸、胶水），每次在桌上放 **任意两种**
- 吸烟者 1：拥有烟草，需要纸 + 胶水
- 吸烟者 2：拥有纸，需要烟草 + 胶水
- 吸烟者 3：拥有胶水，需要烟草 + 纸

```c
semaphore offer1 = 0;    // 对应吸烟者 1（纸+胶水）
semaphore offer2 = 0;    // 对应吸烟者 2（烟草+胶水）
semaphore offer3 = 0;    // 对应吸烟者 3（烟草+纸）
semaphore finish = 0;    // 吸烟完成，通知供应者
int random;              // 随机数（0/1/2）

provider() {
    while (1) {
        random = 产生一个随机数(0, 1, 2);
        if (random == 0)       V(offer1);   // 放纸+胶水
        else if (random == 1)  V(offer2);   // 放烟草+胶水
        else                   V(offer3);   // 放烟草+纸

        将对应两种材料放在桌上;
        P(finish);               // 等待吸烟者完成
    }
}

smoker1() {   // 拥有烟草，缺纸+胶水
    while (1) {
        P(offer1);
        从桌上取走纸和胶水; 制作香烟; 吸烟;
        V(finish);               // 通知供应者
    }
}

// smoker2、smoker3 同理，对应 P(offer2) 和 P(offer3)
```

> 吸烟者问题与生产者-消费者的区别：生产的是"两种特定材料的组合"，消费者各取所需。这里 `offer1/offer2/offer3` 相当于三种不同的产品信号量。核心在于 **正确判断同步关系**——供应者每次只能被缺少对应材料的那个吸烟者唤醒。

#### 2.4.1 死锁的概念

**一、死锁的定义与产生原因**

**1. 死锁的定义**

死锁至少涉及**两个或两个以上**的进程，发生死锁的进程一定处于**阻塞态**。

> 5 位哲学家围坐一桌，每位哲学家左右各有一根筷子。每位哲学家都先拿起左手边的筷子，然后试图拿起右手边的筷子。若 5 人同时行动，则每人持有一根筷子并永远等待右侧筷子——发生死锁。

**2. 死锁产生的原因**

1. **竞争不可剥夺资源**：各进程对不可剥夺资源（如打印机）的竞争可能引起死锁。对可剥夺资源（CPU）的竞争不会引起死锁。
2. **进程推进顺序非法**：请求和释放资源的顺序不当。如 P1 持有 R1 申请 R2，P2 持有 R2 申请 R1，二者互相阻塞。
3. **信号量使用不当**：如生产者-消费者问题中，若互斥 P 操作在同步 P 操作之前，可能导致死锁（可以把互斥/同步信号量也视为抽象的系统资源）。

> 总之，对**不可剥夺资源的不合理分配**，可能导致死锁。

---

**二、死锁产生的四个必要条件 ⭐⭐⭐**

> 产生死锁必须**同时满足**以下四个条件，只要其中**任一条件不成立**，死锁就不会发生。

**1. 互斥条件**

只有对必须**互斥使用的资源**的争抢才会导致死锁。例如打印机、哲学家的筷子。像内存、扬声器等可同时共享的资源不会导致死锁。

**2. 不剥夺条件**

进程所获得的资源在未使用完之前，**不能由其他进程强行夺走**，只能主动释放。

**3. 请求和保持条件**

进程已经保持了**至少一个资源**，但又提出了**新的资源请求**，而该资源又被其他进程占有，此时请求进程被阻塞，但又对自己已有的资源**保持不放**。

**4. 循环等待条件**

存在一种进程资源的**循环等待链**，链中的每一个进程已获得的资源同时被下一个进程所请求。

> 发生死锁时一定有循环等待，但是发生循环等待时**未必死锁**（循环等待是死锁的**必要不充分条件**）。
>
> - 如果同类资源数大于 1，则即使有循环等待，也未必发生死锁。
> - 但如果系统中**每类资源都只有一个**，那循环等待就是死锁的**充分必要条件**。

---

**三、死锁、饥饿、死循环的区别 ⭐⭐**

| 特性 | 死锁 | 饥饿 | 死循环 |
|------|------|------|--------|
| **进程数量** | 至少两个 | 可能只有一个 | 可能只有一个 |
| **进程状态** | 一定是**阻塞态** | 可能是阻塞态或就绪态 | 可以是**运行态** |
| **产生原因** | OS 分配资源策略不合理 | OS 分配资源策略不合理 | 代码逻辑错误 |
| **本质** | 管理者（OS）的问题 | 管理者（OS）的问题 | 被管理者的问题 |

- **饥饿**：在 SPF 算法中，若源源不断有短进程到达，长进程一直得不到处理机。
- **死循环**：程序中存在 `while(true)` 且无跳出条件，可以是程序 bug，也可以是有意设计（如服务器主循环）。

---

**四、死锁的处理策略概览**

| 策略 | 思想 | 实现难度 | 资源利用率 |
|------|------|:---:|:---:|
| **预防死锁** | 破坏死锁四个必要条件中的一个或几个 | 较简单 | 较低 |
| **避免死锁** | 防止系统进入不安全状态（银行家算法） | 中等 | 中等 |
| **检测和解除** | 允许死锁发生，检测后采取措施解除 | 较复杂 | 较高 |

---

#### 2.4.2 死锁预防

死锁预防的核心思路是**破坏四个必要条件之一**。

**一、破坏互斥条件**

**思路**：让资源可以同时访问（如 SPOOLing 技术将独占设备改造为共享设备）。

**局限性**：并非所有资源都能改造为可共享，如打印机在实际打印时仍需互斥。该方法**不太现实**。

**二、破坏不剥夺条件**

**思路**：当进程请求新资源失败时，必须释放已持有的所有资源，之后重新申请。或 OS 可强行夺走进程持有的资源。

**缺点**：
- 实现复杂，需要保存和恢复进程状态
- 增加系统开销，可能导致进程反复申请和释放

**三、破坏请求和保持条件（静态分配法）**

**思路**：进程在运行前**一次性申请**所有需要的资源，若全部可用则分配并运行，否则一个都不分配，进程等待。

**优点**：简单、安全。
**缺点**：
- 资源利用率低——进程可能长期占用但暂不使用某些资源
- 可能发生**饥饿**——所需资源较多的进程可能长期得不到全部资源

**四、破坏循环等待条件（顺序资源分配法）**

**思路**：将系统中所有资源编号，进程必须**按编号递增顺序**申请资源。若已持有编号为 $i$ 的资源，只能申请编号 $> i$ 的资源。

> 按编号递增申请意味着不可能出现 $P_1 \to P_2 \to \cdots \to P_1$ 的环路——因为在回路中必然存在从编号大的资源向编号小的资源的请求，这与"只能递增申请"矛盾。

**优点**：相比前三种方法，资源利用率和系统吞吐量有明显改善。
**缺点**：
- 资源编号难以合理确定
- 新增设备可能需要重新编号
- 限制了用户编程的灵活性
- 若进程实际使用顺序与编号递增顺序不一致，仍需提前申请暂不使用的资源

---

#### 2.4.3 死锁避免

> 死锁避免不破坏必要条件，而是在资源分配过程中，通过**预判**分配后系统是否会进入不安全状态来决定是否分配资源。

---

**一、系统安全状态**

**1. 安全序列的定义**

若系统按某种进程顺序 $\langle P_1, P_2, \dots, P_n \rangle$ 分配资源，使得每个进程 $P_i$ 所需的资源不超过系统当前可用资源与所有排在它前面的进程已占有资源之和，则该序列为**安全序列**。

- 安全序列可能有多个
- 只要能找到**一个**安全序列，系统就是**安全状态**

**2. 安全状态与死锁的关系**

| 状态 | 含义 | 后果 |
|------|------|------|
| **安全状态** | 存在至少一个安全序列 | **一定不会发生死锁** |
| **不安全状态** | 不存在任何安全序列 | **可能发生死锁**（不是一定） |

$$
\text{安全状态} \xrightarrow{\text{一定不}} \text{死锁}
\qquad
\text{不安全状态} \xrightarrow{\text{可能}} \text{死锁}
\qquad
\text{死锁} \Rightarrow \text{不安全状态}
$$

> - 安全状态 $\neq$ 没有死锁，而是说一定不会发生死锁。
> - 不安全状态 $\neq$ 死锁，只是有可能发生。
> - 死锁发生时系统一定处于不安全状态（逆否命题不成立）。

---

**二、银行家算法**

银行家算法由 Dijkstra 提出，核心思想：在进程提出资源申请时，**先预判**此次分配是否会导致系统进入不安全状态。如果会，就**暂时不答应**，让该进程阻塞等待。

**1. 数据结构**

设系统中有 $n$ 个进程、$m$ 种资源：

| 数据结构 | 说明 |
|----------|------|
| **Available** $[m]$ | 系统中当前可用资源向量 |
| **Max** $[n][m]$ | 最大需求矩阵，每个进程对每种资源的最大需求 |
| **Allocation** $[n][m]$ | 已分配矩阵 |
| **Need** $[n][m]$ | 需求矩阵，$\text{Need}[i][j] = \text{Max}[i][j] - \text{Allocation}[i][j]$ |
| **Request** $[m]$ | 进程 $P_i$ 的本次请求向量 |

**2. 银行家算法步骤**

进程 $P_i$ 发出请求 $\text{Request}_i$：

1. **检查是否超过最大需求**：若 $\text{Request}_i[j] > \text{Need}[i][j]$ 对任意 $j$ 成立，则出错（请求超过声明的最大需求）。
2. **检查是否有足够资源**：若 $\text{Request}_i[j] > \text{Available}[j]$ 对任意 $j$ 成立，则 $P_i$ 必须等待。
3. **试探性分配**：
   $$
   \begin{aligned}
   \text{Available} &:= \text{Available} - \text{Request}_i \\
   \text{Allocation}[i] &:= \text{Allocation}[i] + \text{Request}_i \\
   \text{Need}[i] &:= \text{Need}[i] - \text{Request}_i
   \end{aligned}
   $$
4. **执行安全性算法**：若安全则正式分配；否则撤销试探性分配，让 $P_i$ 等待。

**3. 安全性算法**

1. 设工作向量 $\text{Work}[m] = \text{Available}$，完成标志 $\text{Finish}[n] = \text{false}$。
2. 从进程集合中寻找满足以下条件的 $P_i$：
   - $\text{Finish}[i] = \text{false}$
   - $\text{Need}[i][j] \le \text{Work}[j]$ 对所有 $j$ 成立
3. 若找到，则令 $\text{Work} := \text{Work} + \text{Allocation}[i]$，$\text{Finish}[i] := \text{true}$，返回步骤 2。
4. 若所有进程的 $\text{Finish}$ 均为 true，则系统处于安全状态；否则不安全。

**时间复杂度**：$O(n^2 \times m)$。

---

**三、银行家算法完整实例 ⭐⭐⭐**

> 系统有 5 个进程 $P_0 \sim P_4$，3 种资源 $R_0 \sim R_2$，初始数量为 $(10, 5, 7)$。某时刻状态如下：

| 进程 | Max | Allocation | Need (= Max − Allocation) |
|:---:|:---:|:---:|:---:|
| $P_0$ | $(7, 5, 3)$ | $(0, 1, 0)$ | $(7, 4, 3)$ |
| $P_1$ | $(3, 2, 2)$ | $(2, 0, 0)$ | $(1, 2, 2)$ |
| $P_2$ | $(9, 0, 2)$ | $(3, 0, 2)$ | $(6, 0, 0)$ |
| $P_3$ | $(2, 2, 2)$ | $(2, 1, 1)$ | $(0, 1, 1)$ |
| $P_4$ | $(4, 3, 3)$ | $(0, 0, 2)$ | $(4, 3, 1)$ |

**计算 Available：**
$$
\begin{aligned}
\text{已分配总数} &= (0+2+3+2+0,\; 1+0+0+1+0,\; 0+0+2+1+2) = (7, 2, 5) \\
\text{Available} &= (10-7,\; 5-2,\; 7-5) = (3, 3, 2)
\end{aligned}
$$

**安全性算法执行过程：**

| 轮次 | Available | 可满足的进程 | 选择 | 释放后 Available |
|:---:|:---:|:---:|:---:|:---:|
| 1 | $(3, 3, 2)$ | $P_1$（Need $(1,2,2)$ ≤ Avail）、$P_3$（Need $(0,1,1)$ ≤ Avail） | $P_1$ | $(3,3,2)+(2,0,0) = (5,3,2)$ |
| 2 | $(5, 3, 2)$ | $P_3$、$P_4$ | $P_3$ | $(5,3,2)+(2,1,1) = (7,4,3)$ |
| 3 | $(7, 4, 3)$ | $P_0$、$P_2$、$P_4$ | $P_0$ | $(7,4,3)+(0,1,0) = (7,5,3)$ |
| 4 | $(7, 5, 3)$ | $P_2$、$P_4$ | $P_2$ | $(7,5,3)+(3,0,2) = (10,5,5)$ |
| 5 | $(10, 5, 5)$ | $P_4$ | $P_4$ | $(10,5,5)+(0,0,2) = (10,5,7)$ |

> **安全序列：$P_1 \to P_3 \to P_0 \to P_2 \to P_4$**
>
> 系统处于**安全状态**，**一定不会发生死锁**。安全序列不唯一（如也可选 $P_3 \to P_1 \to P_4 \to P_0 \to P_2$），找到一个即可。

---

**四、银行家算法优缺点**

| 优点 | 缺点 |
|------|------|
| 可以避免死锁 | 时间复杂度较高 $O(n^2 \times m)$ |
| 资源利用率较高 | 需要预先知道每个进程的最大需求（实际中难以做到） |
| 实现相对简洁 | 进程数和资源数较多时开销大 |

---

#### 2.4.4 死锁检测与解除

死锁检测与解除策略**允许死锁发生**，系统定期检测是否有死锁，若发现则采取措施解除。

---

**一、资源分配图与死锁定理**

**1. 资源分配图**

资源分配图是描述进程与资源之间关系的**有向图**，由两类结点和两类边组成：

- **进程结点**（圆形 $\bigcirc$）：表示一个进程
- **资源结点**（矩形 $\square$）：表示一类资源，矩形内的**小圆点**表示资源实例数量
- **请求边** $P_i \to R_j$：进程 $P_i$ 申请一个 $R_j$ 资源
- **分配边** $R_j \to P_i$：一个 $R_j$ 资源已分配给进程 $P_i$

**2. 死锁定理**

> **死锁定理**：系统状态 $S$ 为死锁状态的**充分必要条件**是：$S$ 状态的资源分配图是**不可完全简化**的。

**简化步骤**：
1. 在资源分配图中找一个既不阻塞也不孤立的进程结点 $P_i$（即 $P_i$ 的请求资源数 ≤ 系统剩余资源数）。
2. 消去 $P_i$ 的所有请求边和分配边，使其成为孤立结点（释放资源）。
3. 重复上述步骤，直到不能继续。
4. 若所有进程结点都成为孤立结点，则该图**可完全简化**，系统无死锁；否则存在死锁。

> 死锁检测就是对资源分配图进行简化：能完全简化 → 无死锁；不能完全简化 → 有死锁，无法简化的进程即为死锁进程。

---

**二、死锁检测算法**

死锁检测算法与安全性算法思想类似，但不使用 Need 而使用 Request：

**数据结构**（在检测算法中，不要求预知 Max）：
- $\text{Available}[m]$：可用资源向量
- $\text{Allocation}[n][m]$：已分配矩阵
- $\text{Request}[n][m]$：进程当前对各类资源的请求量

**检测步骤**：
1. 设 $\text{Work}[m] = \text{Available}$，$\text{Finish}[n]$ 初始化为：若 $\text{Allocation}[i] \neq 0$ 则 false，否则 true。
2. 寻找满足 $\text{Finish}[i] = \text{false}$ 且 $\text{Request}[i][j] \le \text{Work}[j]$ 对所有 $j$ 成立的进程 $P_i$。
3. 若找到，则 $\text{Work} := \text{Work} + \text{Allocation}[i]$，$\text{Finish}[i] := \text{true}$，返回步骤 2。
4. 若存在 $\text{Finish}[i] = \text{false}$ 的进程，则系统处于**死锁状态**，且 $P_i$ 为死锁进程。

**检测时机**：可定期执行（如每隔一定时间），或在每次资源请求时执行（开销大），或当 CPU 利用率过低时触发。

---

**三、死锁解除方法**

一旦检测到死锁，立即采取措施解除。

**1. 资源剥夺法（抢占资源）**

从某些进程中**强行剥夺**足够数量的资源，分配给死锁进程以解除死锁。

- 需要考虑被剥夺进程的恢复和重新执行问题
- 通常选择代价最小的进程进行剥夺

**2. 撤销进程法（终止进程）**

| 方法 | 操作 | 优点 | 缺点 |
|------|------|------|------|
| **终止所有死锁进程** | 一次性终止全部死锁进程 | 简单直接 | 代价大，已运行的计算全部丢失 |
| **逐个终止** | 按某种顺序逐个终止，每次终止后检测死锁是否解除 | 代价相对较小 | 每次需要重新运行检测算法，开销较大 |

选择终止进程的考虑因素：进程优先级、已运行时间与剩余时间、已使用资源类型与数量、还需要多少资源、进程是交互式还是批处理等。

**3. 进程回退法（回退法/撤销法）**

不终止进程，而是让一个或多个死锁进程**回退到死锁发生前的某个检查点**，释放已分配的资源，然后重新执行。

- 要求系统支持**检查点（Checkpoint）**与**回滚（Rollback）**机制
- 实现复杂度高，需要保存足够的恢复信息
- 可能引发饥饿——同一进程反复被回退

---

### 本章重点总结

**必考关系**

> **四个必要条件**：互斥 + 不剥夺 + 请求保持 + 循环等待，必须同时满足。

> **安全状态与死锁**：
> $$
> \text{安全} \Rightarrow \text{无死锁};\quad
> \text{不安全} \;\not\Rightarrow\; \text{死锁};\quad
> \text{死锁} \Rightarrow \text{不安全}
> $$

> **循环等待**：死锁的必要不充分条件（同类资源数 > 1 时）。

**高频易错**

1. 死锁至少涉及**两个**进程，一个进程不会死锁。
2. 死锁进程一定处于**阻塞态**；饥饿进程可能是阻塞态或**就绪态**；死循环进程可以是**运行态**。
3. **可剥夺资源**（如 CPU）不会导致死锁。
4. 信号量使用不当也可能导致死锁（互斥 P 操作在同步 P 操作之前）。
5. 银行家算法需要**预先知道最大需求**，这是其在实际系统中应用受限的主要原因。
6. 安全性算法可能需要**多轮扫描**，每轮都要找到一个新的可完成进程，否则不安全。
7. 银行家算法中的**试探性分配**若发现不安全，必须**撤销**恢复到分配前状态。

## 第三章 内存管理

> 本章是 408 操作系统**最高频考点**之一，几乎每年必考。选择题 2~3 题（4~6 分）+ 综合题 1 题（约 10 分），重点考查页面置换计算、逻辑地址到物理地址转换、段页式管理及 TLB 命中率计算。

---

### 3.1 内存管理基础

#### 3.1.1 内存管理概述

> **内存管理**：合理分配和使用内存资源。程序执行前需要先放到内存中才能被 CPU 处理，目的之一是**缓和 CPU 与硬盘之间的速度矛盾**。

内存管理的功能：

| 功能 | 描述 |
|------|------|
| **内存分配** | 为进程分配连续/非连续内存空间 |
| **地址转换** | 逻辑地址 → 物理地址（重定位） |
| **内存保护** | 防止进程越界访问 |
| **内存共享** | 共享内存区域 |
| **内存扩充** | 虚拟存储技术（逻辑上扩充内存） |

##### 内存的编址

在多道程序环境下，需要给内存的存储单元编地址以区分各程序的数据。

| 编址方式 | 存储单元大小 | 说明 |
|---------|------------|------|
| **按字节编址** | 1 字节（1B = 8 位） | 最常见的方式 |
| **按字编址** | 1 个字 | 如字长 16 位，则每个存储单元大小为 16 位 |

常用数量单位：

$$
2^{10} = 1\text{K} \quad 2^{20} = 1\text{M} \quad 2^{30} = 1\text{G}
$$

例：4GB 内存 $= 4 \times 2^{30} = 2^{32}$ 个字节，按字节编址需 32 位地址（$0 \sim 2^{32}-1$）。

##### 逻辑地址 vs 物理地址

| 地址类型 | 别名 | 说明 |
|---------|------|------|
| **逻辑地址** | 相对地址 | 相对于进程起始地址的偏移量，程序中使用 |
| **物理地址** | 绝对地址 | 在内存中的实际位置，CPU 最终访问 |

> 程序经过编译、链接后生成的指令中指明的是**逻辑地址**，进程看到的地址都是逻辑地址。物理地址 = 逻辑地址 + 基址寄存器的值。

程序的执行过程：

```
源代码 → 编译 → 链接 → 装入模块（可执行文件） → 装入内存 → CPU 执行
```

##### 程序装入（三种方式）

| 装入方式 | 地址变换时机 | 说明 | 灵活性 | 硬件支持 |
|---------|------------|------|--------|---------|
| **绝对装入** | 编译时 | 编译时确定绝对地址，只适用于**单道程序环境** | 最差 | 无 |
| **可重定位装入（静态重定位）** | 装入时 | 装入时一次性将逻辑地址变换为物理地址；作业必须获得全部内存，装入后不能移动 | 较差 | 无 |
| **动态运行时装入（动态重定位）** | 执行时 | 装入后保持逻辑地址，执行时由重定位寄存器实时转换 | 最好 | 重定位寄存器 |

> - 绝对装入只适用于单道程序环境
> - 静态重定位在**装入时**完成，动态重定位在**执行时**完成
> - 动态重定位需要**重定位寄存器**支持

##### 程序链接（三种方式）

| 链接方式 | 说明 | 特点 |
|---------|------|------|
| **静态链接** | 运行前将各目标模块及库函数链接成完整装入模块 | 链接后不再拆开 |
| **装入时动态链接** | 装入内存时边装入边链接 | 便于修改和更新 |
| **运行时动态链接** | 执行中需要该模块时才链接 | 便于共享（如共享库） |

##### 内存保护

- **上下限寄存器法**：比较地址是否在 $[\text{下限}, \text{上限}]$ 范围内
- **基址 + 限长寄存器法**：比较地址是否 $\lt$ 限长，$\text{物理地址} = \text{基址} + \text{逻辑地址}$

---

#### 3.1.2 覆盖与交换技术

> **覆盖（Overlay）**和**交换（Swapping）**是早期用来扩充内存的两种技术，为虚拟内存的出现奠定了基础。

##### 覆盖技术

- **原理**：将程序划分为常驻内存的**核心部分**和若干**覆盖段**，覆盖段按调用关系分时占用同一块内存区域
- **特点**：由程序员手动划分，对程序员不透明；各覆盖段之间不能有调用关系；只能用于单道程序环境
- **优点**：可在较小物理内存运行较大程序
- **缺点**：编程复杂，增加程序员负担

##### 交换技术

- **原理**：将暂时不能运行的进程换出到外存（对换区），把具备运行条件的进程换入内存
- **对换区**：在磁盘上开辟的专门区域，用于存放换出的进程，采用连续分配方式，I/O 速度快
- **交换时机**：通常在进程时间片用完或等待 I/O 时换出
- **特点**：由操作系统自动完成，对程序员透明；支持多道程序系统

> 覆盖与交换的核心区别：覆盖是**进程内部**的模块覆盖，交换是**进程之间**的换入换出；覆盖由程序员实现，交换由操作系统实现。

---

#### 3.1.3 连续分配管理方式

> 连续分配：为用户进程分配的必须是**一个连续的内存空间**。

##### 单一连续分配

内存分为**系统区**（低地址，存 OS 数据）和**用户区**（存用户进程）。内存中只能有**一道用户程序**，独占整个用户区。

| 优点 | 缺点 |
|------|------|
| 实现简单，无外部碎片 | 只能用于单用户、单任务 OS |
| 可采用覆盖技术扩充内存 | 有**内部碎片**，存储器利用率极低 |
| 不一定需要内存保护 | 不能用于多道程序环境 |

##### 固定分区分配

将用户空间划分为**若干个固定大小的分区**，每个分区装入一道作业。可划分为**大小相等**（灵活性差）或**大小不等**（多小分区 + 适量中等 + 少量大分区）。

操作系统维护**分区说明表**，每表项含：分区大小、起始地址、状态（已分配/未分配）。

| 优点 | 缺点 |
|------|------|
| 实现简单，无外部碎片 | 程序过大时所有分区均不满足 |
| 支持多道程序 | 有**内部碎片**，内存利用率低 |

##### 动态分区分配（可变分区分配）

> **不预先划分**内存分区，进程装入时**按其大小动态建立分区**。系统分区的大小和数目**可变**。

需解决三个问题：(1) 用什么数据结构记录内存使用；(2) 选哪个空闲分区分配；(3) 如何分配与回收。

**数据结构：**

- **空闲分区表**：每表项含分区号、大小、起始地址、状态
- **空闲分区链**：每个分区起始/末尾设前后向指针，起始处记录分区大小

**动态分区分配算法：**

| 算法 | 策略 | 分区排序 | 优点 | 缺点 |
|------|------|---------|------|------|
| **首次适应（First Fit）** | 从低地址找第一个满足的分区 | 地址递增 | 实现简单，保留高地址大分区 | 低地址易产生小碎片 |
| **最佳适应（Best Fit）** | 找最小的能满足的分区 | 容量递增 | 保留大分区 | 产生很多小碎片（外部碎片最多） |
| **最坏适应（Worst Fit）** | 找最大的能满足的分区 | 容量递减 | 分配后剩余空闲区大 | 大分区被迅速用完 |
| **邻近适应（Next Fit）** | 从上次结束位置开始找 | 地址递增（循环链表） | 查找开销均匀 | 高地址大分区更可能被使用 |

> 综合来看，**首次适应算法**的效果反而更好——低地址小分区被优先使用，高地址大分区得以保留。

记忆口诀：

```
首次适应：地址递增，从头找
最佳适应：容量递增，找最小
最坏适应：容量递减，找最大
邻近适应：地址递增，从上次找
```

**分区回收——四种相邻情况：**

| 回收区与空闲分区相邻情况 | 处理方式 |
|-------------------------|---------|
| 后面有相邻空闲分区 | 合并回收区与后面空闲分区 |
| 前面有相邻空闲分区 | 合并回收区与前面空闲分区 |
| 前后都有相邻空闲分区 | 三者合并为一个 |
| 前后都没有相邻空闲分区 | 新建表项 |

##### 内部碎片 vs 外部碎片

- **内部碎片**：分配给进程的内存中没用上的部分（固定分区、单一连续分配）
- **外部碎片**：内存中空闲但太小而难以利用的小分区（动态分区分配）

##### 紧凑/拼接技术

> **紧凑（Compaction）**：将内存中进程移动到一起，合并空闲分区，从而解决外部碎片问题。

> - 内部碎片是"分配了但没用上"，外部碎片是"空闲但太小无法用"
> - 固定分区：有内部碎片，无外部碎片
> - 动态分区：无内部碎片，有外部碎片
> - 最佳适应产生**最多外部碎片**

#### 3.1.4 非连续分配管理方式

##### 基本分页存储管理

> **分页（Paging）**：将进程的逻辑地址空间和内存物理空间都划分为**固定大小**的块，通过页表实现地址映射。属于非连续分配方式，无外部碎片。

| 术语 | 说明 |
|------|------|
| **页面/页（Page）** | 进程逻辑地址空间的等大固定块 |
| **页框/页帧（Frame）** | 内存物理空间的等大固定块（与页面同大小） |
| **页表（Page Table）** | 记录每个页面 → 页框号映射 |
| **页内偏移（Offset）** | 地址在页面内的偏移量 |
| **页表寄存器（PTR）** | 存于 PCB 中，指向当前进程页表起始地址 |

逻辑地址结构：

```
|   页号P   |   页内偏移W   |
```

**地址变换公式（页面大小 $L$，逻辑地址 $A$）：**

$$
P = \left\lfloor \frac{A}{L} \right\rfloor \quad \text{(整除取页号)}
$$

$$
W = A \bmod L \quad \text{(取余得偏移)}
$$

$$
\text{物理地址} = F \times L + W \quad \text{(F 为查页表所得页框号)}
$$

> 页面大小 $L = 4\text{KB} = 4096\text{B}$，逻辑地址 $A = 8196$：
>
> $$
> P = 8196 \div 4096 = 2, \quad W = 8196 \bmod 4096 = 4
> $$
>
> 查页表：第 2 页 → 页框 5 → $\text{物理地址} = 5 \times 4096 + 4 = 20484$

**访问内存次数：**

- 无快表：**2 次**（查页表 + 访问数据）
- 有快表命中：**1 次**

> 页面大小 $= 2^n$ 时，页号是逻辑地址高位，偏移是低 $n$ 位。如页面大小 4KB（$=2^{12}$），则逻辑地址低 12 位是偏移，高位是页号。

页表项结构：

```
页表项 = 页框号 + 有效位 + 修改位 + 访问位
```

基本页表示例：

| 页号 | 页框号 |
|------|--------|
| 0 | 5 |
| 1 | 2 |
| 2 | 7 |
| 3 | 1 |

**页面大小选择的权衡：**

- 太小 → 页表过大
- 太大 → 内部碎片多
- 常见：**4KB（$2^{12}$）**

##### 快表（TLB）

> **TLB（Translation Lookaside Buffer）**是页表的高速缓存（Cache），利用程序的局部性原理：时间局部性（刚访问的页可能再次访问）+ 空间局部性（相邻页可能被访问）。

**带 TLB 的地址转换过程：**

1. CPU 给出逻辑地址，用页号 P 查 TLB
2. **TLB 命中（Hit）**：直接得页框号 → 计算物理地址
3. **TLB 未命中（Miss）**：查内存页表 → 得页框号 → 将该表项加入 TLB → 计算物理地址

**有效访问时间（EAT）计算：**

设 TLB 访问时间 $= t$，内存访问时间 $= m$，TLB 命中率 $= a$：

$$
EAT = a \times (t + m) + (1-a) \times (t + 2m)
$$

$$
EAT = t + m + (1-a) \times m
$$

> 页面大小 4KB，TLB 访问时间 10ns，内存访问时间 100ns，TLB 命中率 80%。求 EAT。
>
> $$EAT = 0.8 \times (10 + 100) + 0.2 \times (10 + 200) = 88 + 42 = 130\text{ns}$$
>
> TLB 命中只需 **1 次访存**，未命中需 **2 次**（页表 + 数据）。

##### 二级/多级页表

> 当单级页表占用过多连续内存时，采用多级页表将页表分页存放，未使用的页表部分不调入内存。

**为什么需要：**

- 32 位系统，4GB 虚拟地址，4KB 页 → $2^{20} = 1\text{M}$ 个页表项
- 每个页表项 4B → 页表占 4MB
- 单级页表占用过多连续内存

两级页表地址结构：

```
| 一级页号 | 二级页号 | 页内偏移 |
```

**地址变换过程：**

1. 用一级页号查**页目录（一级页表）** → 得二级页表基址
2. 用二级页号查**二级页表** → 得页框号
3. $\text{物理地址} = \text{页框号} \times \text{页面大小} + \text{页内偏移}$

**优点：** 未使用的二级页表不调入内存，节省空间；适合大地址空间；64 位系统可扩展到多级（通常 4~5 级）。

**多级页表的访存次数：** 无 TLB 时，$n$ 级页表需 $n + 1$ 次访存（$n$ 次查页表 + 1 次访问数据）。

---

##### 反置页表（Inverted Page Table）

> **反置页表**：全局一张页表，按**物理页框号**索引，每个表项记录（进程 ID，页号），表示该页框被哪个进程的哪一页占用。与传统的按进程逻辑页号索引的页表方向相反。

**特点：**

- 页表大小与物理内存成正比，而非与逻辑地址空间成正比，大幅节省页表空间
- 地址转换时需按内容查找（按进程 ID + 页号搜索），需借助哈希表加速
- 不利于实现页面共享（每个页框只能对应一个进程的一个页面）

---

##### 基本分段存储管理

> **分段（Segmentation）**：按程序的**逻辑结构**（如代码段、数据段、栈段）划分为段，每段大小不固定，由用户程序决定。

逻辑地址结构：

```
|   段号S   |   段内偏移W   |
```

段表项：

```
段表项 = 段基址 + 段长度 + 保护位
```

**地址变换：**

1. 用段号 S 查段表 → 得段基址和段长度
2. 检查段内偏移是否越界（$W \lt \text{段长度}$）
3. $\text{物理地址} = \text{段基址} + \text{段内偏移}$

> 段表：段 0（基址 200H，长度 100H），段 1（基址 400H，长度 200H），段 2（基址 600H，长度 150H）
> 逻辑地址 (1, 0A0H) → $0A0H \lt 200H$（未越界）→ 物理地址 $= 400H + 0A0H = 4A0H$

##### 分页 vs 分段对比

| 比较项 | 分页 | 分段 |
|--------|------|------|
| 目的 | 提高内存利用率 | 满足用户逻辑需求 |
| 大小 | 固定（由硬件决定） | 不固定（由用户程序决定） |
| 地址空间 | **一维**（连续逻辑地址） | **二维**（段号 + 段内偏移） |
| 碎片 | 内部碎片 | 外部碎片 |
| 共享/保护 | 不方便 | 方便（按段共享/保护） |

> 分段是逻辑划分（二维），分页是物理划分（一维），两者目的和效果不同。

**分段优缺点：**

- 优点：无内部碎片，便于共享和保护
- 缺点：有外部碎片，地址变换复杂

---

##### 段页式存储管理

> **段页式**：分段和分页的结合——先按逻辑分段，再将每段分页。综合分段（逻辑清晰、方便共享保护）和分页（无外部碎片）的优点。

逻辑地址结构：

```
|   段号S   |   页号P   |   页内偏移W   |
```

**地址变换过程（需 3 次访存）：**

1. 用段号 S 查**段表** → 得该段的页表起始地址
2. 用页号 P 查**页表** → 得页框号 F
3. $\text{物理地址} = F \times \text{页面大小} + W$

> 段页式需 **3 次访存**（段表 + 页表 + 数据），可用 TLB 加速。

**段页式优缺点：**

- 优点：结合分段和分页的优点，无外部碎片
- 缺点：地址变换开销大（需 3 次访存）

**三种非连续分配方式访存次数对比：**

| 管理方式 | 无 TLB 访存次数 |
|---------|---------------|
| 基本分页 | 2 次（页表 + 数据） |
| 基本分段 | 2 次（段表 + 数据） |
| 段页式 | 3 次（段表 + 页表 + 数据） |

### 3.2 虚拟内存管理

#### 3.2.1 虚拟内存基本概念

> **虚拟内存**：将主存和辅存统一管理，具有**请求调入**和**置换功能**，能从逻辑上扩充内存容量，使进程看到比实际物理内存大得多的地址空间。

**核心思想：** 只将进程当前需要的部分调入内存，暂时不用的保留在外存，需要时再调入。

##### 局部性原理

- **时间局部性**：刚被访问的单元（指令/数据）很快可能再次被访问（如循环、子程序）
- **空间局部性**：刚被访问单元附近的单元可能很快被访问（如顺序执行、数组遍历）

##### 虚拟内存的特征

| 特征 | 说明 |
|------|------|
| **多次性** | 作业可分多次调入内存，非一次性全部装入 |
| **对换性** | 作业在运行期间可换入换出 |
| **虚拟性** | 逻辑容量大于实际物理容量 |

##### 虚拟内存的实现方式

| 方式 | 说明 |
|------|------|
| **请求分页** | 基于分页，最常用 |
| **请求分段** | 基于分段 |
| **请求段页式** | 基于段页式 |

---

#### 3.2.2 请求分页管理方式

##### 页表机制

在基本分页页表基础上增加字段：

| 字段 | 说明 |
|------|------|
| **状态位（存在位 P）** | 该页是否已在内存 |
| **访问字段（A）** | 记录该页被访问次数/时间 |
| **修改位（M）** | 该页在内存中是否被修改过（淘汰时决定是否写回） |
| **外存地址** | 该页在外存中的存放位置 |

##### 缺页中断处理过程

> **缺页中断与一般中断的区别**：缺页中断在**指令执行期间**产生（属于内部异常），一般中断在**指令之间**产生。缺页中断处理完毕后需**重新执行被中断的指令**。

```c
// 缺页中断处理流程伪代码
void page_fault_handler(int page_no) {
    保护CPU现场();
    分析中断原因();  // 缺页

    查找页表，定位该页在外存的位置();

    if (内存有空闲页框) {
        启动I/O，从外存读入该页();
        修改页表();
    } else {
        执行页面置换算法，选择淘汰页();
        if (淘汰页的修改位 == 1) {
            将淘汰页写回外存();
        }
        启动I/O，将新页读入();
        修改页表();
    }

    重新执行被中断的指令();
}
```

##### 地址变换完整流程（带 TLB 的请求分页）

```
1. CPU 给出逻辑地址(P, W)
2. 用页号 P 查 TLB
   ├─ 命中 → 得页框号 F，检查访问权限 → 计算物理地址
   └─ 未命中 → 查页表
               ├─ 页在内存（状态位=1）→ 得页框号 F，更新 TLB
               └─ 页不在内存（状态位=0）→ 产生缺页中断
                                           → 从外存调入
                                           → 修改页表
                                           → 更新 TLB
                                           → 重新执行被中断指令
3. 物理地址 = F × 页面大小 + W
```

**有效访问时间（考虑缺页率）：**

设缺页率 $= f$，内存访问时间 $= m$，缺页处理时间 $= s$（含 I/O）：

$$
EAT = (1-f) \times m + f \times s
$$

#### 3.2.3 页面置换算法

> 当内存已满、需调入新页面时，用置换算法选择被淘汰页面的策略。目标是**减少缺页中断次数**。

**性能指标：** 缺页率 $=$ 缺页次数 $\div$ 总访问次数

##### 最佳置换算法 OPT（Optimal）

- **原理**：淘汰**未来最长时间不再被访问**的页面（理论最优）
- **优点**：缺页率最低
- **缺点**：**不可实现**（无法预知未来访问序列）
- **用途**：作为评价其他算法的基准

> 引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1，分配 3 个物理块
>
> 缺页次数 = **9 次**

##### 先进先出 FIFO（First-In-First-Out）

- **原理**：淘汰**最先进入内存**的页面（队列实现，$O(1)$）
- **优点**：实现简单
- **缺点**：性能差，可能淘汰频繁使用的热点页面

##### Belady 异常

> **现象**：FIFO 算法中，增加物理块数，缺页次数**反而增加**。
>
> 例：引用串 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5
> - 3 个物理块：缺页 **9 次**
> - 4 个物理块：缺页 **10 次**
>
> **原因**：FIFO 不是栈式算法。LRU/OPT 是栈式算法，不会产生 Belady 异常。

> 引用串：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5，分配 3 个物理块
>
> 缺页次数 = **9 次**（若 4 个物理块则缺页 **10 次**，Belady 异常）

##### 最近最久未使用 LRU（Least Recently Used）

- **原理**：淘汰**最长时间未被访问**的页面，性能接近 OPT
- **优点**：性能好，不会产生 Belady 异常（栈式算法）
- **缺点**：硬件实现复杂，开销大

**LRU 的实现方式：**

- **栈实现**：访问页移到栈顶，淘汰栈底，时间复杂度 $O(n)$
- **计数器实现**：每页记录最后访问时间，淘汰计数器值最小的

> 引用串：7, 0, 1, 2, 0, 3, 0, 4, 2, 3, 0, 3, 2, 1, 2, 0, 1, 7, 0, 1，分配 3 个物理块
>
> 缺页次数 = **12 次**

> 引用串：1, 3, 0, 3, 5, 6, 3，分配 3 个物理块
>
> 缺页次数 = **5 次**（缺页序列：1, 3, 0, 5, 6）

##### 时钟算法 CLOCK / 改进型 CLOCK

###### 基本 CLOCK（二次机会算法）

- **原理**：循环链表扫描，每页面有访问位 A：
  - $A = 0$：淘汰该页面
  - $A = 1$：置 0，指针前移，继续检查（给"二次机会"）
- **优点**：近似 LRU，开销小（$O(1)$），实现简单

###### 改进型 CLOCK

在基本 CLOCK 基础上增加**修改位 M**：

| 访问位 R | 修改位 M | 状态 | 淘汰优先级 |
|:-------:|:-------:|------|:---------:|
| 0 | 0 | 最近未访问，未修改 | **最佳淘汰（第一优先级）** |
| 0 | 1 | 最近未访问，已修改（需写回外存） | 第二优先级 |
| 1 | 0 | 最近访问，未修改 | 第三优先级 |
| 1 | 1 | 最近访问，已修改 | 最后才淘汰 |

**淘汰过程（最多 4 轮扫描）：**

1. **第一轮**：找 $(0, 0)$，直接淘汰（不修改 R 位）
2. **第二轮**：找 $(0, 0)$，将经过的 $R = 1$ 置 0（不修改 M 位）
3. **第三轮**：找 $(0, 0)$，此轮 M = 1 需写回（不修改 R 位）
4. **第四轮**：找 $(0, 0)$，将经过的 $R = 1$ 置 0

> 改进型 CLOCK 的核心逻辑：优先淘汰"既未被访问又未修改"的页面，因为淘汰它们不需要写回外存，代价最小。

##### 各算法对比

| 算法 | 最优性 | 实现难度 | 复杂度 | Belady 异常 |
|------|--------|----------|--------|------------|
| **OPT** | 最优（理论上界） | 不可实现 | — | 无 |
| **FIFO** | 差 | 简单 | $O(1)$ | **有** |
| **LRU** | 好（接近 OPT） | 较复杂 | $O(1) \sim O(n)$ | 无 |
| **CLOCK** | 较好 | 中等 | $O(1)$ | 无 |
| **改进型 CLOCK** | 较好 | 中等 | $O(1)$ | 无 |

#### 3.2.4 页面分配策略与抖动

##### 分配策略 vs 置换策略的四种组合

| 策略组合 | 说明 | 特点 |
|---------|------|------|
| **固定分配 + 局部置换** | 每进程分配固定页框数，缺页时只淘汰自己的页 | 实现简单，公平，可能浪费 |
| **可变分配 + 全局置换** | 页框数动态变化，缺页时可淘汰其他进程的页 | 性能更好，但可能影响其他进程 |
| **可变分配 + 局部置换** | 页框数动态调整，但只能淘汰自己的页 | 平衡方案 |
| **固定分配 + 全局置换** | 不可能（固定分配下无法全局置换） | — |

**固定分配 vs 可变分配：**

- **固定分配**：进程创建时确定物理块数，运行期间不变。简单但不够灵活。
- **可变分配**：根据进程需求动态调整物理块数。更灵活，复杂度高。

**局部置换 vs 全局置换：**

- **局部置换**：缺页时只能从进程自己的页面中选择淘汰，不影响其他进程，公平性好
- **全局置换**：可从所有进程的页面中选择淘汰（包括其他进程），性能更好但可能影响其他进程

##### 工作集模型

> **工作集（Working Set）**：进程在某段时间间隔内实际访问的页面集合。
>
> 给定窗口大小 $\Delta$：$W(t, \Delta) = $ 在 $[t-\Delta+1, t]$ 时间内访问的页面集合

**应用：**

- 根据工作集大小分配物理块
- 防止抖动
- 操作系统决定调页策略

> 引用串：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5，窗口 $\Delta = 5$
>
> | 时刻 t | 访问 | 工作集 $W(t, 5)$ | 大小 |
> |--------|------|-------------------|------|
> | 5 | 1 | {1, 2, 3, 4} | 4 |
> | 7 | 5 | {1, 2, 3, 4, 5} | 5 |
> | 10 | 3 | {1, 2, 3, 5} | 4 |

##### 抖动现象及预防

> **抖动（Thrashing）**：进程频繁发生缺页中断，刚换出的页面又被换入，导致系统效率急剧下降（大部分时间用于处理缺页而非真正执行）。

**原因：**

- 分配给进程的物理块数不足
- 并发进程数过多
- 页面置换算法不当

**解决方案：**

1. **增加物理块数**（根本措施）
2. **减少并发进程数**（降低多道程序度）
3. **使用局部置换**策略
4. **采用工作集模型**，按工作集分配页框
5. **预调页策略**：将预计即将使用的页提前调入

> 抖动与 Belady 异常的区别：Belady 异常是 FIFO 算法特有的"增加物理块反而缺页增多"现象；抖动是系统级的"频繁缺页导致效率崩溃"现象，不仅限于 FIFO。

### 3.3 典型例题精选

> 下列关于页面大小的说法，正确的是（ ）
>
> A. 页面越大，页表越小，内部碎片越少
> B. 页面越小，页表越大，内部碎片越少
> C. 页面越大，页表越小，内部碎片越多
> D. 页面越小，页表越大，内部碎片越多
>
> **答案：C**
>
> 页面越大 → 页数越少 → 页表越小 ✓；页面越大 → 每页内浪费可能越多 → 内部碎片越多 ✓

> 某系统页面大小 4KB，页表：页 0 → 框 2，页 1 → 框 4，页 2 → 框 6，页 3 → 框 8。逻辑地址为 0x2A4F，求物理地址。
>
> 页面大小 = 4KB = $2^{12}$B
>
> 页号 $P = \text{0x2A4F} \div \text{0x1000} = 2$
>
> 偏移 $W = \text{0x2A4F} \bmod \text{0x1000} = \text{0xA4F}$
>
> 查页表：第 2 页 → 页框号 6
>
> 物理地址 $= 6 \times 4096 + \text{0xA4F} = \text{0x6000} + \text{0xA4F} = \text{0x6A4F}$

> 页面访问序列：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5，分配 3 个页框。分别用 FIFO 和 LRU 计算缺页次数。
>
> **FIFO：** 缺页次数 = **9 次**（若 4 个物理块则缺页 10 次，出现 Belady 异常）
>
> **LRU：** 缺页次数 = **10 次**

> 段表：段 0（基址 200H，长 100H），段 1（基址 400H，长 200H），段 2（基址 600H，长 150H）
> 求逻辑地址 (1, 0A0H) 的物理地址。
>
> 段号 = 1，偏移 = 0A0H，$0A0H \lt 200H$ → 未越界
>
> 物理地址 = 400H + 0A0H = **4A0H**

> 某段页式系统中，段表如下：段 0 → 页表基址 1000H，段长度 200H；段 1 → 页表基址 2000H，段长度 300H。段 1 的页表：页 0 → 框 5，页 1 → 框 8。页面大小 4KB，求逻辑地址（段 1，页 1，偏移 100H）的物理地址。
>
> 段号 1，查段表 → 页表基址 2000H，段长度 300H，偏移 100H $\lt$ 300H（未越界）
>
> 页号 1，查页表 → 页框号 8
>
> 物理地址 $= 8 \times 4096 + 100H = 8000H + 100H = 8100H$

### 3.4 易错点汇总

1. ❌ "FIFO 一定比 LRU 差" → ✅ 不一定，特定引用串下 FIFO 可能更好（但总体 LRU 更优）
2. ❌ "增加物理块一定降低缺页率" → ✅ FIFO 有 Belady 异常，增加块可能增加缺页
3. ❌ "页面大小为 4KB，地址 0x1234 的页号是 1" → ✅ 页号 = 0x1234 / 0x1000 = 1，偏移 = 0x234，正确！
4. ❌ "分段和分页效果相同" → ✅ 分段是逻辑划分（二维），分页是物理划分（一维）
5. ❌ "TLB 未命中就一定缺页" → ✅ TLB 未命中只表示快表没有，页表中可能还有（还需查页表）
6. ❌ "缺页中断和一般中断相同" → ✅ 缺页中断在**指令执行中**产生（内部异常），一般中断在**指令间**产生；缺页中断处理完后需**重新执行被中断指令**
7. ❌ "绝对装入适用于多道程序" → ✅ 绝对装入只适用于**单道程序环境**
8. ❌ "静态重定位在执行时完成" → ✅ 静态重定位在**装入时**完成，动态重定位在**执行时**完成
9. ❌ "最佳适应算法不产生外部碎片" → ✅ 最佳适应**产生最多外部碎片**
10. ❌ "动态分区分配无内部碎片" → ✅ 正确！动态分区按需分配，分区大小等于进程大小，无内部碎片（但若配合分页，最后一页会有少量内部碎片）
11. ❌ "段页式只需 2 次访存" → ✅ 段页式需 **3 次访存**（段表 + 页表 + 数据）
12. ❌ "多级页表不增加访存次数" → ✅ $n$ 级页表无 TLB 时需 $n + 1$ 次访存

### 4.1 文件系统基础

#### 4.1.1 文件的概念

> **文件（File）**：具有**文件名**的一组**相关信息**的集合，是操作系统对资源管理的抽象，也是信息存储和管理的基本单位。

**文件的属性（元数据）**：

| 属性 | 描述 |
|:-----|:-----|
| **文件名** | 用户使用的文件标识，同一目录下不允许重名 |
| **标识符** | 系统内部唯一标识，对用户无可读性 |
| **文件类型** | 扩展名指示（.txt、.doc 等） |
| **文件大小** | 字节数 |
| **物理位置** | 外存中的地址（对用户不可见） |
| **创建/修改/访问时间** | 时间戳 |
| **所有者** | 文件创建者信息 |
| **访问权限** | R/W/X 等保护信息 |

> **文件名 vs 标识符**：文件名是给用户看的（同一目录下不可重名），标识符是操作系统内部使用的（全系统唯一）。两者不是同一个概念。

**文件的分类**：

| 分类维度 | 类型 |
|:---------|:-----|
| **按用途** | 系统文件、用户文件、库文件 |
| **按保护** | 只读文件、读写文件、可执行文件 |
| **按性质** | 文本文件（ASCII/字符流）、二进制文件 |
| **按存取方式** | 顺序文件、随机文件、索引文件 |
| **按组织形式** | 普通文件、目录文件、特殊文件 |
| **按数据形式** | 源文件、目标文件、可执行文件 |

**文件的基本操作**：

| 操作 | 系统调用 | 关键处理 |
|:-----|:---------|:---------|
| **创建（create）** | 分配外存空间 + 在目录中创建目录项 | 需提供路径、文件名、所需空间大小 |
| **删除（delete）** | 回收磁盘块 + 删除目录项 | 根据目录项记录的位置和大小回收空间 |
| **打开（open）** | 将目录项复制到"打开文件表" | 需提供路径、文件名、操作类型（r/rw），返回文件描述符 |
| **关闭（close）** | 从打开文件表中删除表项 | 系统表 count 减 1，count=0 时删除系统表项 |
| **读（read）** | 根据文件描述符将数据从外存读入内存 | 需提供 fd、数据量、内存缓冲区地址 |
| **写（write）** | 将数据从内存缓冲区写入外存 | 需提供 fd、数据量、内存缓冲区地址 |
| **seek** | 移动读写指针位置 | 支持随机存取定位 |
| **rename** | 重命名文件 | 修改目录项中的文件名 |

> - **进程级打开文件表**：每个进程一张，记录该进程打开的文件
> - **系统级打开文件表**：整个系统一张，记录所有进程打开的文件
> - **作用**：避免每次都查目录，加快文件访问速度；方便实现文件共享和保护

> 读/写文件前**必须先 open**，读/写完后**必须 close**。复制文件 = create 新文件 + read 源文件 + write 新文件。

#### 4.1.2 文件的逻辑结构

> **文件的逻辑结构**：用户视角下，文件内部数据应该如何组织。与之对应的是**物理结构**（操作系统视角，文件在外存上的存放方式）。类比数据结构中"线性表（逻辑结构）"与"顺序表/链表（物理结构）"的关系。

**无结构文件（流式文件）**：

- 由一系列二进制流或字符流组成（如 Windows 的 .txt 文件）
- 没有明显的结构特性，Unix/Linux 系统将文件视为字节流
- **不探讨逻辑结构问题**

**有结构文件（记录式文件）**：

由一组相似记录组成，每条记录由若干数据项组成，通常有一个数据项作为**关键字**（识别不同记录的 ID）。

- **定长记录**：每条记录长度相同，各数据项位置、顺序、长度固定
- **可变长记录**：各条记录长度不确定

**四种逻辑结构对比**：

| 类型 | 组织方式 | 查找方式 | 查找效率 | 特点 |
|:-----|:---------|:---------|:---------|:-----|
| **顺序文件** | 记录按关键字顺序排列 | 顺序查找/折半查找 | $O(n)$ | 定长+顺序存储时可随机存取；增删困难 |
| **索引文件** | 为每条记录建立索引表 | 通过索引快速查找/折半查找 | $O(\log n)$ | 检索速度快，索引表占用额外空间；可建立多个索引表 |
| **索引顺序文件** | 记录分组（$\sqrt{n}$ 组），组间有序、组内可无序 | 索引定位组 → 组内顺序查 | $O(\sqrt{n})$ | 兼顾效率与空间；每组第一条记录对应一个索引项 |
| **散列文件** | 用散列函数计算记录地址 | 直接计算 | $O(1)$ | 有冲突问题，适用于关键字分布均匀的场景 |

> - **顺序文件 vs 索引顺序文件**：顺序文件所有记录按关键字排序；索引顺序文件记录分组，组内可以无序，组间必须有序。
> - **索引文件 vs 索引顺序文件**：索引文件中**每条记录**有一个索引项；索引顺序文件中**每组记录**有一个索引项。
> - **无结构文件**没有记录的概念，不需要探讨逻辑结构。

> 下列关于文件逻辑结构的叙述中，正确的是（**C**）
> A. 顺序文件的记录之间没有顺序关系
> B. 索引文件的记录之间有顺序关系
> C. 索引顺序文件的记录之间有顺序关系
> D. 无结构文件的记录之间有顺序关系
>
> **解析**：顺序文件的记录之间有顺序关系（A 错）；索引文件的记录之间不一定有顺序关系（B 错）；索引顺序文件的组间有顺序关系（C 对）；无结构文件没有记录概念（D 错）。

#### 4.1.3 目录结构

**FCB（文件控制块）与索引节点（inode）**：

> **FCB（File Control Block）**：文件系统为每个文件建立的数据结构，存放文件的元信息，以实现"按名存取"。FCB 的有序集合称为**文件目录**，一个 FCB 就是一个**目录项**。

**FCB 包含的信息**：
- **基本信息**：文件名、文件物理地址、逻辑结构、物理结构等
- **存取控制信息**：读写权限、禁止访问的用户名单等
- **使用信息**：建立时间、修改时间等

> **inode** 是 Unix/Linux 文件系统中用于存储文件元数据的数据结构。是一种**目录瘦身策略**：将 FCB 中除文件名外的所有信息放入索引节点，目录项中只保留**文件名 + 索引节点指针**。
>
> - **磁盘索引节点**：存放在磁盘上，含文件类型、权限、所有者、大小、时间戳、数据块指针等
> - **内存索引节点**：读入内存后，额外增加索引节点号、状态、访问计数等信息
>
> **使用 inode 的优点**：减小目录项体积，一个磁盘块可存放更多目录项，减少检索时的磁盘 I/O 次数。

> - `FCB = 目录项`。引入 inode 后，目录项 = 文件名 + inode 指针，其余信息在 inode 中。
> - **inode 中不包含文件名**，文件名只存在于目录项中。（2017 年 408 真题考点）

**四种目录结构**：

| 结构 | 特点 | 优点 | 缺点 |
|:-----|:-----|:-----|:-----|
| **单级目录** | 整个系统一张目录表 | 简单，实现"按名存取" | **不允许重名**，查找效率低 |
| **两级目录** | MFD（主文件目录）+ UFD（用户文件目录） | 不同用户文件可重名 | 用户不能对自己的文件分类 |
| **多级（树形）目录** | 分层目录树，根 `/` | 层次清晰，便于分类管理 | 路径长，不方便文件共享 |
| **无环图目录** | 树形基础上增加共享边 | 灵活，支持文件共享 | 复杂，需处理共享文件删除问题 |

> - **绝对路径**：从根目录 `/` 出发（如 `/home/user/file.txt`）
> - **相对路径**：从当前目录出发（如 `./file.txt` 或 `../user/file.txt`）

**目录查询技术**：

| 技术 | 原理 | 特点 |
|:-----|:-----|:-----|
| **线性列表法** | 目录中顺序存放目录项，查找时线性扫描 | 实现简单，但查找效率低 $O(n)$ |
| **哈希表法** | 根据文件名哈希值定位目录项 | 查找效率高 $O(1)$，但需处理冲突 |

#### 4.1.4 文件共享

> **文件共享**：多个用户共同使用同一个文件。系统中只有**一份**文件数据，任一用户修改后其他用户可见变化（与文件复制的"多份数据"有本质区别）。

**硬链接（基于索引节点）**：

- 多个目录项指向**同一个索引节点（inode）**
- inode 中维护**链接计数 count**，表示指向该 inode 的目录项数
- 某用户"删除"文件时：仅删除该用户目录中的目录项，count 减 1
- **count > 0 时文件不删除**，count = 0 时系统才真正删除文件数据
- 所有共享用户看到的是同一个文件，权限相同

**软链接（基于符号链）**：

- 创建一个新的 **Link 类型文件**，其中存储被共享文件的**路径**（类似 Windows 快捷方式）
- Link 类型文件有自己的 inode
- 访问时：OS 判断为 Link 类型 → 根据记录的路径层层查找目录 → 定位到目标文件
- 原文件删除后，Link 文件仍在但**路径失效**（悬空链接）

**硬链接 vs 软链接对比**：

| 特性 | 硬链接 | 软链接（符号链接） |
|:-----|:-------|:-------------------|
| **实质** | 多个目录项指向同一 inode | 新文件存储目标文件路径 |
| **是否共享 inode** | ✅ 是 | ❌ 否（各自独立 inode） |
| **删除源文件** | count 减 1，文件不删除 | 链接失效（悬空） |
| **跨文件系统** | ❌ 不可以 | ✅ 可以 |
| **链接目录** | ❌ 不可以（防止环路） | ✅ 可以 |
| **访问速度** | 快（直接访问 inode） | 慢（需路径查找） |
| **额外存储** | 不占用额外空间 | 占用额外空间（Link 文件） |

> - 硬链接删除源文件后，只要还有**至少一个**硬链接存在，文件数据就还在。
> - 软链接删除源文件后，链接**失效**（成为悬空链接）。（2021 年 408 真题考点）

#### 4.1.5 文件保护

> **文件保护**：防止文件被非法访问、修改或删除。

**三种保护方式**：

| 保护方式 | 原理 | 安全性 | 开销 |
|:---------|:-----|:-------|:-----|
| **口令保护** | 为文件设置口令，访问时验证（口令存储在 FCB/inode 中） | 低（口令明文存于系统内部） | 低 |
| **加密保护** | 用密码对文件加密，访问时解密（密码不存于系统内部） | 高 | 高（加密/解密耗时） |
| **访问控制表（ACL）** | 每个文件维护一张表，记录各用户对该文件的访问权限 | 高 | 中（表占存储空间） |

**访问控制**：

- **访问类型**：读（R）、写（W）、执行（X）
- **访问控制表（ACL）**：每个文件存可访问用户及其权限
- **用户权限表**：每个用户存可访问文件列表
- **访问控制矩阵**：用户 × 文件的权限矩阵

**UNIX 文件权限位**：

```
-rwxr-xr--
│ │││ │││ │││
│ │││ │││ └┴┴─ 其他人(others)：r--
│ │││ │└┴─────── 组(group)：r-x
│ │││ └┴───────── 所有者(owner)：rwx
│ ││└──────────── 文件类型（-普通/d目录/l链接）
```

> - **口令 vs 密码**：口令存于系统内部（安全性低）；密码不存于系统内部（安全性高）。
> - ACL 可以为不同用户设置不同权限，是权限粒度最细的方案。

### 4.2 文件系统实现

#### 4.2.1 文件系统层次结构

```
用户接口（文件操作：open/close/read/write）
    ↓
文件目录系统（目录管理、按名存取）
    ↓
存取控制验证（权限检查、ACL）
    ↓
逻辑文件系统（逻辑结构 → 物理结构映射）
    ↓
物理文件系统（物理块分配与回收）
    ↓
设备管理程序（I/O 调度与操作）
    ↓
辅助存储设备（磁盘/SSD/磁带）
```

> ```
> ┌──────────┬──────────┬────────────┬────────────┐
> │ 引导块    │ 超级块    │ 索引节点区  │ 数据块区    │
> │(Boot)    │(Super)   │(inode 区)  │(Data)      │
> └──────────┴──────────┴────────────┴────────────┘
> ```
> - **引导块**：存放引导程序，用于系统启动
> - **超级块**：文件系统元数据（块大小、总块数、空闲块信息等）
> - **索引节点区**：存放所有 inode
> - **数据块区**：存放文件实际数据和目录数据

#### 4.2.2 目录实现

| 实现方式 | 原理 | 优点 | 缺点 |
|:---------|:-----|:-----|:-----|
| **线性列表** | 目录项顺序存放，线性扫描查找 | 实现简单 | 查找效率低 $O(n)$ |
| **哈希表** | 根据文件名哈希值直接定位 | 查找效率高 $O(1)$ | 需处理冲突，大小固定时需扩容 |

#### 4.2.3 文件实现——文件的物理结构（分配方式）

> **文件的物理结构（文件分配方式）**：文件数据在磁盘上的存放方式。文件逻辑地址表示为 `(逻辑块号, 块内地址)`，OS 负责将其映射为物理地址 `(物理块号, 块内地址)`。

> 外存分为一个个等大小的"块/磁盘块/物理块"（如 1KB）；操作系统以**块**为单位分配空间和传输数据。即使文件只有 10B，也占用一个完整的磁盘块。

##### ① 连续分配

- 文件占用一段**连续**的磁盘块
- 目录项记录：**起始块号 + 长度（块数）**
- 地址映射：$物理块号 = 起始块号 + 逻辑块号$

| 优点 | 缺点 |
|:-----|:-----|
| 顺序访问极快 | **外部碎片**严重 |
| 随机访问快（直接计算） | 文件不易扩展（需整体搬移） |
| 可靠性高 | 存储空间利用率低 |

> **连续分配最支持随机访问**，不是"不支持"。知道起始块号即可直接计算任意逻辑块的物理地址。

##### ② 链接分配

**隐式链接**：

- 每个盘块末尾存指向下一盘块的**指针**
- 目录项记录：起始块号 + 末尾块号
- 访问第 $i$ 个逻辑块需 $i+1$ 次磁盘 I/O（顺藤摸瓜）

| 优点 | 缺点 |
|:-----|:-----|
| 无外部碎片，外存利用率高 | **只支持顺序访问**，不支持随机访问 |
| 文件扩展方便 | 指针占空间，可靠性差（指针丢失则链断裂） |

**显式链接（FAT）**：

- 将所有盘块链接指针集中存放在**文件分配表（FAT）**中
- 一个磁盘只有一张 FAT，常驻内存
- FAT 表项在物理上连续存储，每个表项长度相同

| 优点 | 缺点 |
|:-----|:-----|
| 无外部碎片，外存利用率高 | FAT 占用内存/磁盘空间 |
| 支持随机访问（FAT 在内存中） | FAT 表大小与磁盘容量成正比 |
| 逻辑块号 → 物理块号过程**不需要读磁盘** | |

> - 考试中若未指明隐式/显式，"链接分配"默认指**隐式链接**。
> - 显式链接（FAT）访问第 $i$ 个逻辑块**不需要读磁盘**（FAT 常驻内存），仅需从 FAT 中查找链。
> - 隐式链接中每个块的一部分空间被指针占用，实际数据容量略小。

##### ③ 索引分配

- 每个文件有一张**索引表**（存放在索引块中），记录各逻辑块 → 物理块的映射
- 目录项记录：索引块位置

| 优点 | 缺点 |
|:-----|:-----|
| 无外部碎片 | 索引块占用空间 |
| 支持随机访问 | 小文件效率不如连续分配 |
| 文件扩展方便 | 大文件需要多级索引 |

**大文件索引方案**：

- **链接方案**：多个索引块用指针链接 → 访问末尾逻辑块需顺序读前面索引块，效率低。
- **多层索引**：建立多级索引（类似多级页表）。若顶级索引表未调入内存，$K$ 层索引访问一个数据块需 $K+1$ 次访盘。
- **混合索引（Unix 风格）**——最经典的方案：

| 索引级别 | 指针数 | 寻址范围计算（块大小 1KB，块号 4B） |
|:---------|:-------|:-------------------------------------|
| 直接地址 | 10 个 | $10 \times 1\text{KB} = 10\text{KB}$ |
| 一级间接 | 1 个 | $(1\text{KB}/4\text{B}) \times 1\text{KB} = 256 \times 1\text{KB} = 256\text{KB}$ |
| 二级间接 | 1 个 | $256 \times 256 \times 1\text{KB} = 64\text{MB}$ |
| 三级间接 | 1 个 | $256 \times 256 \times 256 \times 1\text{KB} = 16\text{GB}$ |

**访盘次数**（顶级索引已读入内存情况下）：
- 直接地址：1 次（直接读数据块）
- 一级间接：2 次（索引块 + 数据块）
- 二级间接：3 次
- 三级间接：4 次

> - 小文件（< 10KB）：直接地址，1 次访盘，极快
> - 中等文件（< 266KB）：一次间接，够用
> - 大文件：二/三次间接，支持超大文件
> - **兼顾了小文件效率和超大文件支持能力**

##### ④ 三种分配方式总对比

| 比较项 | 连续分配 | 链接分配（隐式） | 链接分配（显式/FAT） | 索引分配 |
|:-------|:---------|:-----------------|:---------------------|:---------|
| **顺序访问** | ✅ 快 | ✅ 快 | ✅ 快 | ✅ 快 |
| **随机访问** | ✅ 快 | ❌ 不支持 | ✅ 快 | ✅ 中等 |
| **外部碎片** | ❌ 有 | ✅ 无 | ✅ 无 | ✅ 无 |
| **文件扩展** | ❌ 困难 | ✅ 容易 | ✅ 容易 | ✅ 容易 |
| **可靠性** | ✅ 高 | ❌ 低（指针丢失） | ✅ 高 | ✅ 高 |
| **额外开销** | 无 | 指针占磁盘空间 | FAT 占内存 | 索引块占空间 |

> 下列关于文件的叙述，**错误**的是（**B**）
> A. 文件的逻辑结构可分为有结构文件和无结构文件
> B. 连续分配适合顺序访问，不适合随机访问
> C. 链接分配不支持随机访问
> D. 索引分配支持随机访问
>
> **解析**：连续分配既适合顺序访问也适合随机访问（地址直接可计算）。

> 某文件系统盘块大小 1KB，盘块号占 4B，采用混合索引：直接地址 6 个、一级间接 1 个、二级间接 1 个。求：
> （1）文件最大长度？（2）访问第 500 个逻辑块需要几次访盘？
>
> **解**：
> （1）每索引块指针数 = 1KB / 4B = 256
>
> | 级别 | 范围 | 容量 |
> |------|------|------|
> | 直接地址 | 逻辑块 0~5 | 6 × 1KB = 6KB |
> | 一级间接 | 逻辑块 6~261 | 256 × 1KB = 256KB |
> | 二级间接 | 逻辑块 262~65797 | 256² × 1KB = 64MB |
>
> 最大文件 = 6KB + 256KB + 64MB ≈ **64.2MB**
>
> （2）第 500 块 > 261，在二级间接范围内 → 读二级索引块（1 次）+ 读一级索引块（1 次）+ 读数据块（1 次）= **共 3 次访盘**。

#### 4.2.4 文件存储空间管理

> 磁盘划分为若干文件卷（逻辑卷），每个卷划分为**目录区**（存放 FCB 和空间管理信息）和**文件区**（存放文件数据）。

**四种方法总对比**：

| 方法 | 原理 | 适用分配方式 | 适用系统规模 |
|:-----|:-----|:------------|:------------|
| **空闲表法** | 记录每个连续空闲区的起始块号 + 长度 | 连续分配 | 小型系统 |
| **空闲链表法** | 空闲块用指针链接成链 | 离散分配 | 小型系统 |
| **位示图法** | 每个盘块用 1 bit 表示（0=空闲/1=占用） | 连续/离散均可 | 各种系统 |
| **成组链接法** | 空闲块分组管理，组间链接（Unix） | 连续/离散均可 | 大型系统 |

##### ① 空闲表法

类似内存管理的动态分区分配：空闲表记录每个**连续空闲区**的起始块号和空闲块数。

- **分配算法**：首次适应、最佳适应、最坏适应
- **回收算法**：与动态分区回收类似，四种情况（前后均不邻接/均邻接/前邻接/后邻接），需注意表项合并

##### ② 空闲链表法

| 变体 | 原理 | 特点 |
|:-----|:-----|:-----|
| **空闲盘块链** | 每个空闲块存下一空闲块指针 | 分配 $K$ 个块需重复操作 $K$ 次，效率低 |
| **空闲盘区链** | 连续空闲块组成盘区，盘区首块存长度 + 下一盘区指针 | 分配多个盘块时效率更高，离散/连续分配均可 |

##### ③ 位示图法

- 每个二进制位对应一个盘块："0" = 空闲，"1" = 已分配（或相反）
- 位示图常用连续"字"（word）表示
- **位示图常驻内存**，分配和回收速度快

**盘块号 ↔ 字号位号转换公式**：

设字长 $n$ 位，盘块号、字号、位号均从 **0** 开始：

| 转换方向 | 公式 |
|:---------|:-----|
| 字号位号 → 盘块号 | $b = n \times i + j$ |
| 盘块号 → 字号位号 | $i = \left\lfloor \frac{b}{n} \right\rfloor,\quad j = b \bmod n$ |

> 注意题目中盘块号、字号、位号到底是从 **0** 还是 **1** 开始！从 1 开始则公式变为 $b = n \times (i-1) + j$ 等。

> 磁盘块 1KB，容量 1GB，字长 32，字号位号从 0 开始。盘块号 2000 对应：（**A**）
> A. 62, 16　B. 63, 16　C. 62, 17　D. 63, 17
>
> **解**：$i = 2000 / 32 = 62,\ j = 2000 \bmod 32 = 16$

##### ④ 成组链接法（Unix 采用）

将空闲盘块**分组**管理，结合了空闲表和链表的思想：

```
超级块（常驻内存）→ 记录第一组空闲块数 + 各块号
                       ↓（最后一组块号=0 表示结束）
                   下一组空闲盘块信息
                       ↓
                   再下一组...
```

- **超级块**：文件卷目录区中专用的一个磁盘块，记录当前可用分组信息
- **分配**：先检查超级块中第一组剩余块数 → 够则直接分配 → 不够则将下一组信息读入内存再分配
- **回收**：第一组未满则加入 → 已满则将当前超级块内容写入回收块，回收块成为新超级块
- **关键**：保证内存与外存中超级块数据一致

> 磁盘 500 盘块，FAT 表项占 2B。文件 F：FAT[100]=200, FAT[200]=350, FAT[350]=-1。求：
> （1）文件 F 占 **3 个盘块**（100→200→350→结束）
> （2）逻辑块 2 → 物理块 **350**
> （3）FAT 大小 = 500×2B = 1000B，每个盘块 1KB，需 ⌈1000/1024⌉ = **1 个盘块**

### 4.3 磁盘组织与管理

#### 4.3.1 磁盘结构

| 概念 | 说明 |
|:-----|:-----|
| **盘面（Platter）** | 磁盘的一个记录面，每个盘面有一个磁头 |
| **磁道（Track）** | 盘面上的同心圆环 |
| **扇区（Sector）** | 磁道上的一段弧，是磁盘读写的最小单位（通常 512B） |
| **柱面（Cylinder）** | 所有盘面中同一半径的磁道集合（柱面号 = 磁道号） |
| **磁盘块** | 操作系统定义的逻辑读写单位（通常 = 多个扇区，如 1KB=2 扇区） |

#### 4.3.2 磁盘调度算法

> $$T_{\text{访问}} = T_{\text{寻道}} + T_{\text{旋转延迟}} + T_{\text{传输}}$$
> - **$T_{\text{寻道}}$**（Seek Time）：磁头移动到目标磁道的时间，**占比最大**
> - **$T_{\text{旋转}}$**（Rotational Latency）：扇区旋转到磁头下的时间，平均 = 转半圈时间
> - **$T_{\text{传输}}$**（Transfer Time）：实际读写数据的时间，**占比最小**

**五大算法详解**：

| 算法 | 原理 | 寻道表现 | 公平性 | 饥饿问题 |
|:-----|:-----|:---------|:-------|:---------|
| **FCFS** | 按请求到达顺序 | 差 | ✅ 公平 | ❌ 无 |
| **SSTF** | 选距当前磁头**最近**的请求 | **最好** | 中 | ⚠️ **有**（远距离请求可能饥饿） |
| **SCAN（电梯）** | 磁头单向移动，沿途处理；到边界后**反向** | 好 | 中 | ⚠️ 两端请求等待久 |
| **C-SCAN** | 单向处理，到边界后**快速返回起点**（不处理） | 中 | **最好**（均匀响应） | ❌ 无 |
| **LOOK/C-LOOK** | SCAN/C-SCAN 的改进版：不到边界就反向 | 好 | 中 | ⚠️ |

> - **SCAN**：到磁盘最内/最外边界才反向
> - **C-SCAN**：单向到边界 → 直接跳回起点（不处理沿途请求），使得各磁道响应更均匀
> - **LOOK**：到最远请求处就反向（不到边界），更实用
> - **C-LOOK**：到最远请求处 → 跳回最近请求处（不到边界）

> 请求队列：98, 183, 37, 122, 14, 124, 65, 67；当前磁头 53。求 SSTF 和 SCAN 的总寻道长度。
>
> **SSTF**（每次选最近）：
> 53→65(12)→67(2)→37(30)→14(23)→98(84)→122(24)→124(2)→183(59)
> 总寻道 = 12+2+30+23+84+24+2+59 = **236**
>
> **SCAN**（假设向增大方向）：
> 53→65(12)→67(2)→98(31)→122(24)→124(2)→183(59)
> →反向→37(146)→14(23)
> 总寻道 = 12+2+31+24+2+59+146+23 = **299**
>
> SSTF 寻道更短，但可能饥饿；SCAN 更公平。

#### 4.3.3 磁盘读写时间计算

**完整公式**：

设磁盘转速为 $r$ 转/秒，则每转时间 = $1/r$ 秒：

$$T_{\text{读写}} = T_{\text{寻道}} + \frac{1}{2r} + \frac{b}{rN}$$

其中：
- $T_{\text{寻道}}$：寻道时间（ms 级）
- $\frac{1}{2r}$：平均旋转延迟 = 半圈时间
- $\frac{b}{rN}$：传输时间，$b$ 为读写字节数，$N$ 为每磁道字节数（$N/r$ 为每秒传输字节）

**简化公式**（已知每扇区参数）：

$$T_{\text{总}} = T_{\text{寻道}} + T_{\text{旋转延迟}} + \frac{\text{扇区数}}{\text{每磁道扇区数}} \times \frac{1}{\text{转速}}$$

> 磁盘转速 7200 RPM（转/分），寻道时间 8ms，每磁道 500 扇区，每扇区 512B。求读取 2500B 的访问时间。
>
> 转速 $r = 7200/60 = 120$ 转/秒，每转时间 = $1/120 \approx 8.33$ms
>
> $T_{\text{旋转}} = \frac{1}{2} \times 8.33 = 4.17$ms
>
> 所需扇区数 = $\lceil 2500/512 \rceil = 5$ 扇区
>
> $T_{\text{传输}} = \frac{5}{500} \times 8.33 = 0.083$ms
>
> $T_{\text{总}} = 8 + 4.17 + 0.083 \approx \mathbf{12.25}$ms

#### 4.3.4 磁盘管理

##### 磁盘初始化

- **低级格式化（物理格式化）**：将磁盘划分为扇区，每个扇区包含头部、数据区（通常 512B）和尾部。头部和尾部包含扇区号、校验码等信息。
- **分区**：将磁盘划分为一个或多个柱面组，每个分区可视为独立的逻辑磁盘。
- **高级格式化（逻辑格式化/创建文件系统）**：在分区上创建文件系统数据结构，包括引导块、超级块、空闲空间管理信息、根目录等。

##### 引导块

- 计算机启动时，ROM 中的**引导程序**读取磁盘的**引导块**（通常位于第 0 块）到内存执行
- 引导块中存放引导加载程序，负责加载操作系统内核
- 没有操作系统的磁盘（纯数据盘）不需要引导块

##### 坏块处理

- **坏块**：磁盘上无法正常读写的扇区
- 处理方式：
  - **硬件层面**：磁盘控制器维护备用扇区，将坏扇区重映射到备用扇区（对 OS 透明）
  - **软件层面**：OS 在文件系统中标记坏块，避免分配使用

> 磁盘管理中的坏块处理与 RAID 中的磁盘容错是不同的层次——前者是单块磁盘扇区级处理，后者是多磁盘系统级容错。

### 5.1 I/O 管理概述

#### 5.1.1 I/O 设备分类

> **I/O 设备**是计算机系统中用于实现数据输入/输出的硬件装置，通过 **I/O 接口（Controller）** 与 CPU 和内存相连。

设备从多个维度进行分类：

| 分类方式 | 类型 | 说明 | 典型例子 |
|:---------|:-----|:-----|:---------|
| **按传输速率** | 低速设备 | 每秒几个~几百字节 | 键盘、鼠标 |
| | 中速设备 | 每秒数千~数万字节 | 打印机 |
| | 高速设备 | 每秒数十万~数兆字节 | 硬盘、SSD、内存 |
| **按信息交换单位** | **字符设备** | 以字符/字节为单位传输，不可寻址 | 键盘、终端、串口 |
| | **块设备** | 以数据块为单位传输，可寻址 | 硬盘、U 盘、SSD |
| **按共享属性** | **独占设备** | 同一时刻仅一个进程可访问 | 打印机 |
| | **共享设备** | 多个进程可交替访问 | 磁盘 |
| | **虚拟设备** | 通过 SPOOLing 将独占改造为共享 | 虚拟打印机 |

**I/O 接口（控制器）**：

> **I/O 接口（Controller）**：连接 CPU 与 I/O 设备的桥梁，屏蔽不同设备之间的物理差异，使 CPU 能够用统一的指令和控制方式访问各类外设。

I/O 接口的主要功能：

1. **数据缓冲**：缓解 CPU 与 I/O 设备之间的速度差异
2. **格式转换**：完成 CPU 数据格式与设备数据格式之间的转换（如串-并转换）
3. **状态反馈**：通过状态寄存器向 CPU 报告设备当前状态（忙/闲、就绪/出错）
4. **控制命令**：接收并执行 CPU 发出的 I/O 控制指令
5. **地址识别**：根据地址线识别 CPU 访问的目标设备

> I/O 接口不等于设备本身。设备 = **机械部件**（执行实际 I/O 操作） + **电子部件（控制器/接口）**（控制机械部件）。控制器通常集成在设备内部或主板上。

#### 5.1.2 I/O 控制方式

四种 I/O 控制方式的演进本质上是在**逐步将 CPU 从繁重的数据传送任务中解放出来**。

##### （1）程序直接控制方式（轮询）

```
CPU 发起 I/O → 循环检查设备状态 → 就绪则传输
```

- CPU 不断轮询设备状态寄存器，处于**忙等**（busy-waiting）状态
- CPU 利用率极低，CPU 与 I/O 完全串行
- 优点：硬件简单、成本低
- 适用：极简单的嵌入式系统

##### （2）中断驱动方式

```
CPU 发起 I/O → 进程阻塞（CPU 可做其他事）→ 设备就绪发中断 → CPU 转去处理
```

- 设备完成一个字符后向 CPU 发中断信号
- CPU 响应中断，执行中断处理程序传输数据
- 优点：CPU 利用率显著提高，CPU 与 I/O 可并行
- 缺点：**每传送一个字符/字节就中断一次**，中断开销大
- 适用：低速字符设备（键盘、鼠标）

##### （3）DMA 方式

> **DMA（Direct Memory Access，直接存储器访问）**：由 DMA 控制器直接控制数据在**内存与 I/O 设备之间**的传输，仅在数据块传送的开始和结束时需要 CPU 介入。

**DMA 控制器组成**：

- 内存地址寄存器（MAR）：存放数据在内存中的目标/源地址
- 数据计数器（DC）：记录剩余待传送字节数
- 控制/状态寄存器：存放控制命令和设备状态
- 数据缓冲寄存器：暂存传输数据

**DMA 工作过程**：

```
1. CPU 准备阶段：设置 DMA 控制器（源地址、目标地址、传送长度、传送方向）
2. CPU 启动 DMA，然后继续执行其他进程
3. DMA 控制器接管总线，完成数据传输（周期窃取/周期挪用）
4. 整块数据传输完毕，DMA 控制器向 CPU 发中断
5. CPU 执行中断处理，做收尾工作
```

- 以**块**为单位传送，大幅减少中断次数
- CPU 仅在开始和结束时介入
- 适用：高速块设备（磁盘、SSD）

##### （4）通道控制方式

> **通道（Channel）**：一种**专用的 I/O 处理器**，拥有自己的指令集（通道指令），能独立执行**通道程序**来控制 I/O 操作，几乎完全解放 CPU。

**通道类型**：

| 通道类型 | 特点 | 适用 |
|:---------|:-----|:-----|
| **字节多路通道** | 以字节交叉方式为多台低速设备服务 | 多台低速字符设备 |
| **数组选择通道** | 一次只为一个高速设备服务，传送完一个数组才释放 | 单台高速设备 |
| **数组多路通道** | 以数组为单位交叉为多台高速设备服务 | 多台高速块设备 |

- 通道独立于 CPU 工作，与 CPU 并行执行
- 通道执行通道程序，能完成复杂的 I/O 控制
- 适用：大吞吐量系统（大型机）
- 缺点：成本高、硬件复杂

##### 四种方式综合对比

| 对比维度 | 程序直接控制 | 中断驱动 | DMA | 通道 |
|:---------|:------------|:---------|:----|:-----|
| **数据传送单位** | 字/字节 | 字/字节 | **块** | 块组 |
| **CPU 介入程度** | 全程介入 | 每次字符中断 | 仅开始/结束 | 极低（发 I/O 指令） |
| **CPU 利用率** | 极低 | 中等 | 高 | 很高 |
| **CPU 与 I/O 并行性** | 完全串行 | 可并行 | 可并行 | 高度并行 |
| **中断次数** | 无 | 每字符一次 | 每块一次 | 每批一次 |
| **硬件成本** | 极低 | 低 | 中等 | 高 |
| **适用设备** | 极简单设备 | 低速字符设备 | 高速块设备 | 大吞吐量系统 |
| **I/O 控制主体** | CPU | CPU + 中断 | DMA 控制器 | 通道（专用处理器） |

> - **DMA 与中断的区别**：中断方式每传送一个字节就中断一次 CPU；DMA 方式传送一整块数据才中断一次，CPU 开销大幅降低。
> - **通道与 DMA 的区别**：DMA 控制器只能完成简单的数据传送，通道是专用处理器，能执行通道程序，控制能力更强。
> - **DMA 的数据传送路径**：内存 ↔ DMA 控制器 ↔ 设备（不经过 CPU）。

#### 5.1.3 I/O 系统层次结构

I/O 软件采用分层设计，自顶向下分为：

```
┌─────────────────────────────┐
│       用户层 I/O 软件        │  用户进程调用库函数（printf、scanf）
├─────────────────────────────┤
│     设备无关软件层          │  统一接口、设备命名、保护、缓冲、分配
├─────────────────────────────┤
│      设备驱动层             │  与硬件交互，设置寄存器、处理中断
├─────────────────────────────┤
│      中断处理层             │  保存/恢复上下文，转入中断处理程序
├─────────────────────────────┤
│         硬件                │  设备控制器 + 机械设备
└─────────────────────────────┘
```

| 层次 | 主要功能 | 说明 |
|:-----|:---------|:-----|
| **用户层 I/O 软件** | 提供 I/O 库函数 | 如 `printf`、`scanf`，将请求传递给下层 |
| **设备无关软件层** | 统一接口、命名、保护、缓冲、分配 | 执行所有设备共用的 I/O 功能，映射逻辑名到物理设备 |
| **设备驱动层** | 将上层请求转换为设备指令 | 直接与控制器交互，每种设备对应一个驱动程序 |
| **中断处理层** | 处理中断，唤醒阻塞进程 | 保存上下文，执行中断服务例程 |
| **硬件** | 执行实际 I/O 操作 | 设备控制器 + 机械设备 |

> - 设备无关性由**设备无关软件层**实现，而不是设备驱动层。
> - 逻辑设备名到物理设备名的映射（LUT）在设备无关软件层完成。

### 5.2 I/O 核心子系统

#### 5.2.1 I/O 调度

> **I/O 调度**主要针对磁盘这类共享块设备。I/O 调度器维护一个**请求队列**，通过重新排列 I/O 请求的次序来优化整体性能。

**调度目标**：优化 I/O 请求的**执行顺序**，以减少磁盘寻道时间、旋转延迟，提高系统吞吐量和响应速度。

**主要调度策略**：

| 算法 | 策略 | 优点 | 缺点 |
|:-----|:-----|:-----|:-----|
| **先来先服务（FCFS）** | 按请求到达顺序处理 | 简单、公平 | 寻道时间长，效率低 |
| **最短寻道时间优先（SSTF）** | 优先处理离当前磁道最近的请求 | 寻道时间短 | 可能导致饥饿 |
| **扫描算法（SCAN/电梯算法）** | 磁头单向移动，处理沿途请求，到一端再反向 | 克服饥饿，较公平 | 两端请求等待时间长 |
| **循环扫描（C-SCAN）** | 单向扫描到底后立即返回起点（返回时不处理） | 均匀响应，公平性更好 | 返回路程空转 |
| **LOOK 算法** | 改进 SCAN，反向前检查是否还有请求 | 减少无请求区域的扫描 | — |
| **C-LOOK 算法** | 改进 C-SCAN，到最后一个请求即返回起点 | 效率比 C-SCAN 高 | — |

> - 磁盘调度算法属于 **I/O 调度**范畴，详细算法推演在第四章文件管理中介绍。
> - SCAN 和 C-SCAN 的核心区别：SCAN 到达一端后反向继续处理；C-SCAN 到达一端后直接返回起点（返回时不处理）。

#### 5.2.2 高速缓存与缓冲区

**缓冲的作用**：

> **缓冲（Buffer）**：在内存中开辟的**临时存储区域**，用于暂存 I/O 数据。

缓冲的三大作用：
1. **缓和 CPU 与 I/O 设备的速度差异**：CPU 速度远快于外设，缓冲削峰填谷
2. **减少对 CPU 的中断次数**：数据积累到一定量再中断
3. **提高 CPU 与 I/O 设备的并行性**：CPU 处理数据的同时，设备传输下一批数据

**四种缓冲方式**：

| 缓冲类型 | 描述 | 并行性 | 适用场景 |
|:---------|:-----|:-------|:---------|
| **单缓冲** | 一个缓冲区，I/O 和处理交替进行 | 低 | 简单设备 |
| **双缓冲** | 两个缓冲区交替使用（一个接收 I/O，另一个被 CPU 处理） | 中 | 中等速度设备 |
| **循环缓冲** | 多个大小相等的缓冲区组成循环队列 | 较高 | 速度波动设备 |
| **缓冲池** | 统一管理多个缓冲区，按类型分配 | 高 | 通用系统 |

**缓冲区时间计算**：

> 设 $C$ = CPU 处理一块数据的时间，$T$ = I/O 设备输入一块数据的时间，$M$ = 将数据从系统缓冲区传送到用户区的时间。
>
> 单缓冲下，处理一块数据的总时间：
>
> $$
> T_{\text{单}} = \max(C, T) + M
> $$
>
> （因为 CPU 处理和对下一块的 I/O 输入可以重叠，但传送到用户区不能与 I/O 重叠）

> 双缓冲下，两个缓冲区交替工作，一块数据被 CPU 处理的同时，I/O 设备可以输入下一块数据。处理一块的平均时间约：
>
> $$
> T_{\text{双}} \approx \max(C + M, \; T)
> $$
>
> 前提：$T \leq C + M$（即 I/O 速度不比处理速度慢太多），否则双缓冲退化为单缓冲。

> - 双缓冲的优势体现条件是 **I/O 设备速度与 CPU 处理速度基本匹配**。若 CPU 处理极快（$C$ 很小），双缓冲和单缓冲差异不大。
> - 只有在系统缓冲区和用户进程空间之间的数据移动（$M$）也需要时间，不可忽略。

**缓冲池**：

> **缓冲池**由系统统一管理，由多个大小相等的缓冲区组成，通过三个队列组织。

**三个队列**：

| 队列 | 全称 | 存放内容 |
|:-----|:-----|:---------|
| **emq**（空缓冲队列） | Empty Buffer Queue | 空闲缓冲区 |
| **inq**（输入缓冲队列） | Input Buffer Queue | 装满输入数据的缓冲区 |
| **outq**（输出缓冲队列） | Output Buffer Queue | 装满输出数据的缓冲区 |

**四种工作缓冲区类型**：

| 类型 | 来源队列 | 去向队列 | 含义 |
|:-----|:---------|:---------|:-----|
| 收容输入 | emq → inq | （留作收容输入用） | 从设备收数据填入缓冲区 |
| 提取输入 | inq → emq | （留作提取输入用） | 用户从缓冲区取走输入数据 |
| 收容输出 | emq → outq | （留作收容输出用） | 用户将输出数据填入缓冲区 |
| 提取输出 | outq → emq | （留作提取输出用） | 将数据从缓冲区发送到设备 |

**两个基本操作**：
- `GetBuf(queue)`：从指定队列取一个缓冲区
- `PutBuf(buf, queue)`：将缓冲区归还到指定队列

#### 5.2.3 设备分配与回收

**设备分配的数据结构**：

| 数据结构 | 全称 | 内容 | 数量 |
|:---------|:-----|:-----|:-----|
| **SDT** | 系统设备表（System Device Table） | 系统中所有设备的入口 | 系统级，一张 |
| **DCT** | 设备控制表（Device Control Table） | 每台设备的状态、类型、等待队列 | 每设备一张 |
| **COCT** | 控制器控制表（Controller Control Table） | 每个控制器的状态、连接设备 | 每控制器一张 |
| **CHCT** | 通道控制表（Channel Control Table） | 每个通道的状态 | 每通道一张 |

> SDT → DCT → COCT → CHCT 构成了设备分配的**四级查找链**，从系统总表逐步定位到具体设备、控制器、通道。

**分配流程**：

```
1. 用户请求 I/O（使用逻辑设备名）
2. 查 SDT → 定位对应物理设备的 DCT
3. 查 DCT → 检查设备状态
   ├─ 忙：将进程挂入该 DCT 的等待队列
   └─ 空闲：分配设备，标记 DCT 为"忙"
4. 查 COCT → 分配控制器（若不空闲则等待）
5. 查 CHCT → 分配通道（可选，无通道系统跳过）
6. 返回设备句柄给进程
```

**分配策略与安全性**：

| 分配方式 | 特点 | 安全性 | 利用率 |
|:---------|:-----|:-------|:-------|
| **独占分配** | 一个设备分配给一个进程，用完才释放 | 安全（无死锁风险） | 低 |
| **共享分配** | 多个进程可交替访问同一设备 | 需额外同步 | 高 |
| **虚拟分配** | 通过 SPOOLing 将独占设备虚拟为共享 | 安全 | 高 |

**逻辑设备名到物理设备名的映射（LUT）**：

> **LUT（Logical Unit Table，逻辑设备表）**：实现逻辑设备名到物理设备名（驱动程序入口）的映射，使应用程序可以用**设备无关**的方式请求 I/O。

- LUT 位于**设备无关软件层**
- 每个表项包含：逻辑设备名、物理设备名、驱动程序入口地址
- 应用程序使用逻辑设备名申请设备，系统通过 LUT 定位实际驱动程序

> - 分配设备时需**同时分配设备、控制器、通道**三级资源，缺一不可。
> - 分配顺序必须是 **先设备 → 再控制器 → 最后通道**。
> - 独占分配虽然安全，但设备利用率极低，SPOOLing 正是为了解决这一矛盾。

#### 5.2.4 SPOOLing 技术（假脱机）

> **SPOOLing（Simultaneous Peripheral Operations On-Line，假脱机技术/同时联机外围操作）**：在**多道程序**环境下，利用**磁盘**来模拟独占 I/O 设备，将一台物理上的独占设备改造成逻辑上的**多台虚拟设备**，供多个用户进程"同时"使用。

**SPOOLing 系统组成**：

```
┌──────────────────────────────────────────┐
│               SPOOLing 系统               │
├──────────────┬───────────────────────────┤
│   输入部分    │         输出部分           │
├──────────────┼───────────────────────────┤
│  输入井（磁盘）│       输出井（磁盘）        │
│  输入缓冲区   │       输出缓冲区           │
│  输入进程     │       输出进程（守护进程）   │
└──────────────┴───────────────────────────┘
```

| 组成部分 | 说明 |
|:---------|:-----|
| **输入井/输出井** | 磁盘上开辟的存储区域，模拟输入/输出设备 |
| **输入缓冲区/输出缓冲区** | 内存中的暂存区域，数据在井和设备间中转 |
| **输入进程（预输入进程）** | 将数据从输入设备预先送入输入井 |
| **输出进程（缓输出进程）** | 系统空闲时将输出井数据送到输出设备 |
| **请求打印队列** | 每个用户打印请求以文件形式存入输出井，排队 |

**共享打印机实现原理**：

>
> 1. 用户进程发出打印请求，系统为该请求在**输出井**中分配一空闲盘块
> 2. 输出进程（守护进程）为用户创建一个**打印请求文件**，将待打印数据写入输出井
> 3. 用户进程的打印操作"完成"（事实上数据还在磁盘），可以继续执行
> 4. 待打印机空闲，守护进程**从输出井逐个取出**打印请求文件，实际打印

**效果**：多个用户进程可以"同时"使用一台打印机，每个用户都感觉自己独占打印机。

**守护进程（Daemon）**：

> **守护进程**：SPOOLing 系统中的专用后台进程（如打印守护进程），负责在设备空闲时将输出井中的请求逐个发送到物理设备执行。

> - SPOOLing 技术需要**多道程序**环境支持，并且需要**磁盘**作为中间存储介质。
> - SPOOLing 将**独占设备**改造为**共享设备**，不是把物理设备变成多个，而是通过磁盘和排队机制虚拟化。
> - SPOOLing 的"假脱机"含义：用户感觉像"脱机"操作（数据先写入磁盘），但实际上系统是"联机"的。

#### 5.2.5 出错处理

> I/O 操作中可能发生的错误：设备机械故障（磁头损坏、电机卡死）、传输错误（数据校验错误）、控制器异常、设备忙/超时、介质损坏（磁盘坏块）等。

操作系统对 I/O 错误的处理策略：

| 策略 | 说明 |
|:-----|:-----|
| **重试** | 对于瞬时性错误（传输校验错、设备忙），进行有限次重试 |
| **出错日志记录** | 将错误信息记入系统日志，供管理员分析 |
| **设备状态恢复** | 复位控制器、重新校准设备，尝试恢复正常工作状态 |
| **上层报告** | 将不可恢复的错误报告给调用进程（返回错误码） |
| **设备隔离** | 将故障设备标记为不可用，重新分配或降级运行 |

> - 重试策略适用于 **瞬时性可恢复错误**（如偶尔的校验失败），不适用于 **永久性硬件故障**（如磁盘物理坏道——需用备用扇区替换）。
> - 大容量存储系统（如 RAID）有专门的容错机制，操作系统层面的出错处理与硬件容错协同工作。

### 第五章高频考点速查

>
> 1. **设备分类**：按速率（低/中/高）、按交换单位（字符/块）、按共享属性（独占/共享/虚拟）
> 2. **四种 I/O 控制方式**：程序直接控制 → 中断驱动 → DMA → 通道，核心是**逐步解放 CPU**
> 3. **DMA 控制器**：组成（MAR/DC/控制寄存器）、工作过程、与中断方式的区别
> 4. **通道**：三种类型（字节多路/数组选择/数组多路）及各自适用场景
> 5. **I/O 软件层次**：用户层 → 设备无关层 → 驱动层 → 中断处理层 → 硬件
> 6. **设备分配数据结构**：SDT → DCT → COCT → CHCT，四级查找链
> 7. **设备分配方式**：独占/共享/虚拟，安全性分析
> 8. **逻辑设备名 → 物理设备名映射（LUT）**
> 9. **SPOOLing 系统**：三大组成（输入/输出井 + 缓冲区 + 进程）、打印机共享原理
> 10. **缓冲区**：单缓冲处理时间 $\max(C, T) + M$、双缓冲、循环缓冲、缓冲池（三个队列 + 四种类型）
> 11. **磁盘访问时间**：$T_{\text{访问}} = T_s + T_r + T_t$，平均旋转延迟 $T_r = \frac{1}{2r}$
> 12. **SSD 特点**：Flash 闪存、写前擦除、寿命有限

> - 四种 I/O 控制方式的**对比**是高频简答题，注意从数据单位、CPU 介入程度、并行性、中断频次四个维度回答。
> - DMA 适用**块设备**，这也是区分 DMA 和中断方式的关键标志。
> - SPOOLing 的核心是将**独占设备**改造成**共享设备**，关键词：磁盘模拟、多道程序环境。
> - 缓冲区时间计算抓住 $\max(C, T) + M$ 公式即可应对单缓冲计算题。
> - 磁盘平均访问时间的计算务必区分"平均旋转延迟"（半圈）与"旋转一周时间"。
