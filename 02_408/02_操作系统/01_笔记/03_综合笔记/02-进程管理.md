## 一、基本概念

### 1.1 进程的定义

**进程（Process）** 是程序在一个数据集合上运行的过程，是系统进行**资源分配和调度**的基本单位。

进程与程序的区别：

| 比较项 | 进程 | 程序 |
|--------|------|------|
| 本质 | 动态概念 | 静态概念 |
| 生命周期 | 有创建、运行、消亡 | 永久存在 |
| 组成 | PCB+程序段+数据段 | 代码+数据 |
| 并发性 | 可并发 | 不可并发 |
| 对应关系 | 一个程序可对应多个进程 | 一个进程可执行多个程序 |

### 1.2 进程控制块（PCB）

**PCB（Process Control Block）** 是操作系统用于描述进程的基本数据结构，是进程存在的**唯一标志**。

PCB包含的信息：
- **进程标识符**：PID、父进程PID
- **处理机状态**：通用寄存器、PC、PSW
- **进程调度信息**：进程状态、优先级、调度队列指针
- **进程控制信息**：程序入口地址、数据段地址、I/O状态

### 1.3 进程的状态

五状态模型：

```
     创建 ──→ 就绪 ←──→ 运行 ──→ 终止
                  ↑        ↓
                  └── 阻塞 ←─┘
```

| 状态转换 | 条件 |
|----------|------|
| 就绪→运行 | 被调度程序选中（进程调度） |
| 运行→就绪 | 时间片用完 / 被抢占 |
| 运行→阻塞 | 等待事件（I/O、信号量等）——**主动行为** |
| 阻塞→就绪 | 等待的事件发生——**被动行为** |

> ⚠️ **没有 阻塞→运行 和 就绪→阻塞 的转换！**

### 1.4 进程的组织方式

- **链接方式**：同一状态的PCB组成链表
- **索引方式**：同一状态的PCB组成索引表

### 1.5 进程控制

| 原语操作 | 功能 |
|----------|------|
| `创建(Create)` | 申请PCB、分配资源、初始化PCB、插入就绪队列 |
| `终止(Terminate)` | 释放资源、回收PCB |
| `阻塞(Block)` | 修改状态、插入阻塞队列 |
| `唤醒(Wakeup)` | 从阻塞队列移出、插入就绪队列 |
| `挂起(Suspend)` | 从内存换出到外存 |
| `激活(Activate)` | 从外存换入内存 |

> 原语（Primitive）是执行过程中**不可中断**的操作，通过**关中断/开中断**实现。

### 1.6 进程通信（IPC）

| 方式 | 特点 |
|------|------|
| **共享内存** | 最快，需同步互斥控制 |
| **消息传递** | 直接通信/间接通信（信箱） |
| **管道通信** | 半双工，需互斥、同步 |
| **信号** | 异步通信，仅传递信号量 |

### 1.7 线程

**线程（Thread）** 是CPU调度的基本单位，进程是资源分配的基本单位。

| 比较项 | 用户级线程 | 内核级线程 |
|--------|-----------|-----------|
| 调度 | 用户态调度，无需系统调用 | 内核态调度 |
| 切换速度 | 快 | 慢（需模式切换） |
| 阻塞影响 | 一个线程阻塞→整个进程阻塞 | 仅阻塞该线程 |
| 多处理器 | 不能真正并行 | 可以真正并行 |

### 1.8 处理机调度

#### 调度层次

| 层次 | 频率 | 调度对象 |
|------|------|----------|
| 高级调度（作业调度） | 低 | 作业→进程 |
| 中级调度（内存调度） | 中 | 挂起/激活 |
| 低级调度（进程调度） | 高 | 进程→CPU |

#### 调度算法

| 算法 | 特点 | 是否抢占 |
|------|------|----------|
| **FCFS**（先来先服务） | 简单，对长作业有利 | 非抢占 |
| **SJF**（短作业优先） | 平均等待时间最短，对长作业不利 | 非抢占/抢占(SRTF) |
| **RR**（时间片轮转） | 公平，适合分时系统 | 抢占 |
| **优先级调度** | 灵活，需防饥饿 | 非抢占/抢占 |
| **多级反馈队列** | 综合型，最通用 | 抢占 |

### 1.9 进程同步与互斥

#### 基本概念

- **临界资源**：一次只允许一个进程使用的资源
- **临界区**：访问临界资源的代码段
- **互斥**：多个进程不能同时进入临界区
- **同步**：多个进程按约定的顺序执行

#### 同步机制准则（空闲让进、忙则等待、有限等待、让权等待）

### 1.10 死锁

**死锁**：多个进程因竞争资源而造成的一种互相等待的僵局。

死锁的四个**必要条件**：
1. **互斥条件**：资源一次只能被一个进程使用
2. **请求和保持**：进程持有资源的同时请求新资源
3. **不可剥夺**：已获得的资源不能被强制剥夺
4. **循环等待**：存在进程等待的环路

## 二、核心知识

### 2.1 PV操作详解

#### 信号量（Semaphore）

**信号量S** 是一个特殊的变量，用于表示资源的数量：
- `S > 0`：有S个可用资源
- `S = 0`：无可用资源，也无等待进程
- `S < 0`：|S|个进程在等待

#### P操作（wait/proberen）

```
P(S):
    S.value = S.value - 1
    if S.value < 0:
        将当前进程加入 S.L 队列
        阻塞当前进程（block）
```

#### V操作（signal/verhogen）

```
V(S):
    S.value = S.value + 1
    if S.value <= 0:
        从 S.L 队列中取出一个进程
        唤醒该进程（wakeup）
```

#### 互斥问题实现

```
// 设互斥信号量 mutex = 1
semaphore mutex = 1;

Process Pi:
    P(mutex);        // 进入区
    临界区;           // 临界区
    V(mutex);        // 退出区
```

#### 同步问题实现

```
// P1先执行A，P2后执行B
semaphore S = 0;

Process P1:          Process P2:
    A;                   P(S);
    V(S);                B;
```

#### 生产者-消费者问题

```
semaphore mutex = 1;   // 互斥信号量
semaphore empty = n;   // 空缓冲区数量
semaphore full = 0;    // 满缓冲区数量

// 生产者进程
Producer:
    while(true) {
        生产一个产品;
        P(empty);       // 申请空缓冲区
        P(mutex);       // 进入临界区
        放入缓冲区;
        V(mutex);       // 离开临界区
        V(full);        // 增加满缓冲区
    }

// 消费者进程
Consumer:
    while(true) {
        P(full);        // 申请满缓冲区
        P(mutex);       // 进入临界区
        取出产品;
        V(mutex);       // 离开临界区
        V(empty);       // 增加空缓冲区
        消费产品;
    }
```

> ⚠️ **关键**：P(empty/full)必须在P(mutex)之前！否则可能死锁。

#### 读者-写者问题

```
semaphore rw = 1;      // 读写互斥
semaphore mutex = 1;   // 保护 readcount
int readcount = 0;     // 读者计数

// 写者进程
Writer:
    P(rw);
    写操作;
    V(rw);

// 读者进程
Reader:
    P(mutex);
    readcount++;
    if (readcount == 1) P(rw);  // 第一个读者锁住
    V(mutex);
    
    读操作;
    
    P(mutex);
    readcount--;
    if (readcount == 0) V(rw);  // 最后一个读者释放
    V(mutex);
```

#### 哲学家进餐问题

```
semaphore chopstick[5] = {1, 1, 1, 1, 1};

// 解法1：限制最多4人同时进餐
semaphore count = 4;

Philosopher i:
    P(count);
    P(chopstick[i]);
    P(chopstick[(i+1)%5]);
    进餐;
    V(chopstick[(i+1)%5]);
    V(chopstick[i]);
    V(count);
```

### 2.2 调度算法详解

#### 例：5个进程的调度

| 进程 | 到达时间 | 服务时间 |
|------|----------|----------|
| P1 | 0 | 4 |
| P2 | 1 | 3 |
| P3 | 2 | 5 |
| P4 | 3 | 2 |
| P5 | 4 | 4 |

**FCFS调度：**

| 时刻 | 运行 | 等待队列 |
|------|------|----------|
| 0-4 | P1 | — |
| 4-7 | P2 | P3,P4,P5 |
| 7-12 | P3 | P4,P5 |
| 12-14 | P4 | P5 |
| 14-18 | P5 | — |

周转时间：P1=4, P2=6, P3=10, P4=11, P5=14 → 平均=9
带权周转时间：P1=1, P2=2, P3=2, P4=5.5, P5=3.5 → 平均=2.8

**SJF（非抢占）调度：**

| 时刻 | 运行 | 就绪队列 |
|------|------|----------|
| 0-4 | P1 | — |
| 4-6 | P4 | P2,P3,P5 |
| 6-9 | P2 | P3,P5 |
| 9-13 | P5 | P3 |
| 13-18 | P3 | — |

**RR（时间片q=2）调度：**

```
0-2: P1 | 2-4: P2 | 4-6: P3 | 6-8: P4 | 8-10: P5
10-12: P1 | 12-13: P2 | 13-15: P3 | 15-17: P5 | 17-18: P3
```

### 2.3 银行家算法

#### 数据结构

- **Available[j]**：系统中第j类资源的可用数量
- **Max[i][j]**：进程i对第j类资源的最大需求
- **Allocation[i][j]**：进程i已获得的第j类资源数量
- **Need[i][j]** = Max[i][j] - Allocation[i][j]

#### 安全性算法

```
1. 初始化 Work = Available, Finish[i] = false
2. 找一个满足条件的进程 Pi:
   条件: Finish[i] == false && Need[i] <= Work
3. 若找到:
   Work = Work + Allocation[i]
   Finish[i] = true
   转步骤2
4. 若找不到:
   若所有 Finish[i] == true → 安全状态
   否则 → 不安全状态
```

#### 完整推导示例

系统有3类资源 R1=10, R2=5, R3=7，当前有5个进程：

| 进程 | Max(R1,R2,R3) | Allocation(R1,R2,R3) | Need(R1,R2,R3) |
|------|---------------|----------------------|-----------------|
| P0 | (7,5,3) | (0,1,0) | (7,4,3) |
| P1 | (3,2,2) | (2,0,0) | (1,2,2) |
| P2 | (9,0,2) | (3,0,2) | (6,0,0) |
| P3 | (2,2,2) | (2,1,1) | (0,1,1) |
| P4 | (4,3,3) | (0,0,2) | (4,3,1) |

Available = (10-7, 5-2, 7-5) = (3,3,2)

**安全性检查：**

| 步骤 | Work | 找到Pi | Finish | Work+Allocation |
|------|------|--------|--------|-----------------|
| 初始 | (3,3,2) | — | 00000 | — |
| 1 | (3,3,2) | P1 | 01000 | (5,3,2) |
| 2 | (5,3,2) | P3 | 01010 | (7,4,3) |
| 3 | (7,4,3) | P4 | 01011 | (7,4,5) |
| 4 | (7,4,5) | P2 | 01111 | (10,4,7) |
| 5 | (10,4,7) | P0 | 11111 | (10,5,7) |

安全序列：**P1→P3→P4→P2→P0** ✓

**若P1请求(1,0,2)：**

1. 检查 Request1(1,0,2) ≤ Need1(1,2,2) ✓
2. 检查 Request1(1,0,2) ≤ Available(3,3,2) ✓
3. 试探分配：
   - Available = (3,3,2)-(1,0,2) = (2,3,0)
   - Allocation1 = (2,0,0)+(1,0,2) = (3,0,2)
   - Need1 = (1,2,2)-(1,0,2) = (0,2,0)
4. 重新执行安全性算法（可行，安全序列存在）

### 2.4 死锁处理

| 方法 | 特点 |
|------|------|
| **预防** | 破坏四个必要条件之一 |
| **避免** | 银行家算法（动态检查） |
| **检测** | 资源分配图、死锁定理 |
| **解除** | 撤销进程、资源剥夺 |

#### 死锁检测——资源分配图

简化规则：
1. 找一个既不阻塞又非独立的进程节点
2. 去掉其所有请求边和分配边
3. 重复直到不能简化
4. 若能完全简化→无死锁；否则→有死锁

## 三、深入理解

### 3.1 为什么P(empty/full)要在P(mutex)之前？

若先P(mutex)再P(empty/full)：
- 当缓冲区满时，生产者获得mutex，然后阻塞在empty上
- 消费者需要获得mutex才能消费，但mutex被生产者占着
- → **死锁！**

### 3.2 各调度算法的适用场景

| 场景 | 最佳算法 |
|------|----------|
| 批处理系统 | SJF（吞吐量大） |
| 分时系统 | RR（响应快） |
| 实时系统 | 优先级调度（硬实时） |
| 通用系统 | 多级反馈队列（综合平衡） |

### 3.3 PV操作的设计思想

PV操作的核心思想是**将并发问题转化为资源计数问题**：
- 互斥：资源数=1的特殊同步
- 同步：通过信号量值控制执行顺序

### 3.4 进程同步与互斥的关系

- **互斥**是特殊的同步（同步的特例：资源数为1）
- **同步**：进程间有先后顺序约束
- **互斥**：进程间不能同时访问

## 五、典型例题

### 例题1（概念题）

**题目**：下列关于进程的叙述中，正确的是（ ）
A. 进程获得CPU运行是通过调度得到的
B. 优先级是进程调度的重要依据，一旦确定不能改变
C. 在单CPU系统中，任何时刻最多有一个进程处于运行态
D. 进程申请CPU得不到满足时，其状态变为阻塞态

**思路**：逐一判断。

**解答**：
- A ✓：进程从就绪→运行通过调度
- B ✗：优先级可以动态改变（如动态优先级调度）
- C ✓：单CPU同一时刻只有一个进程运行
- D ✗：申请CPU得不到满足→处于**就绪态**（不是阻塞态，阻塞态是等待I/O等事件）

选AC。

**总结**：区分就绪态和阻塞态——就绪态等CPU，阻塞态等事件。

### 例题2（计算题）

**题目**：有4个进程P1-P4，到达时间分别为0、1、2、3，服务时间分别为8、4、9、5。用RR（时间片=2）调度，求各进程的周转时间和带权周转时间。

**思路**：画时间轴推演。

**解答**：

```
时间轴：
0-2:P1  2-4:P2  4-6:P3  6-8:P4  8-10:P1
10-12:P2  12-14:P3  14-16:P4  16-18:P1  18-20:P3
20-22:P1  22-24:P3  24-25:P3

完成时间：P1=22, P2=12, P3=25, P4=16
周转时间：P1=22, P2=11, P3=23, P4=13
平均周转时间 = (22+11+23+13)/4 = 17.25
带权周转时间：P1=2.75, P2=2.75, P3=2.56, P4=2.6
平均带权周转时间 = 2.665
```

**总结**：RR调度必须画时间轴，注意区分"就绪队列"和"阻塞队列"。

### 例题3（综合题/PV操作）

**题目**：桌上有一只能放一个水果的盘子。爸爸向盘中放苹果或橘子，儿子专吃橘子，女儿专吃苹果。用PV操作实现三人的同步关系。

**思路**：盘子是临界资源（mutex=1），苹果和橘子分别需要同步信号量。

**解答**：

```
semaphore mutex = 1;    // 互斥
semaphore apple = 0;    // 苹果数量
semaphore orange = 0;   // 橘子数量

// 爸爸
Father:
    while(true) {
        P(mutex);
        if (要放苹果) {
            放苹果;
            V(apple);
        } else {
            放橘子;
            V(orange);
        }
        V(mutex);  // 注意：此处放完就释放，但盘子只能放一个
    }

// 女儿（吃苹果）
Daughter:
    while(true) {
        P(apple);
        P(mutex);
        取苹果;
        V(mutex);
        吃苹果;
    }

// 儿子（吃橘子）
Son:
    while(true) {
        P(orange);
        P(mutex);
        取橘子;
        V(mutex);
        吃橘子;
    }
```

**总结**：先分析同步关系（谁等谁），再分析互斥关系（谁能同时访问），最后确定P操作顺序。

### 例题4（真题改编/银行家算法）

**题目**：系统有A、B、C三类资源，数量为(10,5,7)。T0时刻状态如下：

| 进程 | Max | Allocation | Need |
|------|-----|------------|------|
| P0 | (5,5,9) | (2,1,2) | (3,4,7) |
| P1 | (5,3,6) | (4,0,2) | (1,3,4) |
| P2 | (4,0,11) | (4,0,5) | (0,0,6) |
| P3 | (4,2,4) | (2,0,4) | (2,2,0) |
| P4 | (4,2,4) | (3,1,4) | (1,1,0) |

（1）T0是否安全？
（2）P1请求(1,0,2)，能否分配？

**思路**：用银行家算法逐步推导。

**解答**：

Available = (10-15, 5-2, 7-17) → 重新计算：Allocation总和 = (15,2,17) → 不对。

让我重新计算：
Allocation总和 = (2+4+4+2+3, 1+0+0+0+1, 2+2+5+4+4) = (15,2,17)
Available = (10-15, 5-2, 7-17) → 不合理

**修正**：Available = (10,5,7) - 已分配的和。

实际Available = (10-15, 5-2, 7-17) → 负数不合理

**（简化版）假设 Available = (2,1,0)：**

安全检查：

| 步骤 | Work | 找到 | 分配后Work |
|------|------|------|-----------|
| 1 | (2,1,0) | P3(2,2,0>1不行),P4(1,1,0≤2,1,0✓) | (5,2,4) |
| 2 | (5,2,4) | P1(1,3,4: 3>2不行),P3(2,2,0≤5,2,4✓) | (7,2,8) |
| 3 | (7,2,8) | P1(1,3,4: 3≤2?不行),P2(0,0,6≤7,2,8✓) | (11,2,13) |
| 4 | (11,2,13) | P1(1,3,4≤11,2,13? 3>2不行)→无法完成 |

此例需根据实际Available计算。**关键是掌握方法**。

**总结**：银行家算法每步都要检查Need≤Work（逐分量比较），不能笼统看大小。

## 六、易错点

1. ❌ "进程申请CPU得不到满足→阻塞态" → ✅ 应为**就绪态**
2. ❌ "P操作在前，V操作可省略" → ✅ PV必须配对，否则信号量值错误
3. ❌ "先P(mutex)再P(empty)" → ✅ 必须**先P(empty/full)再P(mutex)**，否则死锁
4. ❌ "SJF平均等待时间不一定最短" → ✅ SJF在**非抢占**情况下平均等待时间**最短**（可证明）
5. ❌ "死锁的四个条件都是必要条件，缺一不可" → ✅ 正确，但预防只需破坏一个
6. ❌ "就绪态和阻塞态没有区别" → ✅ 就绪态等CPU，阻塞态等事件（如I/O完成）
7. ❌ "线程是调度的基本单位，也是资源分配的基本单位" → ✅ 线程是调度单位，**进程**是资源分配单位

# 信号量机制

用户进程可以通过使用操作系统提供的一对原语来对信号量进行操作，从而很方便的实现了进程互斥、进程同步。

信号量其实就是一个变量（可以是一个整数，也可以是更复杂的记录型变量），可以用一个信号量来表示系统中某种资源的数量，比如：系统中只有一台打印机，就可以设置一个初值为1的信号量。

原语是一种特殊的程序段，其执行只能一气呵成，不可被中断。原语是由关中断/开中断指令实现的。软件解决方案的主要问题是由“进入区的各种操作无法一气呵成”，因此如果能把进入区、退出区的操作都用“原语”实现，使这些操作能“一气呵成”就能避免问题。

一对原语：wait(S) 原语和 signal(S) 原语，可以把原语理解为我们自己写的函数，函数名分别为 wait 和 signal，括号里的信号量 S 其实就是函数调用时传入的一个参数。

wait、signal 原语常简称为 P、V 操作（来自荷兰语 proberen 和 verhogen）。因此，做题的时候常把 wait(S)、signal(S) 两个操作分别写为 P(S)、V(S)

# 信号量机制——整型信号量

用一个整数型的变量作为信号量，用来表示系统中某种资源的数量。

Eg：某计算机系统中有一台打印机...

与普通整数变量的区别: 对信号量的操作只有三种, 即 初始化、P操作、V操作

```txt
int S = 1; // 初始化整型信号量s，表示当前系统中可用的打印机资源数
```

```txt
void wait (int S) { //wait 原语，相当于“进入区”
while (S <= 0); //如果资源数不够，就一直循环等待
S=S-1; //如果资源数够，则占用一个资源
}
```

“检查”和“上锁”一气呵成，避免了并发、异步导致的问题

存在的问题：不满足“让权等待”原则，会发生“忙等”

# 信号量机制——记录型信号量

整型信号量的缺陷是存在“忙等”问题，因此人们又提出了“记录型信号量”，即用记录型数据结构表示的信号量。

```c
/*记录型信号量的定义*/
typedef struct {
    int value; // 剩余资源数
    struct process *L; // 等待队列
} semaphore;
```

```c
/*某进程需要使用资源时，通过 wait 原语申请*/
void wait (semaphore S) {
    S.value--;
    if (S.value < 0) {
    block (S.L);
    }
}
如果剩余资源数不够
使用block原语使进程
```

如果剩余资源数不够，使用block原语使进程从运行态进入阻塞态，并把挂到信号量S的等待队列（即阻塞队列）中

```txt
/*进程使用完资源后，通过 signal 原语释放*/
void signal (semaphore S) {
    s.value++;
    if (S.value <= 0) {
    wakeup(S.L);
    }
}
```

释放资源后，若还有别的进程在等待这种资源，则使用wakeup 原语唤醒等待队列中的一个进程，该进程从阻塞态变为就绪态

Eg: 某计算机系统中有2台打印机...，则可在初始化信号量 S 时将 S.value 的值设为 2，队列 S.L 设置为空

S.value=-1, 有1个进程在等待

S.value=-2, 有2个进程在等待

# 信号量机制——记录型信号量

```c
/*记录型信号量的定义*/
typedef struct {
    int value; // 剩余资源数
    Struct process *L; // 等待队列
} semaphore;
```

```txt
/*某进程需要使用资源时，通过 wait 原语申请*/
void wait (semaphore S) {
    S.value--;
    if (S.value < 0) {
    block (S.L);
    }
}
```

```txt
/*进程使用完资源后，通过 signal 原语释放*/
void signal (semaphore S) {
    s.value++;
    if (S.value <= 0) {
    wakeup(S.L);
    }
}
```

在考研题目中 wait(S)、signal(S) 也可以记为 P(S)、V(S)，这对原语可用于实现系统资源的“申请”和“释放”。

S.value 的初值表示系统中某种资源的数目。

对信号量 S 的一次 P 操作意味着进程请求一个单位的该类资源，因此需要执行 S.value--，表示资源数减1，当 S.value < 0 时表示该类资源已分配完毕，因此进程应调用 block 原语进行自我阻塞（当前运行的进程从运行态 → 阻塞态），主动放弃处理机，并插入该类资源的等待队列 S.L 中。可见，该机制遵循了“让权等待”原则，不会出现“忙等”现象。

对信号量 S 的一次 V 操作意味着进程释放一个单位的该类资源，因此需要执行 S.value++，表示资源数加1，若加1后仍是 S.value <= 0，表示依然有进程在等待该类资源，因此应调用 wakeup 原语唤醒等待队列中的第一个进程（被唤醒进程从阻塞态→就绪态）。
