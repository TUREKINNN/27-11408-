# 2.4.3 避免死锁——银行家算法

## 📌 知识总览

银行家算法是荷兰学者Dijkstra为银行系统设计的，以确保银行在发放现金贷款时，不会发生不能满足所有客户需要的情况。后来该算法被用在操作系统中，用于**避免死锁**。

---

## 一、安全序列

### 1.1 生活中的例子

你是一位成功的银行家，手里掌握着**100亿**的资金...

有三个企业想找你贷款：企业B、企业A、企业T（简称BAT）
- B最多会借70亿
- A最多会借40亿
- T最多会借50亿

**江湖规矩：** 如果你借给企业的钱总数达不到企业提出的最大要求，那么不管你之前给企业借了多少钱，那些钱都拿不回来了...

### 1.2 安全序列的定义

> **安全序列**：就是指如果系统按照这种序列分配资源，则每个进程都能顺利完成。只要能找出一个安全序列，系统就是**安全状态**。

**特点：**
- 安全序列可能有多个
- 只要能找到**一个**安全序列，系统就是安全状态

### 1.3 安全状态与不安全状态

| 状态 | 含义 | 后果 |
|------|------|------|
| **安全状态** | 存在至少一个安全序列 | **一定不会发生死锁** |
| **不安全状态** | 找不出任何一个安全序列 | **可能发生死锁** |

> **重要关系：**
> - 安全状态 → 一定不会发生死锁
> - 不安全状态 → 可能发生死锁（不是一定发生）
> - 发生死锁 → 一定是在不安全状态

---

## 二、银行家算法的核心思想

> 在进程提出资源申请时，**先预判**此次分配是否会导致系统进入不安全状态。如果会进入不安全状态，就**暂时不答应**这次请求，让该进程先阻塞等待。

### 2.1 算法步骤

1. 检查请求的资源数是否超过进程声明的**最大需求**
2. 检查请求的资源数是否超过系统**当前可用资源**
3. **试探性分配**：假设将资源分配给该进程
4. 执行**安全性算法**，检查系统是否仍处于安全状态
5. 如果安全，则正式分配；如果不安全，则撤销试探性分配，让进程等待

## 三、银行家算法实例 ⭐⭐⭐

### 3.1 问题描述

某系统中有5个进程P0~P4，3种资源R0~R2，初始数量为(10, 5, 7)。某一时刻的情况如下：

| 进程 | 最大需求 | 已分配 | 最多还需要 |
|------|---------|--------|-----------|
| P0 | (7, 5, 3) | (0, 1, 0) | (7, 4, 3) |
| P1 | (3, 2, 2) | (2, 0, 0) | (1, 2, 2) |
| P2 | (9, 0, 2) | (3, 0, 2) | (6, 0, 0) |
| P3 | (2, 2, 2) | (2, 1, 1) | (0, 1, 1) |
| P4 | (4, 3, 3) | (0, 0, 2) | (4, 3, 1) |

**计算：**
- 已分配总数：(0+2+3+2+0, 1+0+0+1+0, 0+0+2+1+2) = (7, 2, 5)
- 剩余可用资源：(10-7, 5-2, 7-5) = (3, 3, 2)

### 3.2 安全性算法执行过程

**第一步：** 检查剩余可用资源 (3, 3, 2) 是否能满足各进程的需求

| 进程 | 最多还需要 | 能否满足 | 操作 |
|------|-----------|---------|------|
| P0 | (7, 4, 3) | 不能（7>3） | 跳过 |
| P1 | (1, 2, 2) | **能**（1≤3, 2≤3, 2≤2） | 加入安全序列 |
| P2 | (6, 0, 0) | 不能（6>3） | 跳过 |
| P3 | (0, 1, 1) | **能**（0≤3, 1≤3, 1≤2） | 加入安全序列 |
| P4 | (4, 3, 1) | 不能（4>3） | 跳过 |

**选择P1加入安全序列**，P1完成后释放资源：
- 新的剩余可用资源 = (3, 3, 2) + (2, 0, 0) = (5, 3, 2)

**第二步：** 检查剩余可用资源 (5, 3, 2) 是否能满足剩余进程的需求

| 进程 | 最多还需要 | 能否满足 | 操作 |
|------|-----------|---------|------|
| P0 | (7, 4, 3) | 不能（7>5） | 跳过 |
| P2 | (6, 0, 0) | 不能（6>5） | 跳过 |
| P3 | (0, 1, 1) | **能** | 加入安全序列 |
| P4 | (4, 3, 1) | **能** | 加入安全序列 |

**选择P3加入安全序列**，P3完成后释放资源：
- 新的剩余可用资源 = (5, 3, 2) + (2, 1, 1) = (7, 4, 3)

**第三步：** 检查剩余可用资源 (7, 4, 3) 是否能满足剩余进程的需求

| 进程 | 最多还需要 | 能否满足 | 操作 |
|------|-----------|---------|------|
| P0 | (7, 4, 3) | **能** | 加入安全序列 |
| P2 | (6, 0, 0) | **能** | 加入安全序列 |
| P4 | (4, 3, 1) | **能** | 加入安全序列 |

**选择P0加入安全序列**，P0完成后释放资源：
- 新的剩余可用资源 = (7, 4, 3) + (0, 1, 0) = (7, 5, 3)

**第四步：** 选择P2加入安全序列，P2完成后释放资源：
- 新的剩余可用资源 = (7, 5, 3) + (3, 0, 2) = (10, 5, 3)

**第五步：** 选择P4加入安全序列，P4完成后释放资源：
- 新的剩余可用资源 = (10, 5, 3) + (0, 0, 2) = (10, 5, 5)

### 3.3 安全序列

> **安全序列：P1 → P3 → P0 → P2 → P4**
>
> 系统处于**安全状态**，**一定不会发生死锁**。

## 四、银行家算法的实现

### 4.1 数据结构

```c
// 系统中有n个进程，m种资源
int Available[m];      // 可用资源向量
int Max[n][m];         // 最大需求矩阵
int Allocation[n][m];  // 已分配矩阵
int Need[n][m];        // 需求矩阵（Need = Max - Allocation）
int Request[m];        // 进程的请求向量
```

### 4.2 银行家算法伪代码

```c
// 进程Pi提出资源请求Request
bool banker_algorithm(int i, int Request[]) {
    // 第一步：检查请求是否超过最大需求
    for (int j = 0; j < m; j++) {
        if (Request[j] > Need[i][j]) {
            return false;  // 请求超过最大需求，错误
        }
    }
    
    // 第二步：检查请求是否超过可用资源
    for (int j = 0; j < m; j++) {
        if (Request[j] > Available[j]) {
            return false;  // 资源不足，等待
        }
    }
    
    // 第三步：试探性分配
    for (int j = 0; j < m; j++) {
        Available[j] -= Request[j];
        Allocation[i][j] += Request[j];
        Need[i][j] -= Request[j];
    }
    
    // 第四步：执行安全性算法
    if (is_safe()) {
        return true;   // 安全，正式分配
    } else {
        // 第五步：不安全，撤销试探性分配
        for (int j = 0; j < m; j++) {
            Available[j] += Request[j];
            Allocation[i][j] -= Request[j];
            Need[i][j] += Request[j];
        }
        return false;  // 让进程等待
    }
}
```

### 4.3 安全性算法伪代码

```c
bool is_safe() {
    int Work[m];           // 工作向量
    bool Finish[n];        // 完成标志
    
    // 初始化
    for (int j = 0; j < m; j++) {
        Work[j] = Available[j];
    }
    for (int i = 0; i < n; i++) {
        Finish[i] = false;
    }
    
    // 寻找安全序列
    int count = 0;
    while (count < n) {
        bool found = false;
        for (int i = 0; i < n; i++) {
            if (!Finish[i]) {
                // 检查是否能满足进程i的需求
                bool can_allocate = true;
                for (int j = 0; j < m; j++) {
                    if (Need[i][j] > Work[j]) {
                        can_allocate = false;
                        break;
                    }
                }
                
                if (can_allocate) {
                    // 进程i可以完成，释放资源
                    for (int j = 0; j < m; j++) {
                        Work[j] += Allocation[i][j];
                    }
                    Finish[i] = true;
                    count++;
                    found = true;
                }
            }
        }
        
        if (!found) {
            // 找不到可以完成的进程，不安全
            return false;
        }
    }
    
    // 所有进程都能完成，安全
    return true;
}
```

## 五、银行家算法的时间复杂度

### 5.1 时间复杂度分析

- 安全性算法：O(n² × m)
  - 外层循环最多n次
  - 内层循环最多n次
  - 每次检查需要比较m种资源

- 银行家算法：O(n × m) + O(n² × m) = O(n² × m)

### 5.2 优缺点

| 优点 | 缺点 |
|------|------|
| 可以避免死锁 | 时间复杂度较高 |
| 资源利用率较高 | 需要预先知道最大需求 |
| 实现相对简单 | 进程数和资源数较多时开销大 |

## 六、重点考点总结

### 6.1 安全状态与死锁的关系

```
安全状态 → 一定不会发生死锁
不安全状态 → 可能发生死锁（不是一定）
发生死锁 → 一定是在不安全状态
```

### 6.2 银行家算法的核心思想

> 在分配资源前**预判**是否会导致系统进入不安全状态，以此决定是否答应资源分配请求。

### 6.3 考试常见题型

1. **判断系统是否处于安全状态**：执行安全性算法，寻找安全序列
2. **判断是否可以分配资源**：执行银行家算法
3. **计算安全序列**：按照安全性算法逐步执行
4. **分析死锁**：判断是否存在安全序列

### 6.4 计算技巧

1. **计算"最多还需要"**：最多还需要 = 最大需求 - 已分配
2. **计算"剩余可用资源"**：剩余可用 = 总资源 - 已分配总数
3. **安全性算法**：每次选择一个能满足需求的进程，加入安全序列，更新可用资源
4. **多解性**：安全序列可能有多个，找到一个即可

---

## 💡 易错点提醒

1. **安全状态不等于没有死锁**，只是说一定不会发生死锁
2. **不安全状态不一定发生死锁**，只是有可能
3. **银行家算法需要预先知道最大需求**
4. **安全性算法可能需要多轮扫描**
5. **试探性分配后如果不安全，需要撤销**
6. **考试中要注意矩阵的计算和更新**
