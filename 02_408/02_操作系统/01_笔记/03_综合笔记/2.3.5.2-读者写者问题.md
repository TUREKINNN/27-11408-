# 2.3.5.2 读者-写者问题

## 📌 问题描述

有**读者**和**写者**两组并发进程，共享一个文件：
- 两个或两个以上的**读进程**同时访问共享数据时**不会产生副作用**
- 某个**写进程**和其他进程（读进程或写进程）同时访问共享数据时**可能导致数据不一致**

**四个约束条件：**
1. 允许多个读者可以同时对文件执行读操作
2. 只允许一个写者往文件中写信息
3. 任一写者在完成写操作之前不允许其他读者或写者工作
4. 写者执行写操作前，应让已有的读者和写者全部退出

---

## 一、关系分析

### 1.1 互斥关系

| 进程组合 | 是否互斥 | 原因 |
|---------|---------|------|
| 写进程—写进程 | **互斥** | 同时写会导致数据覆盖 |
| 写进程—读进程 | **互斥** | 同时读写会导致数据不一致 |
| 读进程—读进程 | **不互斥** | 多个读者可同时读取 |

### 1.2 与生产者-消费者问题的区别

> 读者进程在读数据后**并不会将数据清空**，并不会改变数据。因此多个读者可同时访问共享数据。

---

## 二、方案一：读优先 ⭐⭐⭐

### 2.1 信号量设置

```c
semaphore rw = 1;     // 用于实现对共享文件的互斥访问
int count = 0;        // 记录当前有几个读进程在访问文件
semaphore mutex = 1;  // 用于保证对count变量的互斥访问
```

### 2.2 写者进程

```c
writer() {
    while (1) {
        P(rw);        // 写之前"加锁"
        
        写文件...;
        
        V(rw);        // 写完了"解锁"
    }
}
```

### 2.3 读者进程

```c
reader() {
    while (1) {
        P(mutex);           // 各读进程互斥访问count
        if (count == 0) {   // 由第一个读进程负责
            P(rw);          // 读之前"加锁"
        }
        count++;            // 访问文件的读进程数+1
        V(mutex);           // 释放对count的互斥访问
        
        读文件...;
        
        P(mutex);           // 各读进程互斥访问count
        count--;            // 访问文件的读进程数-1
        if (count == 0) {   // 由最后一个读进程负责
            V(rw);          // 读完了"解锁"
        }
        V(mutex);           // 释放对count的互斥访问
    }
}
```

### 2.4 核心思想 ⭐⭐⭐

> 设置一个计数器 `count` 记录当前正在访问共享文件的读进程数。用 `count` 的值来判断当前进入的进程是否是**第一个/最后一个读进程**，从而做出不同的处理。

**为什么需要 `mutex` 信号量？**

思考：若两个读进程并发执行，`count=0` 时两个进程可能都满足 `if(count==0)` 条件，都会执行 `P(rw)`，导致第二个读进程阻塞。

解决方案：用互斥信号量 `mutex` 保证各读进程对 `count` 的访问是**一气呵成**的。

### 2.5 读优先的特点

> 潜在问题：只要有读进程还在读，写进程就要一直阻塞等待，可能**"饿死"**。
> 因此，这种算法中，**读进程是优先的**。

---

## 三、方案二：读写公平法 ⭐⭐⭐

### 3.1 信号量设置

```c
semaphore rw = 1;     // 用于实现对共享文件的互斥访问
int count = 0;        // 记录当前有几个读进程在访问文件
semaphore mutex = 1;  // 用于保证对count变量的互斥访问
semaphore w = 1;      // 用于实现"写优先"（读写公平）
```

### 3.2 写者进程

```c
writer() {
    while (1) {
        P(w);         // 申请"排队"信号量
        P(rw);        // 写之前"加锁"
        
        写文件...;
        
        V(rw);        // 写完了"解锁"
        V(w);         // 释放"排队"信号量
    }
}
```

### 3.3 读者进程

```c
reader() {
    while (1) {
        P(w);              // 申请"排队"信号量
        P(mutex);          // 各读进程互斥访问count
        if (count == 0) {  // 由第一个读进程负责
            P(rw);         // 读之前"加锁"
        }
        count++;           // 访问文件的读进程数+1
        V(mutex);          // 释放对count的互斥访问
        V(w);              // 释放"排队"信号量
        
        读文件...;
        
        P(mutex);          // 各读进程互斥访问count
        count--;           // 访问文件的读进程数-1
        if (count == 0) {  // 由最后一个读进程负责
            V(rw);         // 读完了"解锁"
        }
        V(mutex);          // 释放对count的互斥访问
    }
}
```

### 3.4 并发分析

分析以下并发执行 `P(w)` 的情况：

| 执行顺序 | 结果 |
|---------|------|
| 读者1 → 读者2 | 两个读者都可以同时读 |
| 写者1 → 写者2 | 写者2等待写者1完成 |
| 写者1 → 读者1 | 读者1等待写者1完成 |
| 读者1 → 写者1 → 读者2 | 读者2等待写者1完成 |
| 写者1 → 读者1 → 写者2 | 写者2等待读者1完成 |

### 3.5 结论

> 在这种算法中：
> - 连续进入的多个读者可以同时读文件
> - 写者和其他进程**不能同时访问文件**
> - 写者**不会饥饿**
> - 但并不是真正的"写优先"，而是**相对公平的先来先服务原则**
> - 有的书上把这种算法称为**"读写公平法"**

---

## 四、方案对比

| 特性 | 方案一（读优先） | 方案二（读写公平） |
|------|----------------|-------------------|
| 读并发 | 支持 | 支持 |
| 写者饥饿 | **可能** | **不会** |
| 实现复杂度 | 较简单 | 较复杂 |
| 信号量数量 | 2个 + 1个计数器 | 3个 + 1个计数器 |
| 公平性 | 读进程优先 | 先来先服务 |

---

## 五、核心思想总结

### 5.1 关键设计

1. **计数器 `count`**：记录正在读的进程数，判断第一个/最后一个读者
2. **互斥信号量 `mutex`**：保证对 `count` 的互斥访问（一气呵成）
3. **互斥信号量 `rw`**：实现对共享文件的互斥访问
4. **信号量 `w`**：解决写者饥饿问题（读写公平法）

### 5.2 考试重点

> 读者-写者问题为我们解决复杂的互斥问题提供了一个参考思路。核心思想：
> 1. 设置计数器记录当前正在访问共享文件的读进程数
> 2. 用count的值来判断当前进入的进程是否是第一个/最后一个读进程
> 3. 对count变量的检查和赋值不能一气呵成会导致错误，需要用互斥信号量保证
> 4. 解决"写进程饥饿"问题

### 5.3 考试应用

> 绝大多数的考研PV操作大题都可以用**生产者-消费者问题**的思想来解决。如果遇到更复杂的问题，可以想想能否用**读者写者问题**的这几个思想来解决。

---

## 💡 易错点提醒

1. **count不是信号量**，是普通的整型变量，需要用互斥信号量保护
2. 第一个读者执行 `P(rw)`，最后一个读者执行 `V(rw)`
3. 读优先方案中，写者可能饥饿（被无限期阻塞）
4. 读写公平法通过增加 `P(w)` 和 `V(w)` 解决饥饿问题
5. **读者不改变数据**，这是允许多个读者同时读的前提
6. 考试中要根据题目要求选择合适的方案
