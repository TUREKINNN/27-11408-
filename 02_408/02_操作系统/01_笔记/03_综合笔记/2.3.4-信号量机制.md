# 2.3.4 信号量机制

## 📌 知识总览

在学习了进程互斥的软件实现（单标志法、双标志先检查、双标志后检查、Peterson算法）和硬件实现（中断屏蔽、TS/TSL指令、Swap/XCHG指令）后，这些方案存在两个问题：
1. 进入区的"检查"和"上锁"操作无法一气呵成
2. 所有方案都无法实现"让权等待"

1965年，荷兰学者Dijkstra提出了**信号量机制**，卓有成效地实现了进程互斥和同步。

---

## 一、信号量的基本概念

### 1.1 信号量的定义

**信号量**本质上就是一个变量，用来表示系统中某种资源的数量。

- 可以是一个**整数**（整型信号量）
- 也可以是更复杂的**记录型变量**（记录型信号量）

> 例如：系统中只有一台打印机，就可以设置一个初值为1的信号量。

### 1.2 原语（Primitive）

**原语**是一种特殊的程序段，其执行**只能一气呵成，不可被中断**。原语是由**关中断/开中断指令**实现的。

一对原语：
- `wait(S)` 原语
- `signal(S)` 原语

> 可以把原语理解为我们自己写的函数，函数名分别为wait和signal，括号里的信号量S其实就是函数调用时传入的一个参数。

### 1.3 P、V操作的由来

wait、signal原语常简称为**P、V操作**（来自荷兰语proberen和verhogen）。

做题时常写为：
- `P(S)` = `wait(S)`
- `V(S)` = `signal(S)`

---

## 二、整型信号量

### 2.1 定义

用一个**整数型变量**作为信号量，用来表示系统中某种资源的数量。

### 2.2 与普通整数变量的区别

对信号量的操作**只有三种**：
1. **初始化**
2. **P操作**（申请资源）
3. **V操作**（释放资源）

### 2.3 伪代码实现

```c
// 整型信号量 —— 以打印机为例
int S = 1;  // 初始化整型信号量，表示系统中有一台打印机

void wait(int S) {       // P操作（申请资源）
    while (S <= 0);       // 如果资源不够，忙等（"忙等"问题）
    S = S - 1;            // 资源够则分配一个资源
}

void signal(int S) {     // V操作（释放资源）
    S = S + 1;            // 释放资源
}
```

### 2.4 特点分析

| 项目 | 说明 |
|------|------|
| **优点** | "检查"和"上锁"一气呵成，避免了并发、异步导致的问题 |
| **缺点** | **不满足"让权等待"原则**，会发生"忙等" |

> 当资源不足时，进程会一直循环等待（while循环），浪费CPU时间。

---

## 三、记录型信号量 ⭐⭐⭐

### 3.1 问题背景

整型信号量的缺陷是存在"忙等"问题，因此提出了**记录型信号量**，即用记录型数据结构表示的信号量。

### 3.2 数据结构定义

```c
typedef struct {
    int value;        // 剩余资源数
    struct process *L; // 等待队列（阻塞队列）
} semaphore;
```

### 3.3 P操作（wait）实现 ⭐⭐⭐

```c
void wait(semaphore *S) {
    S->value--;                    // 请求一个资源，资源数减1
    if (S->value < 0) {
        // 资源不够，将当前进程阻塞
        block(S->L);              // 将进程加入等待队列
        // block原语：运行态 → 阻塞态，主动放弃处理机
    }
}
```

**关键理解：**
- `S->value--` 后如果 `S->value < 0`，说明资源已分配完毕
- 此时调用 `block` 原语使进程从**运行态→阻塞态**
- 进程被挂到信号量S的等待队列（阻塞队列）中
- **遵循"让权等待"原则，不会出现"忙等"**

### 3.4 V操作（signal）实现 ⭐⭐⭐

```c
void signal(semaphore *S) {
    S->value++;                    // 释放一个资源，资源数加1
    if (S->value <= 0) {
        // 仍有进程在等待该资源
        wakeup(S->L);            // 唤醒等待队列中的一个进程
        // wakeup原语：阻塞态 → 就绪态
    }
}
```

**关键理解：**
- `S->value++` 后如果 `S->value <= 0`，说明还有进程在等待该资源
- 调用 `wakeup` 原语唤醒等待队列中的**第一个进程**
- 被唤醒进程从**阻塞态→就绪态**

### 3.5 S.value的含义 ⭐⭐⭐

| S.value的值 | 含义 |
|-------------|------|
| `S.value = 0` | 资源恰好分配完，没有进程在等待 |
| `S.value > 0` | 还有空闲资源可用 |
| `S.value < 0` | 资源已分配完，且有 `|S.value|` 个进程在等待 |

**具体例子：** 某计算机系统中有2台打印机

```
初始：S.value = 2, S.L = null

P1执行P操作后：S.value = 1（还剩1台打印机）
P2执行P操作后：S.value = 0（打印机恰好分配完）
P3执行P操作后：S.value = -1（1个进程在等待）
P4执行P操作后：S.value = -2（2个进程在等待）

P1释放打印机（V操作）：S.value = -1+1 = 0
  → S.value+1后<=0，说明有进程在等待
  → 唤醒等待队列中的第一个进程

P2释放打印机（V操作）：S.value = 0+1 = 1
  → S.value+1后>0，说明已没有进程在等待
```

## 四、信号量实现互斥 ⭐⭐⭐

### 4.1 基本思想

设置一个互斥信号量 `mutex`，初值为1。

```c
semaphore mutex = 1;  // 互斥信号量

// 进程P1
P(mutex);    // 加锁
// 临界区代码...
V(mutex);    // 解锁

// 进程P2
P(mutex);    // 加锁
// 临界区代码...
V(mutex);    // 解锁
```

### 4.2 执行过程

1. P1先执行 `P(mutex)`：`mutex.value-- = 0`，P1进入临界区
2. P2执行 `P(mutex)`：`mutex.value-- = -1 < 0`，P2被阻塞
3. P1执行完临界区，执行 `V(mutex)`：`mutex.value++ = 0 <= 0`，唤醒P2
4. P2从阻塞态变为就绪态，进入临界区

---

## 五、信号量实现同步 ⭐⭐⭐

### 5.1 基本思想

设置一个同步信号量S，初值为0，实现"前V后P"的关系。

```c
semaphore S = 0;  // 同步信号量

// 进程P1（先执行的操作）
// 代码1...
V(S);            // "前V"：执行完代码1后释放信号量

// 进程P2（后执行的操作）
P(S);            // "后P"：申请信号量，等待P1完成
// 代码2...
```

### 5.2 执行过程

1. 若P1先执行到 `V(S)`：`S.value++ = 1`，P1继续执行
2. P2执行到 `P(S)`：`S.value-- = 0`，P2继续执行
3. 若P2先执行到 `P(S)`：`S.value-- = -1 < 0`，P2被阻塞
4. 当P1执行到 `V(S)`：`S.value++ = 0 <= 0`，唤醒P2，P2继续执行

## 六、重点考点总结

### 6.1 考试注意事项

> **注：若考试中出现P(S)、V(S)的操作，除非特别说明，否则默认S为记录型信号量。**

### 6.2 信号量机制总结

| 特性 | 整型信号量 | 记录型信号量 |
|------|-----------|-------------|
| 数据结构 | 整数 | 结构体（value + 等待队列） |
| 忙等问题 | 存在忙等 | **不存在忙等**（让权等待） |
| 实现复杂度 | 简单 | 较复杂 |
| 考研重要性 | 一般 | **超高频考点** |

### 6.3 P、V操作的使用场景

1. **互斥**：在同一进程中进行一对PV操作
   ```
   P(mutex);  // 进入临界区
   临界区代码
   V(mutex);  // 离开临界区
   ```

2. **同步**：在一个进程中执行P，另一个进程中执行V
   ```
   进程A: ...; V(S);
   进程B: P(S); ...
   ```

3. **前驱关系**：多对同步关系的组合

---

## 💡 易错点提醒

1. **P操作会导致进程阻塞**，V操作不会导致进程阻塞
2. **实现互斥的P操作一定要在实现同步的P操作之后**（否则可能死锁）
3. 记录型信号量的 `S.value` 为负数时，绝对值表示等待的进程数
4. V操作后如果 `S.value <= 0`，需要唤醒等待队列中的进程
